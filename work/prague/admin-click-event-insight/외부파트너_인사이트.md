## 인사이트 P1. "파트너"는 사실상 두 가지 직군 — 여행/티켓 처리자 vs 매장 운영자

상위 사용자별 클릭이 `/reservation`에 거의 100% 쏠리는 그룹과, `/travel-partner` 페이지에 분산된 그룹이 명확히 갈린다.

| 사용자 | 클릭 | 예약 클릭 | 파트너페이지 클릭 | 패턴 |

| Chrissy Tan | 4,290 | 3,776 (88%) | 0 | 예약 + 모바일 |

- "예약 100% 쏠림" 그룹은 **트래블/티켓/픽업 파트너**(레인포·Chrissy Tan·TK Travel). 예약량이 많고 어드민의 `/reservation` 페이지를 워크큐처럼 씀.

**Why it matters**: 두 직군이 같은 어드민 UI를 쓰지만 워크플로가 다르다. `/reservation`은 처리 효율, `/travel-partner/*`는 셀프서비스 안내·검색에 우선순위가 달라야 한다. 두 페이지의 KPI도 분리해 측정하는 게 맞다.

## 인사이트 P3. 같은 예약을 N번의 새 세션에서 반복 조회한다 — 알림 → 자기 확인 → 닫기 루프

레인포컴퍼니 단일 사용자가 `/reservation`에서 동일 예약 행을 **세션마다 1번씩** 144~147회 클릭(예: `260503yfv42e`, `260112…` 행들 다수). Chrissy Tan은 한 예약(`260506ari7zh`)을 **9개의 다른 세션에서 각각 1클릭씩** 열어봄.

- 단순 모달 다시 열기가 아니라 **여러 세션에서 같은 행에 진입** → 외부 링크(이메일/카톡 알림)에서 매번 들어와 같은 예약을 재확인하는 형태.
- 세션 1클릭으로 끝나는 `/reservation` 세션이 **101개**나 있고, 그중 다수가 `예약 상세 - 260506ari7zh` 등 특정 예약 상세 라벨로만 구성.

**Why it matters**: 같은 예약을 N번 보는 행위에는 사람이 못 잊기 때문(단순 재확인)일 수도, 처리 완료된 후에도 알림이 계속 와서일 수도 있다. (a) 처리 완료된 예약은 알림 끊기, (b) 어드민 페이지에 "이 예약은 처리됨" 배지를 명확히 표시하면 반복 진입을 줄일 수 있다.

## 인사이트 P4. 파트너 액션은 ⌬ 결재 동선 — `승인대기 → 예약 확정하기`의 단순 루프

`/reservation*`에서 외부 파트너가 누르는 상태/액션 라벨:

| 라벨 | 클릭 |
|------|------|
| 승인대기 | 441 |
| 확정 | 212 |
| 결제대기 | 196 |
| 예약 확정하기 | 645 |
| 전체환불 + 취소 | 49 |
| 취소 | 33 |

- 상태 필터 클릭 합계(승인대기/확정/결제대기) = 849. 그중 절반(441)이 "승인대기"에 몰림 → 파트너 진입의 가장 흔한 일은 **"내가 승인할 게 있는지" 체크**.
- "예약 확정하기" 클릭 후 다음 클릭까지 간격 중 <2초가 5건(2%) 뿐 → 일반 클릭 패턴은 합리적. 다만 60초 이상 간격이 238건(57%)으로 **확정 후 다음 처리로 넘어가는데 시간이 꽤 든다** = 다른 예약을 찾거나 정보 확인이 끼임.

**Why it matters**: 파트너의 핵심 KPI는 "승인대기 → 확정"의 통과 시간. 어드민 첫 화면을 **승인대기 전용 워크큐**로 만들고, 확정 후 다음 미처리로 자동 점프하는 "처리 모드"를 제공하면 파트너 만족·매출 회수 속도 둘 다 개선.

## 인사이트 P5. `/travel-partner/dashboard`는 사실상 "예약내역 바로가기" 1버튼만 의미 있음

`/travel-partner/dashboard` 클릭 690건 / 376세션 / 74명 — **외부 파트너의 1차 랜딩 페이지**.

라벨 클릭 상위:

| 라벨 | 클릭 | 세션 |
|------|------|------|
| 예약내역 | 306 | 273 |
| 대시보드 | 46 | 35 |
| 처리하러 가기 → | 41 | 32 |
| 스팟 수정 요청 | 32 | 31 |
| Creatrip(로고) | 27 | 21 |
| 메뉴 검색... | 23 | 18 |
| 파트너쉽 정산 | 19 | 18 |

- 273세션(전체 376의 73%)이 대시보드에 와서 "예약내역" 한 번만 클릭하고 떠남.
- "처리하러 가기 →" 41번은 대시보드 위젯의 직접 처리 진입점으로 보임 — 그러나 클릭은 그닥 많지 않음.

**Why it matters**: 대시보드는 73%에게 "예약내역으로 가는 한 클릭"의 인터스티셜에 가깝다. (a) 대시보드 진입 시 미처리 예약이 있으면 즉시 모달/리스트로 노출, (b) 처리할 게 없으면 매출/리뷰 요약을 보여주는 식으로 정보 밀도를 높일 여지.

## 인사이트 P6. 단 1명이 `/travel-partner/spot/14385`에서 722클릭 — 신규 상품 등록 작업의 코스트

`/travel-partner/spot/14385`: 722클릭 / 38세션 / **1명**.
유사하게 `/travel-partner/spot/13573` 197클릭/1명, `/travel-partner/spot/15470` 153클릭/1명…

→ 일부 파트너는 자기 상품을 셀프 등록할 때 **수백 클릭 단위의 노력**을 들이고 있다. 동일 페이지에 38세션 분산 = 여러 날 나눠 작업.

**Why it matters**: 셀프 등록의 마찰을 정성/UX 영상으로 들여다볼 가치가 있음. 가장 클릭을 많이 받는 단일 페이지는 데이터로 가장 비효율적인 워크플로일 가능성. 양식 자동저장, 단계별 진척 표시, 필수항목 우선 표시 등이 후보.
