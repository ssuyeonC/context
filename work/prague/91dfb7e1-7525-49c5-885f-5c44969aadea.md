## 1. 테이블 정규화 위배가 가장 큰 약점

`beauty_receipt_notice`에 캠페인·환급률 컬럼을 붙이면 한 캠페인 = 9개 row(언어별)에 메타가 중복 저장됩니다.
- 환급률 한 번 바꾸려면 9개 row UPDATE
- 캠페인명 바꾸면 9개 row + 매핑 테이블 join key까지 영향
- 매핑 테이블의 FK가 `notice.id`(언어별)? `campaign_name`(문자열)? 둘 다 어색

**살짝만 손보면 해결**: 캠페인 메타를 작은 테이블로 분리 (`beauty_cashback_campaign(id, name, refund_rate, period_*)`), 기존 `beauty_receipt_notice`에 `campaign_id` FK 추가. 어드민 UI 없는 건 그대로지만 정규화는 살아납니다.

```
beauty_cashback_campaign       (메타: id, name, refund_rate, period)
beauty_receipt_notice          + campaign_id FK   (본문: 언어별 row)
beauty_cashback_spot           (매핑: spot_id, campaign_id)
```

## 2. 운영 부담이 엔지니어에 100% 전가

원래 티켓 동기가 "운영팀이 개발 요청 없이 직접"이었는데 지금 안은 정반대로 갑니다. 부산 + 페스티벌 두 건만 처리하고 끝이면 OK, 다만 비슷한 프로모션이 분기마다 들어온다면 엔지니어 운영 비용이 누적 — **이번 캠페인 빈도와 향후 1년 예상 빈도를 먼저 정리하는 게 의사결정 포인트**입니다.

## 3. 매핑 진실 공급원이 두 곳
