# 뱅크오브트립(Bank of Trip) API 연동 요구사항

## 배경

### 문제 정의

트래볼루션(뱅크오브트립)에서 공급하는 티켓/예약 상품의 주문 처리가 현재 수동으로 이루어지고 있음.

- **현재 프로세스**: 크리에이트립 플랫폼에서 예약이 들어오면, 운영 팀원이 트래볼루션 어드민에 직접 접속하여 예약 건을 수동 생성
- **연간 처리량**: 약 300건/년
- **문제점**:
    - 수동 부킹 작업에 인력과 시간 소요
    - 즉시 확정 가능한 상품도 실시간 바우처 발송 불가 (구매 전환율 저하)
    - 휴먼 에러 가능성 (주문 누락, 지연 등)

트래볼루션 측에서 API 연동을 제안하며 공식 API 문서 제공 (v1.1.4).

### 가설적 임팩트

- **운영 효율**: 연 300건의 수동 부킹 작업 제거 → 운영 인력 시간 절감
- **구매 전환율 개선**: 즉시 확정 상품에 대한 실시간 바우처 발송 가능
- **에러 감소**: 수동 입력 과정에서 발생 가능한 휴먼 에러 제거
- **확장성**: 향후 트래볼루션 공급 상품 수 증가 시 추가 운영 부담 없음

## 구현

### 유저 플로우

### 고객 구매 플로우 (티켓 TK 타입)

1. 고객이 크리에이트립에서 트래볼루션 공급 상품 조회 및 구매
2. 결제 완료 시 백엔드에서 뱅크오브트립 주문 생성 API 자동 호출
3. 즉시 확정 상품인 경우, 바우처가 실시간 발급되어 고객에게 전달
4. 고객 취소 요청 시, 뱅크오브트립 주문 취소 API 자동 호출

### 고객 구매 플로우 (예약 BK 타입)

1. 고객이 크리에이트립에서 예약 상품 조회 (스케줄/재고 API로 가용 일자 표시)
2. 고객이 날짜 선택 및 추가 정보 입력 후 구매
3. 결제 완료 시 뱅크오브트립 예약 주문 생성 API 호출
4. 트래볼루션 측에서 예약 승인/거절 → 웹훅으로 수신
5. 승인 시 고객에게 확정 알림, 거절 시 환불 처리

### 상태 변경 수신 (웹훅)

- 사용 처리(REDEEMED) → 내부 주문 상태 '사용 완료'로 변경
- 취소(CANCELED/PARTIAL_CANCELED) → 내부 주문 상태 반영 및 환불 처리
- 예약 승인/거절(BOOKING_ACCEPTED/REJECTED) → 고객 알림 발송

### 백엔드 요구사항

### 1. 뱅크오브트립 API 클라이언트 모듈

기존 외부 연동 패턴(야놀자, LinkKorea 등)을 따라 NestJS 모듈로 구현.

- **위치**: `backend/apps/trip/src/modules/` 하위 (예: `bankoftrip/`)
- **인증**: Bearer 토큰 (API Token, 환경변수로 관리)
- **Base URL**: 개발 `dev.bankoftrip.com` / 운영 `www.bankoftrip.com`
- **프로토콜**: REST + JSON

### 2. 조회 API 연동 (5개)

| API | 메서드 | 엔드포인트 | 용도 |
| --- | --- | --- | --- |
| 상품 조회 | GET | `/api/partner/v1.1/products` | 상품 목록/상세 |
| 옵션 조회 | GET | `/api/partner/v1.1/products/{id}/options` | 상품 옵션 |
| 가격 조회 | GET | `/api/partner/v1.1/products/{id}/options/{id}/units` | 옵션별 가격 |
| 스케줄/재고 조회 | GET | (BK 전용) | 예약 가능 일자/재고 |
| 추가 정보 조회 | GET | (BK 전용) | 예약 시 필요한 입력 항목 |

### 3. 주문 API 연동 (3개)

| API | 메서드 | 엔드포인트 | 용도 |
| --- | --- | --- | --- |
| 주문 생성 | POST | `/api/partner/v1.1/orders` | 결제 완료 후 자동 주문 |
| 주문 조회 | GET | `/api/partner/v1.1/orders/{orderNumber}` | 주문 상태 확인 |
| 주문 취소 | DELETE | `/api/partner/v1.1/orders/{orderNumber}` | 고객 취소 시 연동 취소 |

### 4. 웹훅 수신 엔드포인트

뱅크오브트립에서 보내는 상태 변경 이벤트를 수신하는 엔드포인트 구현.

- **수신 이벤트**: REDEEMED, RESTORED, CANCELED, PARTIAL_CANCELED, BOOKING_ACCEPTED, BOOKING_REJECTED
- **보안**: Authorization 헤더(Bearer 토큰) + x-nonce(재전송 방지) 검증
- **로깅**: 웹훅 수신 기록 저장 (기존 웹훅 로깅 패턴 활용)
- **내부 처리**: 이벤트 타입에 따라 내부 주문 상태 업데이트 및 고객 알림 트리거

### 5. 데이터 매핑

- 뱅크오브트립 상품/옵션 → 크리에이트립 내부 상품 모델 매핑 설계 필요
- 뱅크오브트립 주문번호(TV*) ↔ 크리에이트립 주문번호(referenceNumber) 양방향 추적
- 주문 상태 매핑 (뱅크오브트립 상태 → 크리에이트립 내부 상태)

### 6. 에러 처리 및 모니터링

- 기존 패턴 활용: Sentry 에러 리포팅 + Slack 알림 (주문 실패 시)
- API 호출 실패 시 재시도 로직 (기존 retry wrapper 활용)
- 주문 생성 실패 시 운영팀 수동 처리 가능하도록 fallback 알림

### 참고: 기술 난이도 평가

| 항목 | 난이도 | 비고 |
| --- | --- | --- |
| API 클라이언트 구현 | 낮음 | 기존 패턴(야놀자/LinkKorea) 재활용 |
| 상품 조회 연동 | 낮음 | 단순 GET, 매핑만 필요 |
| 주문 생성/취소 | 중하 | TK 단순, BK는 스케줄+추가정보 스텝 추가 |
| 웹훅 수신 | 낮음 | 기존 웹훅 패턴(온다/PG) 재활용 |
| 데이터 매핑 설계 | 중 | 비즈니스 로직 협의 필요 |
| **종합** | **중하** | 기존 인프라 활용으로 신규 설계 최소화 |
