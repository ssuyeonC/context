#### Priority 1. Type D (미사용 파트너) — 어드민 온보딩 + 진입 UX 개선

**임팩트: 최대 | 실현 가능성: 중상**

**Growth Manager**: Type D만 50%까지 올려도 **약 1,400건/3개월**이 운영에서 파트너로 이동합니다. 이건 현재 파트너 전체 처리 건수 4,345건의 32%에 해당하는 규모예요.

**PM**: 코드베이스를 보면, 현재 파트너 어드민의 가장 큰 문제가 명확합니다.

> **핵심 발견: 파트너 어드민에 "대기 중인 예약 목록" 페이지가 없다.**

현재 파트너가 어드민에서 접근 가능한 메뉴는 다음과 같습니다:

| 메뉴 | 경로 | 기능 |
|------|------|------|
| 스팟 수정 요청 | `/travel-partner/spot` | 스팟 정보 수정 |
| 스팟 등록 요청 | `/travel-partner/spot-create-requests` | 신규 스팟 등록 |
| 파트너쉽 정산 | `/travel-partner/settlements` | 정산 내역 조회 |
| 스팟 리뷰 관리 | `/spotReview` | 리뷰 관리 |
| 예약내역 | `/reservation` | 전체 예약 조회 (어드민 공용) |

**문제점 3가지:**

| # | 문제 | 근거 |
|---|------|------|
| 1 | **예약 승인 전용 페이지 부재** | 파트너는 어드민 공용 예약 페이지(`/reservation`)에서 자기 예약을 찾아야 한다. "확정 대기" 필터링이 파트너 관점으로 최적화되지 않음 |
| 2 | **홈 대시보드 부재** | `Home.tsx`에 파트너용 "대기 건수", "즉시 조치 필요" 같은 요약 정보가 없다. 로그인해도 할 일이 보이지 않음 |
| 3 | **알림톡 → 어드민 연결 단절** | 알림톡은 개별 예약에 대한 1회성 승인 링크(`/travel-partner/reservations/redirect`)만 제공. 승인 후 어드민으로의 자연스러운 전환(onboarding) 경로가 없음 |

**Marketer**: 순시키가 1월 첫 주에 바로 93건을 처리한 건, 아마 내부에 예약 관리 담당자가 이미 있고 어드민 접속을 루틴화한 경우일 거예요. 반면 0% 파트너는 **어디서 뭘 해야 하는지 자체를 모르는 상태**입니다.

**개선 아이디어:**

| # | 아이디어 | 코드 변경 범위 | 기대 효과 |
|---|---------|-------------|----------|
| **1-1** | **파트너 대시보드 신설**: `Home.tsx`에 파트너 역할(`TravelPartner`) 감지 시 "확정 대기 N건", "오늘 예약 N건" 카드를 표시. 클릭하면 필터링된 예약 목록으로 이동 | `Home.tsx` + 예약 집계 Query 추가. Nav.tsx의 권한 분기(`TRAVEL_PARTNERS`) 활용 가능 | 로그인 시 즉시 행동 유도. "할 일이 보이는" 경험 |
| **1-2** | **파트너 전용 예약 관리 페이지**: "확정 대기" 상태 예약만 필터링된 파트너 전용 뷰. 일괄 승인(체크박스 선택 → 일괄 확정) 기능 포함 | 새 페이지 + `confirmReservation` mutation을 배치 호출하는 기능. 기존 `ConfirmReservationRequestButton` 로직 재활용 | 순시키처럼 하루 2~3회 접속하여 일괄 처리하는 워크플로우를 표준화 |
| **1-3** | **알림톡 승인 완료 → 어드민 안내 삽입**: 현재 `StatusView.tsx`의 `success` 상태에서 "승인 완료" 메시지만 노출. 여기에 "파트너 어드민에서 모든 예약을 한눈에 관리하세요" CTA 추가 | `StatusView.tsx` success 상태에 CTA 버튼 1개 추가 (최소 변경) | 알림톡 사용자를 어드민으로 자연 유입 |

**PO**: 1-1과 1-3은 소규모 변경이니 빠르게 가능하고, 1-2는 핵심 기능이지만 개발 범위가 크다. **1-1, 1-3을 선행 → 1-2를 본격 구현**하는 순서가 맞겠습니다.

#### Priority 2. Type A 부분 활용 파트너 (네일박스 74%, 모델로 37%) — 어드민 UX 마찰 제거

**임팩트: 중 | 실현 가능성: 높음**

**PM**: 어드민을 쓰고 있지만 승인율이 74~37%인 파트너가 있습니다. 이미 어드민에 접속하는 행동은 하고 있으니, UX 마찰만 줄이면 처리율이 올라갈 수 있어요.

**코드 분석 결과:**

현재 예약 확정 플로우는 예약 상세 페이지(`/reservation?id=...`)에서 `ConfirmReservationRequestButton`을 클릭하는 방식입니다. 이건 **1건씩 상세 페이지에 진입해야 하는 구조**예요. 예약 10건을 확정하려면 10번 페이지를 열어야 합니다.

**개선 아이디어:**

| # | 아이디어 | 코드 변경 범위 | 기대 효과 |
|---|---------|-------------|----------|
| **2-1** | **예약 목록에서 바로 승인 버튼**: 목록 테이블 각 행에 "승인" 버튼 추가. 상세 페이지 진입 없이 1클릭 승인 | 예약 목록 컴포넌트에 inline action 버튼 추가. `confirmReservation` mutation은 이미 존재 | 승인까지의 클릭 수 3→1로 감소. 처리 속도 체감 향상 |
| **2-2** | **일괄 선택 승인**: 체크박스로 복수 예약 선택 → "선택 승인" 버튼으로 배치 처리 | Ant Design Table의 `rowSelection` 활용 + batch mutation 호출 | 순시키처럼 100건/주 처리하는 파트너에게 결정적 효율화 |

**Growth Manager**: 네일박스가 74%인 건, 어드민에 접속하는 날과 안 하는 날이 있다는 뜻일 수 있어요. 접속했을 때의 처리 효율을 높이는 2-1, 2-2가 직접적으로 도움이 됩니다.

#### Priority 3. Type B (알림톡 주력) — 알림톡 채널 고도화

**임팩트: 중하 | 실현 가능성: 중**

**PM**: 알림톡 주력 파트너는 이미 68~100%의 높은 처리율을 보이고 있어서, 추가 개선의 **절대 임팩트는 상대적으로 작습니다.** 하지만 알림톡이 중소규모 파트너의 주력 채널이므로, 신규 파트너를 알림톡으로 온보딩시킬 때의 경험 품질이 중요해요.

**코드 분석 결과:**

현재 알림톡 승인 플로우(`TravelPartnerReservationsRedirect`)는:
1. 파트너가 카카오톡 링크 클릭
2. `action_token` + `reservation_id` + `type`(Approve/Rejection/Change) 파라미터로 진입
3. `usePartnerReservationAction` 훅이 상태 머신(reducer)으로 플로우 관리
4. 로그인 불필요 (`skipAuth: true`)
5. 승인 완료 → success 화면 → 상세 페이지로 리다이렉트

**문제점:**

| # | 문제 | 근거 |
|---|------|------|
| 1 | **단건 처리 한계** | 예약 3건이 동시에 들어오면 알림톡 3개를 각각 열어서 처리해야 함 |
| 2 | **승인 외 액션 부재** | 현재 Approve만 가능. 거절 사유 입력, 시간 변경 제안 등 불가 |
| 3 | **승인 알림 미수신 파트너 존재** | `ToggleSendingReservationAcceptanceKakaoMessageSwitch` 설정이 스팟별로 존재하나, 미등록 상태인 스팟이 다수일 가능성 |

**개선 아이디어:**

| # | 아이디어 | 코드 변경 범위 | 기대 효과 |
|---|---------|-------------|----------|
| **3-1** | **알림톡 승인 페이지에 "다른 대기 예약" 표시**: `StatusView.tsx`의 success 상태에서, 같은 스팟의 다른 확정 대기 예약을 보여주고 연속 처리 가능하게 | `reservationPartnerAction` query에 같은 스팟 대기 건수 필드 추가 필요 (백엔드 변경) | 건당 처리에서 세션당 처리로 전환. 알림 1개로 다건 처리 가능 |

#### Priority 4. Type C (혼용 파트너) — 현 상태 유지 + 모니터링

**임팩트: 낮음 | 실현 가능성: 높음**

**Growth Manager**: Type C는 이미 합산 79~100%의 처리율을 보이고 있어요. 이경민foret과 한순간은 100%입니다. 이 그룹은 **별도 개선 없이 현 상태를 유지하는 것이 합리적**입니다.

**PM**: 다만 이 그룹의 패턴이 재미있어요. 호수도산은 어드민 78% + 알림톡 10%인데, 이건 매장 내 역할 분담 패턴일 수 있습니다. **이 패턴을 다른 파트너에게 레퍼런스로 공유**하면 Type D 온보딩에 활용할 수 있어요.

### 2-3. 우선순위 최종 정리

**PO**: 전체를 정리하겠습니다.

| 순위 | 타겟 | 핵심 개선 | 예상 임팩트 (3개월 기준) | 개발 규모 |
|---:|------|----------|:-----:|------|
| **P1-Quick** | Type D 미사용 파트너 | 1-1 파트너 대시보드 + 1-3 알림톡→어드민 안내 | 인식 전환 기반 마련 | S (1~2일) |
| **P1-Core** | Type D 미사용 파트너 | 1-2 파트너 전용 예약 관리 페이지 (확정 대기 목록 + 일괄 승인) | **~1,400건 이동** | L (1~2주) |
| **P2** | Type A 부분 활용 | 2-1 목록 인라인 승인 + 2-2 일괄 승인 | **~200건 추가** | M (3~5일) |

**Growth Manager**: P1-Quick과 P1-Core를 병렬로 가면 좋겠어요. Quick은 바로 배포하고, Core는 설계-개발-배포 사이클을 타는 거죠. P2는 P1-Core와 상당 부분 겹치니까(예약 목록 + 일괄 승인), **P1-Core를 만들면서 P2도 자연스럽게 해결되는 구조**로 설계하면 됩니다.

**PM**: 맞습니다. P1-Core의 "파트너 전용 예약 관리 페이지"가 곧 P2의 "목록 인라인 승인 + 일괄 승인"이기도 해요. **하나의 기능으로 P1과 P2를 동시에 커버하는 설계**가 가능합니다.

## 4. 인사이트 정리

### Insight 1. Type D가 "가장 큰 한 방"이다

2,793건(TOP 30 기준)이 100% 운영자 처리. 이 그룹의 절반만 전환해도 현재 파트너 처리 건수의 30%+ 증가 효과. **개발 투자 대비 ROI가 가장 높은 타겟.**

### Insight 2. "파트너 전용 예약 관리 페이지"가 P1과 P2를 동시에 해결한다

Type D 온보딩과 Type A 효율화의 핵심 니즈가 동일 — **"확정 대기 예약을 한눈에 보고, 빠르게 일괄 처리하는 것."** 하나의 기능이 두 그룹을 동시에 커버.

### Insight 3. 기존 코드 자산이 충분하다

`confirmReservation` mutation, 파트너 역할 분기, Ant Design Table 패턴이 이미 존재. 완전히 새로운 인프라가 필요한 게 아니라, **기존 패턴의 조합과 파트너 관점 재배치**가 핵심.

### Insight 4. Quick Win(P1-Quick)으로 가설을 빠르게 검증할 수 있다

대시보드 카드와 알림톡 CTA는 1~2일 내 배포 가능. 배포 후 2주간 Type D 파트너의 어드민 접속률 변화를 측정하면, P1-Core 투자 전에 **"알려주면 쓸 것인가"라는 핵심 가설을 검증**할 수 있다.

### Insight 5. 운영적 개입과 제품 개선을 병렬로 가야 한다

제품 개선(P1~P3)은 2~4주 소요. 그 사이 **운영팀의 Type D 파트너 개별 안내(전화/카톡)**를 병행하면, 제품이 준비되기 전에도 참여율 상승이 가능. 4/6 회의에서 합의한 "선행 조사"가 이 역할을 한다.
