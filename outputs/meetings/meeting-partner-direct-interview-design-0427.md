## 0. 한 문장 요약

> **운영팀 인터뷰로 도출한 3축(예약 UX / 수정 요청 UX / UI 개선)을 외부 파트너 입장에서 다시 검증한다. 운영팀이 본 문제와 파트너가 체감하는 문제가 같은가, 우리가 제안하는 방향이 파트너의 행동을 실제로 바꿀 것인가를 확인한다.**

## 1. 왜 외부 파트너 인터뷰가 필요한가

### 1-1. 운영팀 인터뷰만으로는 답할 수 없는 것

| 구분 | 운영팀 인터뷰로 안 것 (4/20) | 파트너 인터뷰로 알아야 할 것 |
|---|---|---|
| 카톡 의존 | 운영팀이 카톡으로 받는다 | **파트너는 왜 카톡을 선호하는가? 어드민 대신 카톡을 쓰는 진짜 이유** |
| 어드민 미사용 | 운영팀 추정: 인지 부재 + 동기 부재 | **파트너 본인 입으로: 모르는가, 알지만 안 쓰는가, 알고 시도했다 실패했는가** |
| 수정 요청 | 운영팀 부담이 가장 크다 | **파트너 입장에서도 수정 요청이 가장 큰 페인 포인트인가? 다른 페인이 있나** |
| 블로그 수정 | 파트너가 직접 못함 (대행 중) | **파트너가 직접 수정하고 싶은가? 어떤 범위까지? 책임 부담은 어떻게 느끼나** |
| 가격 변경 | 정산타입별 분기 부재가 운영 이슈 | **파트너가 가격 변경 시 어떤 어려움을 겪는가? 자동 계산이 신뢰 가능한가** |
| 고령 파트너 | 성민 담당 풀에 다수 존재 | **본인이 고령 담당자라면, 어드민의 어떤 부분이 어려운가** |

### 1-2. 검증할 가설

| # | 가설 | 인터뷰로 확인할 것 |
|---|---|---|
| **H1** | 파트너는 어드민 존재 자체를 모르거나, 알아도 활용 동기가 없다 | 인지 여부 + 미사용 사유의 실제 분포 |
| **H2** | 어드민에 진입했더라도 "수정 요청" 흐름의 마찰로 카톡으로 우회한다 | 실제 어드민 시도 경험과 좌절 지점 |
| **H3** | 대시보드/알림톡 개선이 파트너의 직접 처리 행동을 실제로 바꾼다 | 와이어프레임 노출 시 "이게 있으면 쓸 것 같다"의 진정성 |
| **H4** | 블로그 수정 권한 이양은 파트너가 원하는가, 부담스러워하는가 | 의향 + 적정 범위 + 책임 인식 |
| **H5** | 가격 자동 계산은 파트너의 신뢰를 얻을 수 있는가 | 현재 가격 변경 워크플로우의 마찰 + 자동화 수용도 |
| **H6** | 고령 파트너에게 UI 개선이 실질적 진입 장벽을 낮추는가 | 현재 어드민의 구체적 어려움 |

## 2. 인터뷰 + 설문 구조

지윤님 제안(4/20)대로 **심층 인터뷰 + 서면 설문 병행**으로 설계.

### 2-1. 두 트랙의 역할 분담

| 트랙 | 대상 | 목적 | 산출물 |
|---|---|---|---|
| **A. 심층 인터뷰** | 8~10개 파트너 (Type A/B/C/D 골고루) | "왜"와 "어떻게"의 맥락 수집 | 가설 H1~H6 정성 검증 |
| **B. 서면 설문** | 활성 파트너 전체 (수십~수백) | 정량 분포 검증 | 인터뷰 결과의 일반화 가능성 판단 |

- **순서**: A를 먼저 → 인터뷰에서 발견한 패턴을 객관식 항목으로 만들어 B 설계 → B로 정량 검증
- **이유**: 설문 먼저 하면 "우리가 묻고 싶은 것"만 묻게 되어 새로운 발견이 어려움. 인터뷰에서 우리가 모르던 변수를 먼저 찾는다.

### 2-2. 본 문서의 범위

본 문서는 **A. 심층 인터뷰 설계**에 집중. B 설문 설계는 인터뷰 결과 종합 후 별도 문서로.

## 3. 인터뷰 대상 선정

### 3-1. 선정 원칙

| 기준 | 이유 |
|---|---|
| **Type A~D 분포 커버** | 각 유형의 사용·미사용 사유는 다를 가능성이 큼 |
| **카테고리 다양성** | 뷰티/메디컬/체험/식당 등 — 카테고리별 어드민 활용 패턴 차이 가능 |
| **담당자 연령 다양성** | "젊은 담당자 = 전환 쉬움" 가설 검증을 위해 |
| **담당자 1인 vs 다인** | 지윤 인사이트("1인 체제일 때 전환 쉬움") 검증 |
| **관계 양호 파트너 우선** | 부정 피드백 받을 수 있는 라포가 형성된 곳부터 |

### 3-2. 후보 리스트 (제휴팀과 협의 필요)

| Type | 우선 후보 | 선정 사유 |
|---|---|---|
| **A. 어드민 주력** | 순시키 (홍대 본점 / 모먼트점) | 97.6%·96.4% 처리, **잘 쓰는 이유**가 핵심 |
| **A. 어드민 주력** | 메디큐브의원 (강남) | 지윤 성공 사례 — 카톡에서 어드민으로 **전환된 케이스**. 전환의 성공 요인 청취 |
| **A. 부분 활용** | 네일박스 / 모델로스튜디오 | 어드민을 쓰지만 처리율 74%·37% — **마찰 지점**이 무엇인가 |
| **B. 알림톡 주력** | 레브치과 / 퍼스 | 알림톡 95%+ — 알림톡이 충분한지, 어드민 필요성 인식은 어떤지 |
| **C. 혼용** | 호수도산 / 이경민foret | 채널을 나눠 쓰는 **운영 구조** 청취 |
| **D. 미사용** | 시현하다 (3개점 중 1) | 예약 200건+ / 0% — **가장 큰 미사용 풀**의 진짜 이유 |
| **D. 미사용** | 코코리아 명동 / 이가자 명동 | 중규모 미사용 — Type D의 일반적 패턴 |
| **고령 담당** | 성민 담당 풀 중 1~2개 (한식다이닝, 한복 등) | UI 친화 가설 검증 |

목표: **8~10명**, Type 분포 A 3 / B 1 / C 1 / D 3 / 고령 1~2.

### 3-3. 라포·접근 방식

- 1차 컨택: **담당 매니저(혜원·지윤·성민)를 통한 소개**가 답변의 솔직함을 좌우
- 인센티브: 결과 공유 + (가능 시) 소액 사례 (커피 쿠폰 등)
- 부담 경감 메시지: "평가가 아니라, 우리가 더 나은 도구를 만들기 위한 것"

## 4. 인터뷰 방법론

| 항목 | 결정 |
|---|---|

### 영역 1. 도입 & 라포 (3분)

**목적**: 인터뷰 부담 경감, 답변자의 자기소개

| # | 질문 |
|---|---|
| 1-1 | 간단히 본인 역할과 현재 매장에서 어떤 업무를 담당하시는지 알려주세요. |
| 1-2 | 매장에서 크리에이트립 예약을 관리하시는 분이 한 분이세요, 여러 분이세요? |

### 영역 2. 크리에이트립과의 일상 업무 (7~10분)

**목적**: 파트너가 우리와 어떤 일들을 하는지, 어떤 채널로 하는지 **백지 상태에서** 청취

| # | 질문 | 의도 |
|---|---|---|
| 2-1 | 크리에이트립 예약을 받고 처리하시는 일상이 어떻게 되세요? 아침에 출근해서 어떤 순서로 일하시는지요. | **전체 워크플로우** 파악. 사전 가설 없이 듣기 |
| 2-2 | 크리에이트립과 연락하실 때 주로 어떤 방식으로 하세요? (카톡 / 전화 / 어드민 / 다른 방식) | 채널 분포의 **체감** 확인 |
| 2-3 | 가장 자주 발생하는 요청이나 변경 사항은 어떤 것이 있으세요? | 파트너 입장에서의 **빈도 높은 업무** 발견 |
| 2-4 | 그 중에서 가장 시간이 많이 걸리거나 번거로운 것이 있다면? | **페인 포인트** 자발적 발화 유도 |
| 2-5 | 최근 한 달 안에 크리에이트립 매니저(혜원/지윤/성민님 등)와 카톡으로 주고받으신 일이 어떤 것이 있으셨어요? 기억나는 대로요. | 카톡 의존의 **실제 내용** 매핑 |

**진행 팁**: 2-5가 핵심. 카톡으로 하는 일을 분류하면 "어드민으로 흡수 가능한 항목"이 명확해짐.

### 영역 3. 파트너 어드민 인지·사용 (5~7분)

**목적**: H1(인지/동기) 검증

> ⚠️ **중요한 분기점**: 답변자가 어드민을 아는지 모르는지에 따라 이후 질문 갈래가 달라짐.

| # | 질문 | 의도 |
|---|---|---|
| 3-1 | 파트너 어드민이라는 페이지가 있다는 것은 알고 계세요? | **인지 자체 확인** — 0%/혼용 파트너의 일부는 모를 가능성 |
| 3-2 *(아는 경우)* | 어떻게 알게 되셨어요? | 인지 경로 — 카톡 안내, 알림톡, 매니저 설명 등 |
| 3-3 *(아는 경우)* | 평소에 어드민을 얼마나 자주 들어가시나요? | 사용 빈도의 자기 보고 |
| 3-4 *(자주 안 쓰는 경우)* | 어드민이 있는 걸 아시는데 자주 안 쓰시는 이유가 있으세요? | **미사용의 진짜 이유** — 모름이 아닌 의식적 회피 |
| 3-5 *(잘 쓰는 경우)* | 처음 어드민을 쓰기 시작하실 때 어떤 계기가 있었어요? | **전환의 트리거** — 다른 파트너 온보딩에 활용 가능 |
| 3-6 *(잘 쓰는 경우)* | 어드민에서 가장 자주 쓰는 기능과 가장 안 쓰는 기능은? | 기능 가치 분포 |

### 영역 4. 수정 요청·운영 변경 흐름 (7~10분)

**목적**: H2 검증 — 본 프로젝트 §3-2(스팟/블로그 수정 UX)의 핵심 영역

| # | 질문 | 의도 |
|---|---|---|
| 4-1 | 매장 정보(가격, 운영시간, 메뉴, 주의사항 등)를 변경해야 할 때 보통 어떻게 처리하세요? | 현재 워크플로우 |
| 4-2 | 변경 요청을 보낸 뒤, 실제로 사이트에 반영되기까지 얼마나 걸린다고 느끼세요? | **체감 처리 시간** — 운영팀 인지 부재의 결과 확인 |
| 4-3 | 그 시간 동안 답답하셨던 적이 있으세요? 그럴 땐 어떻게 하셨어요? | 카톡 우회의 트리거 |
| 4-4 | 만약 어드민에서 직접 가격이나 운영시간을 수정 요청할 수 있다면, 그 기능을 쓰실 것 같으세요? *(이미 어드민에 있다는 사실을 모를 수 있음을 감안)* | **현 기능 인지 vs 의향** 분리 |
| 4-5 | 가격 변경 시, 수수료가 자동으로 계산돼서 판매가에 반영되는 기능이 있으면 어떨까요? 우려되는 점이 있다면? | §3-2-C 검증 |
| 4-6 *(블로그 연결 파트너)* | 블로그 내용을 직접 수정하고 싶으셨던 적이 있으세요? 어떤 부분을요? | §3-2-B 검증 — **의향과 범위** |
| 4-7 *(블로그 연결 파트너)* | 만약 블로그를 직접 수정할 수 있게 된다면, 잘못 수정해서 문제가 될까봐 부담되는 점이 있을까요? | 책임/부담 인식 |

### 영역 5. 예약 처리 경험 (5~7분)

**목적**: §3-1(예약 처리 UX) 검증. **사용 유형에 따라 분기.**

#### 5-A. Type A/B/C (어드민 또는 알림톡 사용 중)

| # | 질문 |
|---|---|
| 5-A-1 | 예약 들어왔을 때 어떻게 확인하고 처리하세요? |
| 5-A-2 | 알림톡 vs 어드민 중 어느 쪽이 편하세요? 이유는? |
| 5-A-3 | 예약 변경 요청을 처리할 때 어려운 점이 있으세요? |
| 5-A-4 | 예약을 한 번에 여러 건 처리해야 할 때, 지금 방식이 충분히 빠른가요? |

#### 5-B. Type D (미사용)

| # | 질문 |
|---|---|
| 5-B-1 | 예약은 평소에 어떻게 확정되고 있나요? (운영팀이 대신 처리한다는 사실을 인지하고 있는지) |

## 8. 사전 예상 인사이트 (가설)

> 인터뷰 전에 예상되는 결과를 적어두고, 끝난 뒤 실제와 비교하면 우리의 사고 편향을 발견할 수 있다.

### Insight 1 (예상). Type D 미사용은 "모름"보다 "동기 부재"가 클 것

운영팀 인터뷰에서 "혜원이 알림톡 버튼 누르고서야 어드민 존재를 알게 됨"이 있었으나, 1년+ 거래 파트너의 다수는 어드민을 안다. 다만 **"운영팀이 대신 해주니 안 해도 됨"**이 본질일 가능성.

### Insight 2 (예상). 카톡 선호의 진짜 이유는 "즉시 응답"

어드민으로 요청하면 언제 처리될지 모르는 불확실성 vs 카톡 보내면 매니저가 곧 답함. **§3-2-A(어드민 알림 + 비동기 코멘트)**가 이 불확실성을 해소할 수 있는지가 관건.

### Insight 3 (예상). 블로그 직접 수정은 양분될 것

"내가 직접 고치고 싶다" 파트너 vs "잘못 건드려서 문제 될까 무섭다" 파트너로 갈릴 가능성. **템플릿 제한 + 검수 워크플로우**의 설계 방향이 정당화됨.

### Insight 4 (예상). 가격 자동 계산은 신뢰 이슈가 클 것

수수료율을 시스템이 알아서 계산해서 판매가가 바뀐다 → 파트너 입장에서 **"내 돈 계산을 시스템이 한다"**는 불안. 투명성(계산 과정 공개) UX가 필수.

### Insight 5 (예상). 고령 파트너의 핵심 어려움은 "용어"

UI 컴포넌트보다도 **"승인 대기"·"확정"·"정산 예정" 같은 용어**가 직관적이지 않을 가능성. 용어 단순화가 그래픽 개선보다 임팩트가 클 수도.
