**일시**: 2026-03-10
**참석자**: PO (제품 방향/전략 총괄), PM (세부 기획), Growth Manager (데이터/전환 판단), Marketer (고객 접점/커뮤니케이션)
**목적**: 확정된 3개 태스크(COM-1895/COM-2320 좋아요→플랜, COM-2328 럭키드로우×교차탐색, 앱 전용 쿠폰)와 연계하여 D1 리텐션을 더 끌어올릴 수 있는 추가 아이디어 발굴

## 0. 오늘 회의의 전제

**PO**: 3개 태스크는 이미 확정됐습니다. 오늘은 **이 태스크들을 평가하는 게 아니라, 이것들 위에 얹거나 옆에 붙일 수 있는 것**들을 아이데이션합니다. 기준은 세 가지입니다.

1. **확정 태스크와 시너지가 있는가** — 독립적으로 좋은 아이디어여도, 3개 태스크와 연결되지 않으면 우선순위가 낮음
2. **RETURN_D1에 직접 기여하는가** — "좋은 기능"이 아니라 "내일 돌아올 이유"를 만드는 기능인가
3. **개발 비용 대비 효과가 있는가** — Phase 1 안에 함께 태울 수 있는 규모인가

먼저 확정 태스크의 구조를 한 번 짚고, 빈 곳을 찾아봅시다.

## 1. 확정 태스크의 커버리지 — "어디가 비어있는가"

**Growth Manager**: 확정 태스크 3개가 커버하는 유저 여정 구간을 2차 회의의 레이어 모델로 맵핑합니다.

```
Layer 0: 방문 ─── (커버 없음)
   ↓
Layer 1: 탐색 ─── COM-2328 (럭키드로우×교차탐색): 상세 3회 조회 유도
   ↓
Layer 2: 저장 ─── COM-1895/COM-2320 (좋아요→플랜): 좋아요 클릭 후 플랜 연결
   ↓
Layer 3: 계획 ─── COM-1895 (좋아요 아이템으로 플랜 생성): 플래너 진입
   ↓
Layer 4: 구매 ─── 앱 전용 쿠폰: 앱에서 구매 전환 유인
   ↓
Layer 5: 공유/재방문 ─── (커버 없음)
```

**빈 구간이 두 곳** 보입니다.
- **Layer 0(방문)**: 유저가 처음 크리에이트립에 오는 것 자체, 즉 D0 모수
- **Layer 5(공유/재방문)**: 구매 후, 또는 여행 중/후에 다시 오는 메커니즘

그리고 각 Layer 사이의 **전환을 강화하는 장치**도 추가할 여지가 있어요.

**PM**: 2차 회의에서 합의한 CRM 시나리오 5개도 상기합시다. 이것들은 확정 태스크와 별개로 진행되어야 할 항목인데, 아직 구체 실행이 안 됐습니다.

| CRM 시나리오 | 상태 | 관련 태스크 |
|-------------|------|-----------|
| 1. 좋아요 유저 D+1 시의성 리마인더 | 설계 완료 | COM-1895/2320 |
| 2. 1개 탐색 이탈 유저 → 2개째 추천 | 설계 완료 | COM-2328 |
| 3. 플래너 유저 "빈 날짜" 리마인더 | 설계 완료 | COM-1895 |
| 4. 비교하기 유저 "결정 유도" | **전제 기능 미구현** | — |
| 5. "관심 있어요" 유저 구매 넛지 | **전제 기능 미구현** | — |

**PO**: 좋습니다. 이 빈 곳들과 CRM을 염두에 두고, 자유롭게 아이디어를 꺼내봅시다. 구조는 나중에 잡겠습니다.

## 2. 아이데이션 — 확정 태스크 연계 추가 시책

### 아이디어 1: "좋아요 스트릭" — 좋아요 행동 자체에 연속성 부여

**Marketer**: COM-1895/COM-2320이 "좋아요 → 플래너"의 **출구**를 만드는 거잖아요. 그런데 **좋아요 자체의 빈도를 올리는 장치**는 없어요. 좋아요 일 108명이라는 모수가 더 커져야 COM-2320의 바텀시트 CTA도 더 많이 노출됩니다.

제안: **좋아요 N개 달성 시 보상**. 예를 들어:
- 좋아요 3개 → "좋아요한 스팟 중 1개 5% 할인 쿠폰" (앱 전용 쿠폰과 연동)
- 좋아요 5개 → 럭키드로우 추가 응모권 (COM-2328과 연동)
- 좋아요 10개 → "나만의 여행 스타일" 뱃지 + 맞춤 추천

**Growth Manager**: COM-2328의 "상세 3회 조회 → 응모권"과 구조가 같습니다. **"탐색하면 보상(COM-2328)" + "저장하면 보상(좋아요 스트릭)"이 나란히 있으면, Layer 1과 Layer 2를 동시에 강화하는 게이미피케이션 시스템**이 됩니다.

그리고 앱 전용 쿠폰과 연결하면 — 좋아요 3개 달성 쿠폰이 앱에서만 사용 가능하면, 웹 유저에게 앱 전환 동기까지 줄 수 있어요.

**PM**: 개발 비용은 어떨까요?

**Growth Manager**: COM-2328에서 이미 "N회 행동 → 보상" 로직을 구현합니다. 좋아요 스트릭은 **카운트 대상만 "상세 조회"에서 "좋아요 클릭"으로 바꾸는 것**이라 추가 공수가 크지 않아요. 보상 지급 API도 COM-2328의 응모권 API를 재사용하면 됩니다.

**PO**: 좋아요 스트릭은 **COM-2328 인프라를 재활용하면서 COM-1895/2320의 모수를 키우는** 설계네요. 세 태스크 모두와 시너지가 있습니다. 후보에 넣겠습니다.

### 아이디어 2: "여행 D-day 카운트다운" — 플래너 유저에게 매일 올 이유

**PM**: COM-1895로 좋아요에서 플랜을 생성하면, 여행 날짜가 설정됩니다. 이 유저에게 **"출발까지 D-14"** 같은 카운트다운을 제공하면 어떨까요?

단순 카운트다운이 아니라, **D-day에 따라 단계별 콘텐츠를 제공**하는 거예요.

```
D-14: "여행 준비 시작! 아직 빈 날짜가 있어요 → 플래너 확인"
D-10: "환전 팁: 한국에서 환율 좋은 곳은?" → 블로그 링크 + 환전 상품 카드
D-7:  "짐 싸기 체크리스트 + 이번 여행에서 놓치면 아쉬운 스팟 추천"
D-3:  "공항에서 시내 이동법 가이드 → 교통 패스 상품"
D-1:  "내일 출발! 1일차 일정 미리보기 → 플래너"
D-day: "한국 도착! 오늘 일정 확인하기 → 플래너"
```

**Marketer**: 이건 2차 회의에서 P2로 미뤘던 "여행 준비 대시보드"의 경량 버전이에요. 대시보드를 만들 필요 없이, **푸시 알림 + CRM만으로 구현 가능**합니다. 앱 전용 기능으로 만들면 T3(앱 쿠폰)과도 시너지가 있고요.

**Growth Manager**: D-day 카운트다운의 가장 강력한 점은 **매일 다른 콘텐츠가 온다**는 거예요. "오늘은 뭐가 올까?"라는 기대 자체가 Open Loop입니다. 그리고 각 메시지에 상품 카드를 넣으면 **콘텐츠 → 교차탐색 → 구매 파이프라인이 작동**해요.

**PM**: 주의할 점은, 이게 작동하려면 **유저가 플래너에 여행 날짜를 입력해야 한다**는 전제가 필요합니다. COM-1895의 플랜 생성 플로우에서 날짜 입력이 포함되어 있으니, **COM-1895 완료 후 자연스럽게 붙일 수 있는 후속 기능**입니다.

**PO**: 개발 비용은?

**PM**: 제품 개발이라기보다 **CRM 자동화**에 가깝습니다. 플래너에 여행 날짜가 있는 유저를 대상으로, D-day 기반 스케줄 푸시를 셋업하면 됩니다. 콘텐츠 제작(각 D-day별 가이드)이 주요 공수고, 개발은 **푸시 스케줄러 1~2일**이면 충분해요. 마케팅 콘텐츠 제작이 3~5일.

**Marketer**: 그리고 이 콘텐츠 중 일부는 2차 회의에서 합의한 **콘텐츠 시리즈("서울 벚꽃 3박4일 완전 공략" 등)를 재활용**할 수 있어요. D-7에 "짐 싸기 체크리스트"는 새로 만들어야 하지만, D-3의 "공항→시내 이동법"은 Reddit 가이드로 이미 제작 계획에 있습니다.

**PO**: 후보에 넣겠습니다. COM-1895와의 직접적 연계 + CRM 기반 저비용 실행이 매력적입니다.

### 아이디어 3: "탐색 하이라이트 리캡" — D+1 재방문의 가장 가벼운 트리거

**Growth Manager**: COM-2328에서 유저가 상세페이지를 3개 이상 보게 됩니다. 이 유저에게 **D+1에 "어제의 탐색 리캡"을 보내는** 건 어떨까요?

```
[D+1 아침 푸시/이메일]

"어제 경복궁, 남산타워, 북촌한옥마을을 둘러보셨네요!
 가장 인기 있는 조합은 경복궁 → 북촌 → 남산 순서예요.

 👉 이 동선으로 플랜 만들기
 👉 비슷한 스팟 더 보기"
```

**Marketer**: 이건 2차 회의의 CRM 시나리오 2번("1개 탐색 이탈 → 2개째 추천")을 **업그레이드한 버전**이에요. 시나리오 2번은 "1개만 본 유저"를 대상으로 했는데, 이건 **COM-2328 덕분에 3개 이상 본 유저**를 대상으로 합니다. 데이터가 더 풍부하니 메시지도 더 개인화됩니다.

핵심은 **"어제 본 것들을 정리해줬다"**는 가치예요. 유저가 직접 정리하지 않아도 크리에이트립이 해줍니다. 이건 구글 스프레드시트나 노션으로는 불가능한 — **크리에이트립에서만 받을 수 있는 서비스**죠.

**Growth Manager**: 그리고 "이 동선으로 플랜 만들기" CTA는 COM-1895의 **"좋아요 아이템으로 플랜 생성"과 유사한 메커니즘을 "탐색 아이템"으로 확장**하는 거예요. 좋아요를 누르지 않은 유저도 탐색만으로 플래너에 진입할 수 있게 됩니다.

**PM**: 이건 CRM 자동화 + 프론트엔드 작업이 섞여 있어요.
- **CRM 메시지 발송**: COM-2328의 조회 로그 기반, D+1 자동 발송 — 마케팅팀 셋업 1~2일
- **"탐색 아이템으로 플랜 만들기" 딥링크**: COM-1895의 "좋아요 아이템으로 플랜 생성" API를 확장하여, 최근 조회 아이템도 플랜에 추가 가능하게 — 프론트+백엔드 2~3일

**PO**: COM-2328 데이터를 COM-1895 기능에 연결하는 **"브릿지" 성격의 아이디어**네요. 후보에 넣겠습니다.

### 아이디어 4: "스팟 간 숨겨진 연결" — 교차탐색에 스토리를 입히기

**Marketer**: COM-2328이 교차탐색을 "응모권"이라는 외적 보상으로 유도한다면, **내적 동기로도 유도할 수 있는 방법**이 있어요.

마켓 스캔에서 확인한 Reddit 데이터를 보면, 유저들이 가장 좋아하는 건 **"현지인만 아는 조합"** 같은 정보입니다. 이걸 제품에 반영하면:

상품 상세페이지에 **"이 스팟의 Hidden Combo"** 섹션을 추가합니다.

```
[경복궁 상세페이지]

🔗 Hidden Combo
"경복궁에 오는 사람들이 잘 모르는 조합:
 경복궁 → 도보 5분 → 서촌 카페거리 → 도보 10분 → 통인시장 엽전 도시락

 이 3곳을 다 보면 🎰 추가 응모권!"
```

**Growth Manager**: 이건 COM-2328의 "3개 조회 → 응모권"에 **스토리라인을 붙이는** 거예요. 무작위로 3개를 보는 게 아니라, **"이 세 곳은 이런 이유로 같이 가면 좋다"**는 맥락이 생깁니다.

차이가 큽니다:
- **현재 COM-2328**: "아무 스팟 3개 보면 응모권" — 외적 보상만
- **Hidden Combo 추가 시**: "이 조합을 완성하면 응모권 + 현지인 팁" — 외적 보상 + 내적 동기(발견의 재미)

그리고 마켓 스캔의 SWOT에서 확인한 **"한국 특화 1st party 데이터가 유일한 지속 가능 경쟁 우위"**에 정확히 부합합니다. AI 플래너가 "경복궁 → 남산" 같은 뻔한 추천은 할 수 있지만, **"경복궁 → 서촌 → 통인시장 엽전 도시락"은 크리에이트립의 현지 데이터가 있어야 가능**해요.

**PM**: 개발 비용은?

**PM**: 이건 기술적으로는 **추천 영역의 변형**이에요. 상품 상세 하단에 "Hidden Combo" 섹션을 추가하고, 미리 큐레이션된 콤보 데이터를 노출합니다. 콤보 데이터 제작이 주요 공수(콘텐츠팀 작업)이고, 프론트 개발은 2~3일. 다만 **콤보 데이터를 만드는 데 콘텐츠팀 리소스가 많이 들어요** — 스팟별로 콤보를 설계해야 하니까.

**Marketer**: 처음부터 모든 스팟에 할 필요 없습니다. **Top 20 인기 스팟에만 Hidden Combo를 붙이면** 대부분의 트래픽을 커버해요. 20개 콤보 제작이면 콘텐츠팀 3~5일.

**PO**: "교차탐색에 맥락을 입히는" 아이디어로 후보에 넣겠습니다. COM-2328의 응모권 + Hidden Combo의 발견 재미가 합쳐지면 교차탐색 동기가 두 배가 됩니다.

### 아이디어 5: "여행 버디 매칭" — Layer 5(공유/재방문)의 빈 곳 채우기

**Marketer**: 빈 구간 중 Layer 5(공유/재방문)를 짚어보겠습니다.

Reddit 데이터에서 **"itinerary check please"(19.6%) + "is this realistic?"(14.7%) = 34.3%가 검증 니즈**라고 했잖아요. 이 유저들은 **자기 일정을 다른 사람에게 보여주고 피드백받고 싶어하는** 거예요.

만약 COM-1895로 만든 플랜을 **공유 가능한 링크로 만들어서**, 다른 유저나 커뮤니티에 올릴 수 있으면:

1. **공유한 유저**: "댓글이 달렸을까?" → 돌아와서 확인 = D+1 재방문
2. **본 유저**: "이 플랜 좋다, 나도 복사할까?" → 크리에이트립 유입 = D0 모수 확대
3. **Reddit에서**: "itinerary check" 글 대신 **크리에이트립 플랜 링크를 공유** → 오가닉 인지도 구축

**Growth Manager**: 이건 마켓 스캔의 Strategic Imperative 4번 "Reddit/SEO 오가닉 인지도를 최소 1%로 끌어올리기"와 직결됩니다. 현재 0.1%인 Reddit 인지도를 올리려면 **유저가 자발적으로 크리에이트립 링크를 공유하는 구조**가 필요해요.

플랜 공유가 되면, Reddit에서 이런 시나리오가 가능합니다:

```
[Reddit r/koreatravel]

유저 A: "Here's my 4-day Seoul itinerary, let me know if it's too packed!"
→ 크리에이트립 플랜 링크 첨부
→ 링크 클릭하면 크리에이트립 플래너 뷰 → 지도, 동선, 시간 다 보임

유저 B: "This looks great! I'm copying this to my plan."
→ "내 플래너에 복사" → 크리에이트립 가입 → D0 유입
```

**PM**: 2차 회의 로드맵에서 Phase 2에 "커뮤니티 × 플래너 공유 기능"이 있었는데, **공유 링크 생성 자체는 Phase 1에 가능한 규모**입니다.

- 플랜을 읽기 전용으로 볼 수 있는 공개 URL 생성 — 백엔드 2~3일
- 공유 페이지 (지도 + 스팟 리스트 + "내 플래너에 복사" CTA) — 프론트 3~4일
- "공유하기" 버튼 (플래너 내) — 프론트 0.5일

**PO**: 공유를 하면 **럭키드로우 추가 응모권**도 줄 수 있어요. COM-2328의 기존 구조가 이미 "공유 → 추가 응모권"이 있잖아요. 플랜 공유도 같은 메커니즘에 태울 수 있습니다.

**Growth Manager**: 그러면 응모권 획득 경로가 세 가지로 늘어나요:
1. **교차탐색** (COM-2328): 상세 3회 조회
2. **공유**: 플랜 링크 공유
3. **기존**: SNS 공유

세 가지 경로 모두 전략적으로 의미 있는 행동이고, **응모권이라는 하나의 보상 시스템에 모이니까 유저 입장에서 단순해요.**

**PO**: 후보에 넣겠습니다. Layer 5(공유)를 채우면서 D0 모수 확대(Reddit 인지도)까지 기여하는 아이디어입니다.

### 아이디어 6: "오늘의 발견" 푸시 — 앱 쿠폰과 결합한 일일 콘텐츠

**Marketer**: 앱 전용 쿠폰의 Layer 2가 **주간 갱신**이잖아요. 그런데 D1 리텐션은 **매일** 돌아오게 해야 합니다. 주간 쿠폰 사이의 6일을 채울 장치가 필요해요.

제안: **"오늘의 발견" 일일 푸시**. 매일 하나의 스팟/상품을 큐레이션해서 앱 푸시로 보냅니다.

```
[월] "🌸 벚꽃 시즌 숨은 명소: 석촌호수 야경" → 스팟 상세
[화] "🍜 현지인 맛집: 을지로 골목 냉면" → 스팟 상세
[수] "💰 오늘만 10% OFF: 남산타워 야경투어" → 앱 전용 한정 딜
[목] "🗺️ 이 주의 Hidden Combo: 성수 카페 3곳" → 교차탐색 유도
[금] "🎰 주말 럭키드로우 특별 경품!" → 럭키드로우
[토] "📋 이번 주 가장 많이 찜한 스팟 Top 3" → 좋아요 유도
[일] "📅 새로운 주간 쿠폰이 내일 도착해요!" → 앱 오픈 유도
```

**Growth Manager**: 이건 확정 태스크 3개 모두에 트래픽을 보내는 **"일일 배급 장치"** 역할을 합니다.

| 요일 | 연결되는 태스크 |
|------|-------------|
| 수(한정 딜) | 앱 전용 쿠폰 — 앱에서만 가능한 일일 딜 |
| 목(Hidden Combo) | COM-2328 — 교차탐색 유도 |
| 금(럭키드로우) | COM-2328 — 럭키드로우 참여 유도 |
| 토(인기 찜) | COM-1895/2320 — 좋아요 행동 유도 |

**PM**: 개발 비용은 거의 0에 가깝습니다. **앱 푸시 스케줄링만 하면 되고, 콘텐츠만 매주 준비하면 돼요.** 마케팅팀 운영 업무입니다.

**Marketer**: 맞아요. 다만 "매일 콘텐츠를 준비하는 운영 부담"이 있습니다. 이걸 줄이려면 **템플릿화**가 필요해요. 요일별 포맷을 고정하면(월=스팟, 화=맛집, 수=딜...) 콘텐츠 준비가 반복 작업이 됩니다.

**PO**: 운영 비용이 주요 이슈지만, 기술 비용은 낮으니 후보에 넣겠습니다.

### 아이디어 7: "여행 준비 퀘스트" — 게이미피케이션 통합 시스템

**Growth Manager**: COM-2328의 "3회 조회 → 응모권"은 본질적으로 **퀘스트 시스템**입니다. 이걸 확장하면 어떨까요?

```
[일일 퀘스트 — 매일 갱신]
├─ 🔍 스팟 3개 구경하기 → 럭키드로우 추가 응모권 (COM-2328)
├─ ❤️ 좋아요 1개 누르기 → 포인트 50P
└─ 📋 플랜에 스팟 1개 추가하기 → 포인트 100P

[주간 퀘스트 — 매주 월요일 갱신]
├─ 🎯 서로 다른 카테고리 상품 3개 탐색 → 앱 전용 5% 쿠폰

├─ ✈️ 첫 플랜 생성 → 앱 전용 10% 쿠폰

**PM**: 잠깐, 이건 규모가 너무 커요. 퀘스트 시스템 자체가 하나의 큰 기능이에요.

이 정도면 COM-2328 인프라 위에 **추가 2~3일**로 구현 가능합니다.

**PM**: 동의합니다. "퀘스트 시스템"이라는 거창한 이름보다, **"COM-2328의 행동-보상 루프를 좋아요와 플래너까지 확장"**으로 프레이밍하는 게 맞아요.

### 아이디어 8: "검색 결과 내 플랜 템플릿 배너" — Layer 0(유입) 강화

**PM**: 확정 태스크 3개가 **이미 크리에이트립에 온 유저**를 대상으로 한다면, **유입 자체를 늘리는 장치**도 필요합니다. 2차 회의에서 "검색 클릭이 일 950명으로 가장 큰 모수"라고 했어요.

이 검색 유저가 **시즌 키워드**(벚꽃, 봄, spring)를 검색할 때, 검색 결과 최상단에 **플랜 템플릿 배너**를 노출합니다.

```
[검색: "cherry blossom"]

┌─────────────────────────────────────┐
│ 🌸 벚꽃 시즌 3박4일 베스트 코스      │
│ 여의도 → 석촌호수 → 경복궁 → 남산   │
│ [내 플래너에 복사하기]               │
└─────────────────────────────────────┘

[일반 검색 결과...]
```

이건 2차 회의 캠페인 2("검색 → 플래너 자동 연결")를 구체화한 거예요.

**Growth Manager**: COM-1895와의 연결 포인트는: **템플릿을 복사하면 즉시 플래너에 5~7개 스팟이 추가**됩니다. 플래너에 스팟이 있으면 COM-2320의 바텀시트가 의미 있어지고, D-day 카운트다운(아이디어 2)도 시작할 수 있어요.

그리고 마켓 스캔에서 확인한 **"Cold Start 문제 해소"**에 직결됩니다. 빈 플래너를 보면 뭘 해야 할지 모르지만, 템플릿이 있으면 "여기서 수정하면 된다"는 시작점이 생깁니다.

**Marketer**: 시즌 키워드뿐 아니라, Reddit에서 가장 많이 검색되는 패턴에도 맞출 수 있어요:
- "seoul 3 days" → 서울 3일 템플릿
- "seoul busan" → 서울+부산 멀티시티 템플릿
- "korea first time" → 첫 한국 여행 템플릿

**PM**: 개발 비용은 검색 결과 상단에 조건부 배너 1개 추가 + 템플릿 복사 API (COM-1895의 플랜 생성 API 활용). 프론트 1~2일.

**PO**: 후보에 넣겠습니다. 검색이라는 가장 큰 모수에서 플래너로의 직통 경로를 만드는 거니까요.

### 아이디어 9: "좋아요 기반 개인화 추천 강화" — COM-1895 데이터의 2차 활용

**Growth Manager**: COM-1895로 좋아요 데이터가 플래너와 연결되면, **좋아요 패턴 자체가 유저 취향의 시그널**이 됩니다. 이걸 추천 영역에 반영하면 어떨까요?

현재 추천 영역이 "이 상품을 본 사람들이 같이 본 상품"이라면, 좋아요 기반으로 **"당신이 찜한 상품과 비슷한 스타일"** 추천을 추가할 수 있어요.

```
[상품 상세페이지]

📌 추천 (기존): "이 스팟을 본 다른 유저가 같이 본 곳"
📌 🆕 맞춤 추천: "당신이 찜한 [경복궁, 북촌]과 비슷한 곳" → 창덕궁, 인사동
```

**Marketer**: 이건 교차탐색을 강화하는 거니까 COM-2328과 시너지가 있어요. 추천이 더 개인화되면 → 클릭률이 올라가면 → 상세 조회가 늘어나면 → COM-2328의 "3회 달성"이 더 자연스럽게 일어납니다.

**PM**: 다만 이건 **추천 로직 변경이 필요해서 백엔드 공수가 상당**합니다. 좋아요 데이터 기반 collaborative filtering이나 content-based filtering을 구현해야 해요. Phase 1에 넣기에는 무거울 수 있습니다.

**Growth Manager**: 풀 스펙은 무겁지만, **경량 버전은 가능**해요. "좋아요한 스팟과 같은 지역/같은 카테고리의 인기 스팟"을 보여주는 건 **좋아요 데이터의 지역/카테고리 태그만 매칭하면 되니까** 기존 추천 인프라에 필터 하나 추가하는 수준이에요. 백엔드 1~2일.

**PO**: 경량 버전으로 후보에 넣겠습니다.

### 아이디어 10: "여행 중 모드" — Layer 4(구매+실행) 강화

**PM**: 플래너 당위성 회의에서 확인한 핵심 인사이트를 상기합시다:

> **Layer 3(플래너)를 거친 유저와 거치지 않은 유저의 결정적 차이는 "구매 후"에 나타난다. 플래너에 일정이 있는 유저는 여행 중에 크리에이트립에 들어올 이유가 있다.**

COM-1895로 플랜을 생성한 유저가 실제 여행을 시작하면, **플래너가 "여행 중 가이드"로 역할이 바뀌어야** 합니다. 현재는 여행 전에만 플래너를 쓰고, 여행 중에는 바우처 확인만 하고 끝이에요.

**"여행 중 모드"**: 여행 날짜가 되면 플래너 UI가 바뀝니다.

```
[여행 중 모드 — D-day 기준 자동 전환]

오늘 일정 (3/25 화요일)
├─ 09:00 경복궁 가이드 투어 ✅ 바우처 확인
├─ 12:00 광장시장 점심
│  └─ 💡 "근처 맛집 추천: 을지로 골목 냉면 (도보 8분)"
├─ 15:00 북촌한옥마을
│  └─ 💡 "지금 여유 시간이 있다면: 삼청동 카페 (도보 3분)"
└─ 18:00 남산타워 야경투어 ✅ 바우처 확인

[빈 시간 발견] 13:00~15:00 사이 2시간 여유
→ "이 시간에 갈 수 있는 곳:" [인사동] [종로 북카페]
```

**Marketer**: "여행 중 모드"는 마켓 스캔의 Ansoff Matrix에서 **Product Development의 핵심**이에요. "계획→검증→예약→실행"의 풀 플로우가 완성되면, 경쟁사가 단기간에 복제하기 어렵습니다.

**Growth Manager**: 리텐션 관점에서 보면, **여행 중 매일 플래너를 열게 되면 여행 기간(3~7일) 동안 매일 D+1 재방문이 발생**합니다. 여행 전 2~4주 + 여행 중 3~7일 = 총 3~5주 동안 지속적인 리텐션이에요.

**PM**: 다만 이건 개발 규모가 큽니다. "여행 중 모드" UI 전환 + 빈 시간 감지 + 실시간 주변 추천은 **Phase 2 이상**의 기능이에요.

경량 버전으로 Phase 1에 넣을 수 있는 건: **"오늘 일정" 뷰만 추가**. 여행 당일이면 플래너 최상단에 "오늘 일정"이 하이라이트되는 것. 이건 프론트 2~3일이면 됩니다. 빈 시간 추천은 Phase 2로.

**PO**: 경량 버전으로 후보에 넣겠습니다.

## 4. A등급 아이디어의 실행 설계 스케치

### A-1: 좋아요 스트릭

**연계 태스크**: COM-2328 (인프라), COM-1895/2320 (모수), 앱 쿠폰 (보상)

```
좋아요 1개: (변화 없음)
좋아요 2개: COM-2320 바텀시트 CTA 노출 (확정 태스크)
좋아요 3개: 🆕 "축하! 앱 전용 5% 할인 쿠폰 획득" 토스트
좋아요 5개: 🆕 럭키드로우 추가 응모권 + "탐험가" 라벨
```

**핵심**: COM-2328의 "N회 카운트 → 보상 지급" 로직을 재사용. 카운트 대상을 "상세 조회"에서 "좋아요 클릭"으로 변경하는 것.

**보상 연결**:
- 3개 달성 쿠폰 → 앱 전용 (T3 시너지)
- 5개 달성 응모권 → COM-2328 럭키드로우 (T2 시너지)

**예상 공수**: 프론트 1.5일 + 백엔드 1일 = **2.5일**

### A-2: 여행 D-day 카운트다운

**연계 태스크**: COM-1895 (플랜 날짜 데이터), 앱 쿠폰 (앱 전용)

```

### A-3: 탐색 하이라이트 리캡

**연계 태스크**: COM-2328 (조회 데이터), COM-1895 (플랜 생성)

## 5. A등급 아이디어의 통합 파이프라인

**PO**: A등급 5개를 확정 태스크 위에 얹으면 전체 그림이 이렇게 됩니다.

```
[유입 — Layer 0]
  검색 → A-8 플랜 템플릿 배너 → 플래너 Cold Start 해소
  앱 푸시 → A-6 "오늘의 발견" → 매일 앱 오픈

[탐색 — Layer 1]
  COM-2328 (확정): 상세 3회 → 응모권
  A-4 (B등급): Hidden Combo → 맥락 있는 교차탐색

[저장 — Layer 2]
  COM-1895/2320 (확정): 좋아요 → 플랜 연결
  A-1 좋아요 스트릭: 좋아요 3개 → 쿠폰, 5개 → 응모권

[계획 — Layer 3]
  COM-1895 (확정): 좋아요 아이템으로 플랜 생성
  A-3 탐색 리캡: 조회 아이템으로도 플랜 생성 (좋아요 안 한 유저 커버)

[구매 — Layer 4]
  앱 전용 쿠폰 (확정): 앱에서 구매 전환 유인

[재방문 — 크로스 레이어]
  A-2 D-day 카운트다운: 여행 전 매일 단계별 콘텐츠
  A-3 탐색 리캡: D+1 "어제 본 스팟 정리" 재방문 트리거
  A-6 "오늘의 발견": 매일 다른 콘텐츠로 앱 오픈
```

**Growth Manager**: 이 통합 파이프라인에서 **빠졌던 두 가지 빈 곳(Layer 0, Layer 5)이 채워집니다.**

- **Layer 0(유입)**: A-8(검색 템플릿 배너) + A-6(일일 푸시)
- **Layer 5(공유/재방문)**: B등급의 "플랜 공유 링크"(아이디어 5)가 Phase 1 후반에 추가되면 완성

그리고 **크로스 레이어 재방문 장치가 3개(D-day, 리캡, 일일 푸시)**로 되면, **유저가 어떤 단계에 있든 "내일 올 이유"가 하나 이상 존재**하게 됩니다.

| 유저 상태 | 내일 올 이유 |
|----------|-----------|
| 탐색만 한 유저 | A-3 탐색 리캡 ("어제 본 스팟 정리해줬어요") |
| 좋아요 누른 유저 | CRM 시나리오 1 + A-1 스트릭 ("2개만 더 누르면 쿠폰!") |
| 플랜 만든 유저 | A-2 D-day 카운트다운 ("D-10 환전 팁") |
| 앱 설치한 유저 | A-6 "오늘의 발견" + 주간 쿠폰 |
| 럭키드로우 참여 유저 | COM-2328 + CRM ("오늘의 럭키드로우") |

**Marketer**: 이건 2차 회의에서 PO가 말한 **"유저의 여행 단계와 의도에 맞는 Open Loop를 배치한다"**의 구체적 구현이에요. 확정 태스크가 레이어를 만들고, A등급 아이디어가 레이어 사이를 연결하고, CRM이 재방문을 트리거합니다.

> '좋아요 스트릭'은 '좋아요'라는 유저의 액션을 인위적으로 끌어올리는 의도를 가지고 있어 부적절한 것 같아. 상품의 매력도와 그로 인해 발생하는 유저의 관심에 따라 좋아요를 클릭하는 게 바람직할텐데 좋아요 수에만 집중해서 인위적으로 올리게 되면 좋아요 기능의 퍼포먼스(좋아요 기능의 구매 및 리텐션에 대한 리프트)에 악영향을 미칠 수 있다고 생각해

### 기존 논의와의 연관성

회의에서 **아이디어 1(좋아요 스트릭)**은 A등급 1번으로 채택되었으며, 핵심 논리는 다음과 같았습니다:

- **Marketer**: 좋아요 일 108명이라는 모수를 키워야 COM-2320 바텀시트 CTA 노출이 늘어남 (62~64행)
- **Growth Manager**: COM-2328 인프라를 재활용하며, Layer 1(탐색)과 Layer 2(저장)를 동시에 강화하는 게이미피케이션 시스템이 됨 (71행)
- **PO**: COM-2328 인프라 재활용 + COM-1895/2320 모수 확대 + 앱 쿠폰 시너지로 평가 (398행)

즉, 좋아요 스트릭의 채택 근거는 **"좋아요 행동 빈도 자체를 올려 하위 퍼널(플래너 전환, 구매)의 모수를 확대한다"**는 것이었습니다.

### 추가 논의 사항

**PO의 우려는 좋아요 기능의 "시그널 품질(signal quality)" 문제를 짚고 있습니다.**

좋아요 데이터는 이 리텐션 시스템에서 두 가지 역할을 동시에 수행합니다:

1. **퍼널 진입점**: 좋아요 → 플래너 연결 (COM-2320 바텀시트)의 트리거
2. **유저 취향 시그널**: 아이디어 9(좋아요 기반 개인화 추천), CRM 시나리오 1(좋아요 유저 D+1 리마인더)의 데이터 소스

좋아요 스트릭이 보상(쿠폰, 응모권)으로 좋아요를 유도하면, **"진짜 관심이 있어서 누른 좋아요"와 "쿠폰을 받기 위해 누른 좋아요"가 섞입니다.** 이로 인해:

- **개인화 추천 정확도 하락**: 쿠폰 목적의 무작위 좋아요가 섞이면, "좋아요 기반 추천"의 관련성이 떨어짐
- **CRM 메시지 효과 감소**: "좋아요한 상품이 할인 중!" 같은 CRM 메시지가 실제 관심 없는 상품에 대해 발송될 수 있음
- **플래너 전환률 희석**: COM-2320 바텀시트가 뜨더라도, 진짜 관심 없는 상품 기반이면 플래너 전환이 일어나지 않음. 모수는 늘지만 전환률이 떨어져 **절대 전환 수는 동일하거나 오히려 감소**할 가능성

이는 마케팅에서 흔히 발생하는 **"허수 지표(vanity metric)" 함정**과 유사합니다. 좋아요 수라는 선행 지표는 올라가지만, 좋아요가 실제로 기여해야 하는 후행 지표(구매 전환, 리텐션 리프트)에는 악영향을 줄 수 있습니다.

**반면, 현재 확정 태스크(COM-1895/2320)의 접근이 더 바람직한 이유:**

COM-2320은 유저가 **자발적으로 좋아요를 누른 시점에** 바텀시트를 보여주는 구조입니다. 이는 "유저의 관심이 확인된 순간"에 자연스럽게 플래너로 연결하는 것이므로, 좋아요 시그널의 품질이 보장됩니다. 모수를 키우고 싶다면, **좋아요 행동 자체를 인위적으로 늘리는 것이 아니라, 좋아요를 누를 만한 매력적인 상품/콘텐츠를 더 많이 노출하는 방향**이 맞습니다.

### 수정/보완이 필요한 기존 결론

1. **A-1(좋아요 스트릭)의 A등급 분류 재검토 필요**
   - "좋아요 빈도 ↑"가 곧 "D1 직접 기여도 높음"이라는 전제가 성립하려면, 인위적으로 늘어난 좋아요가 실제 전환과 리텐션에 동일한 리프트를 가져온다는 가정이 필요하나, 이는 의문
   - A등급에서 B등급으로 하향하거나, 좋아요 스트릭 대신 **좋아요 노출 기회를 늘리는 방향**으로 재설계 검토

2. **아이디어 7(행동-보상 루프 확장)의 "좋아요 3개 → 쿠폰" 부분도 동일한 우려 적용**
   - 287행에서 "확장 1: 좋아요 3개 클릭 → 앱 전용 쿠폰"으로 제안되었으나, 이 역시 좋아요 시그널 품질을 훼손할 수 있음

3. **대안적 접근 방향 제안**
   - **좋아요 노출 강화**: 좋아요 버튼의 가시성/접근성을 개선하여 자연스러운 좋아요 클릭을 유도 (UI/UX 개선)
   - **매력적인 콘텐츠 노출 확대**: 검색 결과, 홈 피드에서 유저 맥락에 맞는 상품을 더 잘 노출 → 자발적 좋아요 증가
   - **A-6(오늘의 발견)과 A-4(Hidden Combo) 활용**: 좋아요 스트릭 없이도, 매력적인 콘텐츠/스팟을 지속 노출하면 자연스럽게 좋아요 모수가 올라감
   - **좋아요 후 가치 강화**: 좋아요를 누른 뒤 받는 경험(COM-2320 바텀시트, 플래너 연결, 개인화 추천)의 품질을 높여, "좋아요를 누르면 좋은 일이 생긴다"는 학습 효과로 자연스러운 반복 유도
