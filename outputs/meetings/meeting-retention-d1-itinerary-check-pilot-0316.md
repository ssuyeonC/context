## 2. 새로운 시책 — "여행일정 검증" 파일럿

**PO**: Phase 1 핵심 태스크와 별도로, **플래너 전략의 새로운 축**을 하나 논의합니다. 플래너 당위성 회의(3/5)에서 합의한 핵심 원칙을 상기합시다.

> **"플래너는 '일정 짜는 도구'가 아니라 '일정 검증 도구'다. 이 리프레이밍이 '크리에이트립이 아니면 안 되는 이유'를 만든다."**

그때 로드맵에서 "일정 검증 라이트"를 즉시 실행 카테고리에 넣었지만, 구체적인 실행 형태는 미확정이었습니다. 오늘은 **이걸 "사람이 직접 검증해주는 파일럿"으로 시작하자**는 제안을 논의합니다.

### 2-1. Why — 왜 파일럿인가

**PM**: 노션 문서에 정리한 내용을 공유합니다. 핵심 데이터부터요.

| 근거 | 수치 |
|------|------|
| Reddit "내 계획 괜찮아?" (검증 요청) | **19.6%** |
| Reddit "이게 현실적이야?" (가능성 확인) | **14.7%** |
| 합산 검증 니즈 | **34.3%** — 추천 니즈(24.6%)보다 10pp 높음 |
| 일 플랜 생성 수 (현재) | ~34건 (3월 초 ~50건) |
| 검증 요청 예상 | 플랜 생성자의 30~50% → **일 10~25건** |

**Growth Manager**: 잠깐, 검증 요청 예상의 전제를 짚겠습니다. "플랜 생성자의 30~50%가 검증을 요청한다"는 가정인데, 이건 **Reddit에서 검증 질문을 올리는 비율(34.3%)**을 그대로 적용한 건가요?

**PM**: Reddit 데이터를 참고 수치로 쓴 건 맞습니다. 다만 Reddit에서 글을 올리는 행위와 크리에이트립에서 버튼 하나 누르는 행위는 마찰이 다릅니다. Reddit은 글을 작성하고 기다려야 하지만, 우리는 버튼 클릭 + 텍스트/파일 첨부로 끝이니까 **요청 마찰이 훨씬 낮아요.** 반면, 크리에이트립 플래너 자체의 사용자가 아직 일 34~50명이라는 모수 제한도 있고요.

**Growth Manager**: 그래서 현실적 예측은 **일 10~25건**이 적절하다고 봅니다. 운영 관점에서 1~2명이 처리 가능한 규모예요. 이 정도면 **파일럿으로 시작해서 수요와 효과를 검증하기에 적합한 스케일**입니다.

**PO**: 제가 이 시책을 파일럿으로 먼저 돌리려는 이유가 세 가지 있습니다.

**첫째, 검증 니즈의 실체를 확인해야 합니다.** Reddit 데이터에서 34.3%라는 숫자를 봤지만, "크리에이트립 안에서" 유저가 이 기능을 실제로 쓰는지는 별개의 질문이에요. Reddit에서 검증 글을 올리는 유저는 이미 일정 작성을 끝낸 상태이고, 크리에이트립에서 플랜을 만드는 유저의 여행 준비 단계가 같은지 모릅니다. 파일럿을 돌리면 **"실제 요청이 얼마나 오는가", "어떤 형태의 검증을 원하는가"**를 확인할 수 있어요.

**둘째, 검증 패턴 데이터를 축적해야 합니다.** 3/5 플래너 당위성 회의에서 "일정 검증 라이트"의 범위를 논의할 때, "빡빡함 경고 + 이동 시간 추정"을 최소 스펙으로 잡았어요. 그런데 이걸 시스템으로 만들려면 **"어떤 경우에 빡빡하다고 판단하는가", "유저가 가장 유용하다고 느끼는 피드백이 뭔가"**를 알아야 합니다. 사람이 수동으로 검증하면서 이 패턴을 모으면, 2단계(규칙 기반 자동화), 3단계(AI 검증)를 설계할 때 훨씬 정확해져요.

**셋째, 시장 선점 압박입니다.** 마켓 스캔(3/9)에서 SupTravel AI가 "AI Itinerary Checker"를 이미 유료로 운영 중이라고 확인했어요. 우리가 시스템을 완성할 때까지 기다리면 시장 인식이 "일정 검증 = SupTravel"로 굳어질 수 있습니다. **사람이 해주든 AI가 해주든, "크리에이트립에서 일정 검증을 해준다"는 인지를 먼저 심는 게 중요**해요. 파일럿이라도 빨리 시작하는 게 유리합니다.

**Marketer**: PO 의견에 강하게 동의합니다. 그리고 "사람이 직접 봐준다"는 것 자체가 **AI보다 강력한 신뢰 시그널**이에요. SupTravel AI가 알고리즘으로 검증하는 것과, "한국 여행 전문가가 직접 내 일정을 봤다"는 건 유저 인식에서 차원이 다릅니다.

r/koreatravel에서 유저들이 가장 신뢰하는 답변은 "한국에 살고 있는 사람"이나 "여러 번 가본 사람"의 답변이에요. **"크리에이트립 한국 전문가의 일정 검증"**이라는 포지셔닝은 AI 도구와 정면으로 차별화됩니다. 그리고 이 경험에 만족한 유저가 Reddit에서 "Creatrip checked my itinerary for free!"라고 쓰면, 현재 0.1%인 Reddit 인지도를 올리는 오가닉 채널이 됩니다.

**Growth Manager**: 한 가지 챌린지를 하겠습니다. **운영 비용 대비 효과의 지속 가능성**이에요. 파일럿이 일 10~25건이면 1~2명이 감당하지만, 봄 시즌에 플래너 모수가 올라가고 T1 효과로 진입률이 높아지면 요청이 급증할 수 있어요. 일 50건, 100건이 되면 사람으로는 감당이 안 됩니다.

**PO**: 정확한 지적입니다. 그래서 **파일럿의 목적이 "서비스를 영원히 수동으로 운영하겠다"가 아니라 "자동화를 위한 데이터를 모으면서 동시에 유저 가치를 제공하겠다"**인 거예요.

노션 문서에 정리한 3단계 로드맵이 이 구조입니다.

| 단계 | 방식 | 트리거 |
|------|------|--------|
| 1단계 (파일럿) | 사람이 직접 검증 (+ AI 보조) | 즉시 시작 |
| 2단계 | 반복 패턴 규칙 기반 자동화 (빡빡함 경고, 이동 시간 자동 계산) | 검증 패턴 데이터 축적 후 |
| 3단계 | AI/알고리즘 기반 자동 검증 + 사람 보완 | 2단계 효과 검증 후 |

파일럿에서 수집하는 데이터가 2단계 설계의 직접적 입력이 됩니다. "일정 과밀 경고가 전체 피드백의 60%를 차지한다"는 사실을 알면, 2단계에서 그 규칙부터 자동화하면 돼요. 그리고 요청이 감당 불가 수준으로 올라가는 건... 솔직히 **좋은 문제**입니다. 그때는 이미 수요가 검증된 상태니까, 자동화 투자의 정당성도 확보된 거예요.

## 3. 제품 설계 — 유저 플로우

**PM**: 구체적인 제품 구조를 논의하겠습니다.

### 3-1. 전체 유저 플로우

```
[유저 측]

1. 여행 플래너 인덱스 페이지(/itinerary) 진입
   └─ "여행 계획 검증하기" 버튼 노출

2. 버튼 클릭 → 검증 요청 화면 (페이지 또는 모달)
   ├─ 텍스트 입력: 자유 형식으로 여행 계획 작성
   ├─ 파일 첨부: 이미지, 스크린샷, PDF 등
   └─ 플랜 데이터 첨부: 크리에이트립에서 생성한 플랜 연동

3. Submit → "검증 요청이 접수되었습니다" 확인 화면
   └─ 예상 응답 시간 안내 (24시간 이내)

4. 검증 완료 후
   ├─ 푸시/이메일 알림: "여행 계획 검증이 완료되었어요!"
   └─ /itinerary 페이지에서 검증 결과 확인

[운영 측]

1. 유저 Submit → 슬랙 알림 (검증 요청 채널)
2. 어드민에서 검증 요청 확인 + 검토
3. 어드민에서 답변 입력 → 저장
4. 유저에게 알림 발송 + /itinerary에 답변 노출
```

**Marketer**: 유저 입력 부분에서 질문이 있어요. **"텍스트 입력"과 "플랜 데이터 첨부"가 병렬 옵션이면, 유저가 어떤 걸 선택할지**가 중요합니다.

크리에이트립 플래너에서 이미 플랜을 만든 유저는 플랜 데이터를 첨부하는 게 당연히 편해요. 하지만 **"아직 크리에이트립에서 플랜을 만들지 않은 유저"** — 구글 스프레드시트나 노션으로 일정을 짠 유저, Reddit에서 올리려던 일정을 가져온 유저 — 이 사람들은 텍스트 입력이나 파일 첨부(스크린샷)를 쓸 거예요.

**PM**: 맞습니다. 이게 사실 **전략적으로 중요한 분기점**인데요.

**분기 A: 크리에이트립 플랜이 있는 유저**
→ 플랜 데이터를 자동 첨부 → 검증 피드백이 플랜의 각 날짜/스팟에 매핑 → 피드백 반영이 쉬움 → "수정 → 재검증" 루프 활성화

**분기 B: 크리에이트립 플랜이 없는 유저**
→ 텍스트/파일로 업로드 → 검증 피드백은 일반 텍스트로 전달 → **그런데 여기서 기회가 생김**: 피드백을 줄 때 "이 일정을 크리에이트립 플래너에서 편하게 관리해보세요" → **플래너 전환 유도**

**Growth Manager**: 분기 B가 흥미롭습니다. 이건 사실상 **플래너의 새로운 진입 경로**를 만드는 거예요. 지금까지 논의한 플래너 진입 경로가 세 가지였잖아요 — 스팟 상세 "내 일정에 추가", 좋아요 리스트에서 플랜 만들기(T1), 검색 결과 템플릿 배너(A-8). 여기에 **"일정 검증을 받으러 왔다가 플래너에 정착하는 경로"**가 추가되는 겁니다.

```
[외부에서 일정을 짠 유저]
텍스트/이미지로 일정 업로드 → 검증 받음
  → "이 일정을 크리에이트립 플래너에 옮기면 더 편하게 관리할 수 있어요"
  → 플래너 생성 → 이제 이 유저의 여행 맥락이 크리에이트립에 존재
  → Lock-in 시작
```

**PO**: 이건 좋은 발견입니다. 하지만 파일럿 단계에서는 **기능 범위를 최소화**해야 합니다. "검증 결과를 보고 플래너로 전환"하는 유도는 자동화가 아니라 **검증 답변 텍스트에 플래너 링크를 포함**하는 수준으로 시작하면 됩니다. 시스템 개발 없이 운영으로 가능해요.

### 3-2. 입력 방식 상세

**PM**: 유저가 여행 계획을 제출하는 입력 방식을 정리합니다.

| 입력 방식 | 대상 유저 | 구현 | 우선순위 |
|----------|----------|------|---------|
| 텍스트 입력 (자유 형식) | 외부에서 일정을 짠 유저, 간단히 적고 싶은 유저 | 텍스트 에어리어 | **P0 — 필수** |
| 파일 첨부 (이미지/PDF) | 스크린샷, 구글 독스 PDF 등으로 일정을 만든 유저 | 파일 업로드 | **P1 — 있으면 좋음** |
| 플랜 데이터 첨부 | 크리에이트립 플래너에서 플랜을 생성한 유저 | 기존 플랜 선택 UI | **P0 — 필수** |

**Growth Manager**: P0이 텍스트 입력과 플랜 데이터 첨부 두 개라면, 최소한 이 두 가지로 런칭하고 파일 첨부는 이후에 추가하는 게 맞겠네요.

**PO**: 파일 첨부는 비용 대비 효과를 따져봐야 합니다. 파일 업로드 자체는 기술적으로 어렵지 않지만, **어드민에서 이미지/PDF를 열어서 해석하는 운영 비용**이 텍스트보다 높아요. 텍스트로 적힌 일정은 바로 읽을 수 있는데, 이미지는 OCR이 필요하거나 눈으로 읽어야 하니까요.

**PM**: 그런데 유저 관점에서 보면, 많은 유저가 이미 구글 스프레드시트나 노션에서 일정을 정리해놨어요. 이걸 다시 텍스트로 옮겨 적는 건 마찰이에요. **이미지 한 장 첨부**가 훨씬 편합니다. Reddit에서도 일정 리뷰 요청할 때 이미지로 올리는 유저가 상당수예요.

**Marketer**: 동의합니다. 유저의 "기존 도구에서 만든 일정을 가져오는" 경험을 최대한 쉽게 만들어야 검증 요청 자체의 진입 마찰이 줄어들어요. **텍스트 + 이미지 첨부 + 플랜 데이터, 세 가지 모두 P0으로 가져가는 게 맞다**고 봅니다. 파일 첨부 자체는 기술적으로 간단하고, 운영 부담은 일 10~25건 수준에서 감당 가능합니다.

**PO**: 근거가 타당합니다. **텍스트 입력 + 파일 첨부(이미지) + 플랜 데이터 첨부, 세 가지 모두 P0으로 확정**합니다. PDF는 이미지로 커버 가능하니 별도 처리 없이 이미지 업로드에 포함시킵시다.

## 5. 진입점 논의

**PO**: 검증 요청 버튼의 진입점을 논의합니다. 기본적으로 **/itinerary(여행 플래너 인덱스 페이지)에 두는 건 확정**이고, 추가 진입점이 필요한지가 논점입니다.

**Growth Manager**: 진입점 선택의 기준부터 세워야 합니다. 이 기능의 타겟 유저가 누구인지에 따라 진입점이 달라져요.

**타겟 유저 1: 이미 플래너에서 플랜을 만든 유저**
→ /itinerary 페이지에 이미 있으니, **여기에 진입점을 두면 자연스럽게 노출**

**타겟 유저 2: 아직 플랜은 없지만 여행을 준비 중인 유저 (스팟 탐색, 좋아요 등)**
→ /itinerary에 안 들어올 수 있음 → **다른 곳에 진입점 필요**

**타겟 유저 3: 외부에서 일정을 짠 유저 (구글 스프레드시트, Reddit 등)**
→ 크리에이트립에 유입되는 시점에 검증 기능을 알아야 함 → **스팟 홈, 검색 결과 등**

**PM**: 파일럿 단계에서는 **"이미 플래너에 있는 유저"를 우선 타겟**으로 잡는 게 맞아요. 이유가 두 가지입니다.

1. **플랜 데이터가 있으면 검증 품질이 높아집니다.** 텍스트만으로 일정을 해석하는 것보다, 구조화된 플랜 데이터(날짜, 스팟, 순서)가 있으면 담당자가 훨씬 정확하고 빠르게 검증할 수 있어요. 운영 효율 관점에서 중요합니다.
2. **파일럿의 모수 조절이 필요합니다.** 진입점을 넓히면 요청이 너무 많아질 수 있어요. 일 10~25건이 운영 가능 범위라고 했는데, 스팟 홈에 배너를 걸면 노출 모수가 몇 배로 뛰어서 감당이 안 될 수 있습니다.

**Marketer**: 동의하지만, 하나 추가 제안이 있습니다. **스팟 홈이나 스팟 리스트 화면에 "여행 계획, 괜찮은지 확인해볼까요?" 같은 가벼운 배너를 두되, 클릭하면 /itinerary로 보내는 구조**는 어떨까요? 직접 검증 요청 UI를 여는 게 아니라, 플래너 페이지로 안내하는 거예요.

이렇게 하면:
- 아직 플랜이 없는 유저에게 "일정 검증"이라는 가치를 인지시키고
- /itinerary로 유도하면서 플래너 자체의 인지도도 올리고
- 실제 검증 요청은 플래너 페이지에서만 발생하니 모수 조절도 가능

**Growth Manager**: 이건 사실 **플래너 진입률을 올리는 간접 레버**이기도 합니다. 3/5 플래너 당위성 회의에서 "진입점 확대"를 카테고리 A로 분류했잖아요. "여행 계획 검증"이라는 가치 제안 자체가 새로운 진입점 역할을 합니다.

```
[스팟 홈]
┌─────────────────────────────────────┐
│ 🔍 여행 계획, 괜찮은지 확인해볼까요?   │
│ 한국 여행 전문가가 직접 체크해드려요    │
│ [확인하러 가기 →]                     │
└─────────────────────────────────────┘
→ 클릭 시 /itinerary로 이동
```

**PO**: 좋은 설계입니다. 정리하면:

| 진입점 | 형태 | 동작 | 우선순위 |
|--------|------|------|---------|
| /itinerary (플래너 인덱스) | "여행 계획 검증하기" CTA | 검증 요청 화면/모달 오픈 | **P0** |
| 스팟 홈 | 배너 | /itinerary로 이동 | **P1** |
| 스팟 리스트 | 배너 | /itinerary로 이동 | **P1** |

**P0은 /itinerary 내 직접 CTA만**으로 런칭하고, **스팟 홈/리스트 배너(P1)는 파일럿 첫 데이터를 보고 추가 여부 결정**합시다. 요청이 너무 적으면(일 5건 미만) 진입점을 넓히고, 적당하면(10~25건) 유지합니다.

## 6. T1/T2/T3와의 시너지

**Growth Manager**: 여행일정 검증 파일럿이 기존 삼각 파이프라인과 어떻게 연결되는지 정리해야 합니다.

### 6-1. T1 (좋아요→플랜 추가)과의 시너지

**Growth Manager**: T1이 좋아요 유저를 플래너로 끌어들이잖아요. 이 유저가 플랜을 만들면 **검증 기능의 잠재 대상 모수가 커집니다.**

```
좋아요 클릭 → T1 바텀시트 "일정에 넣어볼까?" → 플랜 생성
  → /itinerary에서 "이 일정 괜찮은지 검증받아볼까요?" 노출
  → 검증 요청
```

T1이 플래너 진입률을 0.27%에서 올리면, 검증 요청 모수도 비례해서 올라갑니다.

### 6-2. T2 (럭키드로우×교차탐색)와의 시너지

**Marketer**: 검증 피드백에서 **"이 스팟 대신 이걸 추천"**이라는 가이드를 줄 때, 추천 스팟의 상세 링크를 포함하면 **교차탐색이 발생**합니다. 그리고 그 유저가 추천 스팟을 3개 보면 T2의 "3회 조회 → 응모권" 메커니즘이 작동해요.

### 6-3. T3 (앱 전용 쿠폰)과의 시너지

**PM**: 검증 결과를 **앱 푸시로 알려주면**, 검증 결과를 확인하러 앱을 열게 됩니다. 이건 앱 활성화에 기여하고, T3의 앱 전용 쿠폰 시스템과 맞물려요.

**PO**: 그리고 가장 중요한 시너지는 **리텐션 메커니즘** 자체입니다.

```
[기존 삼각 파이프라인]
앱으로 와(T3) → 더 봐(T2) → 일정에 넣어(T1) → 내일 또 와(D+1)

[+여행일정 검증 파일럿 추가]
앱으로 와(T3) → 더 봐(T2) → 일정에 넣어(T1) → 일정 검증 요청
  → 24시간 내 결과 도착 → D+1 재방문 (결과 확인)
  → 피드백 반영해서 수정 → 재검증 요청 (선택)
  → D+2~D+3 재방문
```

**검증 요청 → 결과 대기 → 결과 확인**이라는 플로우 자체가 **"내일 돌아와야 하는 이유"의 가장 직접적인 형태**입니다. Open Loop가 "빈 날짜" 같은 암묵적 트리거가 아니라, **"내 검증 결과가 아직 안 왔다"는 명시적 트리거**가 돼요. 이건 지금까지 논의한 어떤 Open Loop보다 강력할 수 있습니다.

**Growth Manager**: 맞습니다. 그리고 "수정 → 재검증" 루프가 반복되면 여행 전 2~4주 동안 **매번 돌아올 이유**가 생기고, 그 과정에서 스팟 탐색/좋아요/구매가 자연스럽게 발생합니다.
