# 운영팀 인터뷰 설계 — 파트너 예약 처리 실태 파악

**일시**: 2026-04-14
**참석자**: PO (제품 방향/전략 총괄), PM (세부 기획), Growth Manager (데이터/전환 판단), Marketer (고객 접점/커뮤니케이션)
**목적**: 내일(4/15) 진행할 내부 운영팀 인터뷰의 목적·질문·방법을 설계하고, 인터뷰 결과를 어떻게 활용할지 합의
**참고 자료**:
- 4/6 현황 진단 회의록 (운영 의존도 52.4%, 파트너 처리 19.2%)
- 4/7 타겟 선정 회의록 (Type A~D 분류, P1 대시보드 우선순위 확정)
- 파트너 어드민 대시보드 기획서 (requirements_partner-admin-dashboard.md)

**PO**: 대시보드 기획서까지 나왔고, 내일 운영팀 3분을 인터뷰합니다. 이 인터뷰를 그냥 "어떠세요?"로 시작하면 산으로 갑니다. 오늘은 세 가지를 확정합니다.

|---|------|---------|

| Q2 | **무엇을 물어야 하는가** | 인터뷰 질문지 확정 |

## 1. Q1 — 왜 운영팀 인터뷰가 필요한가

### 1-1. 데이터로 아는 것과 모르는 것

**Growth Manager**: 지금까지 데이터로 확인한 것과, 데이터만으로는 알 수 없는 것을 정리하겠습니다.

| 구분 | 데이터로 아는 것 (Fact) | 데이터로 모르는 것 (Gap) |
|------|----------------------|----------------------|
| **운영 규모** | 운영팀이 3개월간 11,833건(52.4%)을 수동 확정 | 이 작업에 하루 몇 시간이 걸리는지, 어떤 순서로 하는지 |
| **미사용 파트너** | Type D 14개+ 스팟, 2,793건이 승인율 0% | 운영팀이 이 파트너들에게 안내를 시도한 적이 있는지, 했다면 어떤 반응이었는지 |
| **자동확정** | 현재 22.7%(5,117건)가 자동확정 | 자동확정으로 전환 가능한데 아직 수동인 스팟이 있는지, 전환 기준이 무엇인지 |
| **파트너 어드민** | 파트너용 메뉴 5개 존재, 대시보드 없음 | 파트너가 어드민을 실제로 어떻게 쓰는지 운영팀이 관찰한 바 |
| **알림톡** | 12월 배포, 승인율 68~100% | 알림톡 미발송 파트너가 있는지, 미발송 사유가 무엇인지 |
| **대시보드 기획** | FR-1~8, BR-1~6 기획 완료 | 운영팀 관점에서 빠진 것이 있는지, 실제로 쓸지 |

**PM**: 정리하면, 인터뷰의 목적은 크게 **세 가지 가설을 검증**하는 것입니다.

| # | 가설 | 검증 방법 |
|---|------|---------|
| H1 | Type D 파트너는 어드민/알림톡 존재를 **모르거나**, 운영팀이 대신 해주니 **동기가 없어서** 안 쓴다 | 운영팀이 안내를 한 적이 있는지, 파트너의 반응은 어떠했는지 확인 |
| H2 | 운영팀의 수동 확정 업무는 **구조화된 루틴**이 있고, 그 루틴에 **비효율 구간**이 존재한다 | 실제 업무 프로세스를 처음부터 끝까지 들어보고, 어디서 시간이 많이 걸리는지 확인 |
| H3 | 대시보드가 있으면 파트너가 **직접 처리할 동기와 수단**을 갖게 된다 | 대시보드 기획서를 보여주고, 운영팀 관점에서 실효성을 평가받기 |

**PO**: 좋습니다. 이 세 가설이 인터뷰의 뼈대입니다.

## 2. Q2 — 무엇을 물어야 하는가

### 2-1. 인터뷰 질문지 설계 원칙

**PM**: 질문지 설계에 몇 가지 원칙을 정하겠습니다.

| 원칙 | 이유 |
|------|------|
| **열린 질문으로 시작, 구체 질문으로 수렴** | 운영팀만 아는 맥락을 놓치지 않기 위해 |
| **"왜"보다 "어떻게"를 먼저** | 업무 프로세스를 있는 그대로 듣기 위해. "왜 이렇게 하세요?"는 방어적 답변을 유도함 |
| **데이터를 먼저 보여주지 않기** | 데이터를 보여주면 답변이 데이터에 맞춰짐. 운영팀의 체감 먼저, 데이터 검증은 나중에 |
| **대시보드 기획서는 마지막에** | 기획서를 먼저 보여주면 "잘 만들었네요"로 끝남. 페인 포인트를 먼저 듣고, 기획서로 검증 |

**Marketer**: 동의합니다. 특히 **답변자의 경험 범위를 먼저 파악**하는 게 중요해요. 3분이 모두 같은 업무를 하는 건지, 각자 담당 파트너가 다른 건지에 따라 질문 해석이 달라집니다.

### 2-2. 질문지 — 영역별 구성

**PM**: 5개 영역으로 구성합니다. 총 소요 시간은 인터뷰 1건당 40~50분 목표입니다.

---

#### 영역 1. 도입 — 역할과 업무 범위 파악 (5분)

**목적**: 답변자가 보는 세계의 범위를 먼저 이해

| # | 질문 | 의도 |
|---|------|------|
| 1-1 | "예약 관련 업무에서 본인이 주로 담당하는 파트 가 어떻게 되세요?" | 수동 확정 외에 다른 업무(CS, 파트너 연락 등)도 하는지 파악 |
| 1-2 | "담당 파트너(스팟)가 정해져 있나요? 있다면 몇 개 정도 담당하세요?" | 파트너별 전담 여부에 따라 이후 질문의 맥락이 달라짐 |
| 1-3 | "하루 업무 시간 중 예약 확정 작업은 대략 어느 정도 비중이에요?" | 수동 확정이 전체 업무에서 차지하는 부담 체감 |

---

#### 영역 2. 수동 확정 프로세스 — 있는 그대로 (10~15분)

**목적**: H2 검증 — 구조화된 루틴과 비효율 구간 발견

| # | 질문 | 의도 |
|---|------|------|
| 2-1 | "예약이 들어왔을 때, 확정까지의 과정을 처음부터 끝까지 알려주세요. 어디서 시작해서 어디서 끝나나요?" | 전체 플로우를 사전 지식 없이 듣기 |
| 2-2 | "확정할 때 어떤 화면(도구)을 사용하세요? 어드민 페이지를 쓰시면 어떤 메뉴에서 하시는지요?" | 실제 사용 경로 파악. 기대와 다른 우회 경로가 있을 수 있음 |
| 2-3 | "한 건 확정하는 데 평균적으로 얼마나 걸려요? 가장 빨리 끝나는 경우와 오래 걸리는 경우는?" | 처리 시간의 분산과 병목 지점 |
| 2-4 | "하루에 몇 번 정도 확정 작업을 하세요? 아침에 몰아서 하시나요, 수시로 하시나요?" | 배치 처리 vs 실시간 처리 패턴 |
| 2-5 | "확정하면서 가장 번거롭거나 시간이 많이 드는 부분이 있다면?" | 직접적 페인 포인트 |
| 2-6 | "확정하기 전에 파트너에게 연락하는 경우가 있나요? 어떤 경우에?" | 단순 클릭이 아닌, 커뮤니케이션이 끼는 경우의 비중 |

**Growth Manager**: 2-6이 중요합니다. 만약 "확정 전에 파트너에게 카톡/전화로 확인하고 확정한다"면, 그건 **데이터에 '운영자 수동 확정'으로 잡히지만 실제로는 파트너와 소통이 포함된 건**이에요. 순수하게 운영팀이 단독 판단으로 확정하는 비율이 실제로는 다를 수 있습니다.

#### 영역 3. 파트너와의 관계 — Type D 가설 검증 (10~15분)

**목적**: H1 검증 — 미사용 파트너의 실제 사유

| # | 질문 | 의도 |
|---|------|------|
| 3-1 | "파트너가 직접 예약 확정을 하는 경우가 있나요? 어떤 파트너가 잘 하고, 어떤 파트너가 안 하나요?" | 운영팀이 체감하는 파트너 유형 분류 (데이터 분류와 비교) |
| 3-2 | "확정을 안 하는 파트너에 대해서, 직접 처리하도록 안내해본 적 있으세요?" | 안내 시도 여부 확인 — 핵심 질문 |
| 3-3 | *(안내한 적 있다면)* "어떤 방식으로 안내하셨어요? 반응은 어땠나요?" | 안내 방법과 파트너 반응 |
| 3-4 | *(안내한 적 없다면)* "안내하지 않은 특별한 이유가 있나요?" | 운영팀의 판단 기준. "우리가 해야 하는 업무"로 인식하는지 |
| 3-5 | "파트너가 어드민 사용법을 물어본 적이 있나요? 있다면 어떤 걸 물어봤나요?" | 파트너 관점에서의 어드민 사용성 이슈 |
| 3-6 | "파트너가 알림톡을 받고 있는지 알고 계세요? 알림톡을 안 받는 파트너가 있다면 왜 그런지 아시나요?" | 알림톡 미발송 사유 — 기술적 미설정 vs 의도적 거부 |
| 3-7 | "만약 파트너가 직접 확정을 많이 하게 되면, 본인 업무가 어떻게 달라질 것 같으세요?" | 운영팀의 기대와 우려 동시 파악 |

**Marketer**: 3-7이 미묘하지만 중요해요. 운영팀 입장에서는 "파트너가 직접 하면 내 일이 줄어든다"가 **긍정적이 아닐 수 있어요.** 업무가 줄면 역할이 줄어든다고 느낄 수 있고, 반대로 "그 시간에 더 가치 있는 일을 하고 싶다"일 수도 있어요. 이 질문으로 얼라인먼트를 확인해야 합니다.

**PO**: 좋은 포인트입니다. 인터뷰에서 **"우리가 이 업무를 줄이려는 건 당신의 역할을 줄이려는 게 아니라, 더 가치 있는 업무에 집중하게 하려는 것"**이라는 프레이밍이 필요합니다.

| 4-2 | "수동 확정 중에서 '사실 이건 자동으로 해도 될 텐데' 싶은 것이 있나요?" | 자동확정 확대 후보 직접 식별 |

| 4-4 | "날짜/시간 변경 요청이 들어오면 어떻게 처리하세요?" | 확정 외 관련 업무의 범위. 대시보드에 포함할 기능 판단 |

## 3. Q3 — 어떻게 진행하고 활용할 것인가

### 3-1. 인터뷰 포맷

**PM**: 3분을 어떻게 인터뷰할지 포맷을 정합니다.

| 항목 | 결정 |
|------|------|
| **방식** | 1:1 개별 인터뷰 (3회) |
| **시간** | 인터뷰당 40~50분 (영역별 시간 배분 상단 참고) |
| **장소** | 오프라인 or 온라인 (상황에 따라) |
| **진행자** | PM |
| **기록** | 노트 필기 (녹음은 동의 시에만) |

**Marketer**: 왜 그룹 인터뷰가 아니라 개별로 하나요?

**PM**: 세 가지 이유입니다.

| # | 이유 | 설명 |
|---|------|------|
| 1 | **솔직한 답변 확보** | 그룹에서는 선임 의견에 동조하는 경향. 개별이면 "사실 이 업무 좀 비효율적이에요"를 말하기 쉬움 |
| 2 | **담당 범위 차이 반영** | 3분의 담당 파트너/스팟이 다를 수 있음. 각자의 경험을 깊게 들어야 함 |
| 3 | **교차 검증 가능** | A가 "파트너에게 안내한 적 없다"고 하고, B가 "안내했는데 무시당했다"고 하면 — 둘 다 사실이고 둘 다 중요 |

**PO**: 동의합니다. 다만 3건의 인터뷰 결과를 정리해서 **교차 비교하는 시간**이 별도로 필요합니다.

### 3-2. 인터뷰 대상 선정 기준

**PM**: 운영팀 중 3분을 선정하는 기준입니다.

| 기준 | 이유 |
|------|------|
| **예약 확정 업무 직접 수행자** | 관리자/감독자가 아닌 실제 클릭하는 사람 |
| **근무 기간이 다양하면 좋음** | 신입(1~3개월) vs 경력자(6개월+)로 관점 차이 확보 |
| **담당 파트너 영역이 겹치지 않으면 좋음** | 다양한 파트너 유형에 대한 경험을 커버 |

**Growth Manager**: 가능하면 한 명은 **Type A 파트너(순시키, 아비쥬 등)를 담당하는 사람**, 한 명은 **Type D 파트너(휩드, 시현하다 등)를 담당하는 사람**이면 좋겠어요. 잘 되는 곳과 안 되는 곳의 운영 경험을 비교할 수 있으니까요.

### 3-3. 인터뷰 시 주의사항

**Marketer**: 인터뷰어(PM)가 유의할 점을 정리합니다.

| # | 주의사항 | 구체적 행동 |
|---|---------|-----------|
| 1 | **판단하지 않기** | "그건 비효율적이네요" ❌ → "그 부분이 시간이 많이 드는군요" ✅ |
| 2 | **유도하지 않기** | "대시보드가 있으면 좋겠죠?" ❌ → "이런 화면이 있다면 어떨 것 같으세요?" ✅ |
| 3 | **침묵 허용하기** | 5초 정도의 침묵은 기다림. 답변자가 생각을 정리하는 시간 |
| 4 | **구체적 사례 요청** | "최근에 그런 경우가 있었나요? 어떤 파트너였는지 기억나세요?" |
| 5 | **감사 전달** | "이 인터뷰가 실제 제품 개선에 바로 반영됩니다"를 시작과 끝에 전달 |

### 3-4. 결과 정리 및 활용 방식

**PO**: 인터뷰 결과를 어떻게 정리하고 활용할지 확정합니다.

| 단계 | 내용 | 기한 |
|------|------|------|
| **1. 원본 정리** | 인터뷰별 답변을 질문 영역별로 정리 | 4/15 (인터뷰 당일) |
| **2. 교차 비교** | 3명의 답변을 영역별로 교차 비교하여 공통점/차이점 도출 | 4/16 |
| **3. 가설 판정** | H1(인지·동기 부재), H2(루틴 비효율), H3(대시보드 실효성) 각각에 대해 지지/기각/수정 판정 | 4/16 |
| **4. 기획서 반영** | 인터뷰 결과를 바탕으로 대시보드 기획서 보완점 정리 | 4/17 |

**Growth Manager**: 3번 가설 판정 때, 데이터와 대조하면 좋겠어요. 예를 들어 운영팀이 "휩드는 안내를 한 적 있다"고 하면, 데이터에서 휩드가 그 이후에도 0%인지 확인해서 **"안내만으로는 부족하다"는 결론**을 내릴 수 있고요.

## 4. 인사이트 (사전 예상)

### Insight 1. "데이터의 빈칸"을 채우는 인터뷰여야 한다

데이터로 "52% 운영 의존, Type D 0%"는 알지만, **"왜"와 "어떻게"는 모른다.** 이 인터뷰의 가치는 숫자 뒤의 맥락을 채우는 것이다. 숫자를 재확인하는 질문은 낭비.

### Insight 2. 운영팀은 "문제의 전문가"다

PM이 데이터를 보고 세운 가설(H1~H3)이 틀릴 수 있다. **운영팀만 아는 여섯 번째, 일곱 번째 가설**이 있을 수 있다. 열린 질문(영역 2-5, 3-7)이 그 여지를 남겨둔다.

### Insight 3. 대시보드 검증은 "기능 목록" 검증이 아니다

영역 5에서 와이어프레임을 보여주는 목적은 "FR-1~8이 맞는지" 확인이 아니다. **"이 대시보드가 파트너의 행동을 바꿀 것인가"에 대한 운영팀의 직관**을 구하는 것이다. 운영팀이 "이거 있어도 안 쓸 것 같아요"라고 하면 그것이 가장 귀한 피드백.

### Insight 4. 운영팀의 "일이 줄어드는 것"에 대한 감정을 존중해야 한다

이 프로젝트의 성공은 운영팀의 협조에 달려 있다. 파트너에게 직접 처리를 안내하는 것도, 자동확정을 전환하는 것도 운영팀의 실행이 필요하다. **운영팀이 "이건 우리 일을 없애려는 거다"라고 느끼면 협조를 얻기 어렵다.** 인터뷰 프레이밍이 중요.
