## 1. 마켓 스캔 시사점 반영 — 오늘 논의에 미치는 영향

**Growth Manager**: 3/9에 완성된 여행 일정 플래닝 마켓 스캔에서 오늘 3개 태스크에 직접 영향을 주는 발견이 있습니다.

### 1-1. SupTravel AI의 "AI Itinerary Checker" — 속도가 중요

**Growth Manager**: 마켓 스캔에서 위협도 최고로 평가된 경쟁사가 **SupTravel AI**입니다. 이미 "AI Itinerary Checker"를 출시했고, 우리의 "일정 검증" 전략과 직접 겹칩니다. 한국 특화 데이터의 정확도에서는 아직 우리가 우위지만, **시간이 지나면 범용 AI도 데이터를 축적**합니다.

이것이 오늘 T1(좋아요→플래너 브릿지)에 주는 시사점은: **플래너 진입점 확대는 단순히 리텐션 레버가 아니라, "일정 검증" 기능의 모수를 키우는 전제 조건**이라는 겁니다. 마켓 스캔의 Strategic Imperative 1번이 "일정 검증 라이트를 최대한 빨리 출시"인데, 출시해도 쓰는 사람이 없으면 의미가 없어요. **T1이 플래너 모수를 키워야 일정 검증의 임팩트도 올라갑니다.**

### 1-2. Klook/KKday에 없는 것 = 우리의 차별화

**Marketer**: 마켓 스캔의 Porter's Five Forces에서 확인된 핵심: **"Switching cost를 만드는 것이 핵심"**입니다. 구매자 교섭력이 최고 수준(🔴)이고, 대체재 위협도 최고(🔴🔴)예요. 유저는 무료 대안(구글 스프레드시트, 노션, Reddit)으로 즉시 이탈 가능합니다.

이것이 T2(럭키드로우 × 교차탐색)에 주는 시사점: **럭키드로우의 "매일 올 이유"와 교차탐색의 "더 볼 이유"는 Klook/KKday에서 제공하지 않는 경험**입니다. Klook은 개별 상품 판매에 집중하지, "상품 N개를 더 보면 보상"이라는 게이미피케이션은 없어요. **이건 switching cost의 일부가 됩니다.**

### 1-3. 앱 전환의 거시적 맥락

**Growth Manager**: 마켓 스캔의 PESTLE에서 여행 앱 시장이 2026년 $16.23B → 2035년 $63.87B(CAGR 15.6%)으로 성장하고, **59% 여행 앱이 AR/VR 통합**을 시작했습니다. 앱은 단기적 전환율 개선뿐 아니라, 중장기적으로 푸시 알림, 오프라인 캐싱, 위치 기반 서비스 등 **웹에서 불가능한 Open Loop 메커니즘의 플랫폼**입니다.

T3(앱 전용 쿠폰)는 이 맥락에서 **앱 전환의 첫 번째 당근**이에요. 앱에서만 가능한 기능(푸시, 오프라인, 위치 기반)을 쓰게 하려면 먼저 앱으로 데려와야 하고, 쿠폰은 가장 직접적인 유인입니다.

**PO**: 마켓 스캔 시사점을 정리하면:

| 태스크 | 마켓 스캔 시사점 | 우선순위 영향 |
|--------|----------------|-------------|
| T1 (좋아요→플랜) | 일정 검증 선점 전략의 모수 확보 전제 조건 | **시급성 ↑** |
| T2 (럭키드로우×교차탐색) | Klook/KKday 대비 차별화 + switching cost 구축 | **차별화 가치 ↑** |
| T3 (앱 전용 쿠폰) | 웹 불가 Open Loop(푸시, 오프라인)의 전제 조건 | **전략적 기반 ↑** |

세 태스크 모두 마켓 스캔이 뒷받침합니다. 이제 하나씩 구체화합시다.

## 2. T1: 좋아요 페이지에서 플랜에 추가 + 모바일 좋아요 진입점 강화

### 2-1. Why — 데이터 근거 재확인

**Growth Manager**: 2차 회의 데이터를 재요약합니다.

| 지표 | 값 | 출처 |
|------|-----|------|
| 좋아요 리텐션 리프트 | **2.17x** (7개 행동 중 1위) | 2차 회의 §4-B |
| 좋아요 구매 전환 리프트 | **6.62x** (7개 행동 중 1위) | 2차 회의 §4-B |
| 좋아요 일 활성 사용자 | **~108명/일** | 2차 회의 §2-3 |
| 디바이스별 비중 | 앱 55% / 모바일웹 32% / 데스크탑 13% | 2차 회의 §2-3 |
| 3/1 피크 | 221명 (앱 128, 모바일웹 66, 데스크탑 27) | 2차 회의 §2-3 |
| 플래너 진입률 | **0.27%** (여행 방문자 대비) | 2차 회의 §4-B |
| L2→L3 전환 | **가장 높은 마찰 구간** — 발산→수렴 심리 전환 | 플래너 당위성 회의 §3 |

**PM**: 그리고 플래너 당위성 회의에서 합의한 핵심 프레임을 상기합시다.

> **좋아요(L2)는 "발산"** — 선택지를 넓히는 중, "이것도 괜찮나?"
> **플래너(L3)는 "수렴"** — 선택지를 좁히는 중, "이걸 언제 하지?"
> **이 심리 전환이 자연스럽게 일어나지 않는 것이 액션율 0.27%의 본질적 원인**

T1의 핵심 목적은 **이 마찰을 UI 수준에서 줄이는 것**입니다.

### 2-2. What — 기능 범위 확정

**PM**: T1은 두 가지 하위 기능으로 구성됩니다.

#### T1-A: 좋아요 페이지에서 플랜에 추가하는 기능

2차 회의 §7에서 합의한 내용을 구체화합니다.

**기능 1: 좋아요 리스트 내 개별 아이템 "일정에 추가" CTA**

```
[좋아요 리스트 페이지]
┌─────────────────────────────────────┐
│ ❤️ 남산타워 야경투어                   │
│ ★ 4.8 · ₩25,000 · 종로            │
│ [일정에 추가]  [비교함에 담기]         │
├─────────────────────────────────────┤
│ ❤️ 경복궁 가이드 투어                  │
│ ★ 4.9 · ₩15,000 · 종로            │
│ [일정에 추가]  [비교함에 담기]         │
├─────────────────────────────────────┤
│ ❤️ 한강 피크닉 세트                    │
│ ★ 4.6 · ₩35,000 · 여의도           │
│ [일정에 추가]  [비교함에 담기]         │
└─────────────────────────────────────┘
  [📋 찜한 3개로 플랜 만들기]  ← 상단 고정 CTA
```

**Marketer**: "찜한 X개로 플랜 만들기"가 핵심입니다. 이건 2차 회의에서 합의한 **"좋아요 리스트의 스팟들을 일괄로 플래너에 추가하는 원클릭 기능"**이에요. 유저가 직접 고른 스팟으로 만든 플랜이라 개인화도가 높고, 캠페인 1의 "템플릿 복사"보다 능동적입니다.

**Growth Manager**: 여기서 2차 회의의 합의 사항을 다시 짚겠습니다. **플랜이 없는 유저에게는 "비교하기"로 분기**하기로 했어요.

```
좋아요 리스트 페이지 진입 시:
├─ [플랜이 있는 유저] → "찜한 X개로 플랜에 추가" + 개별 "일정에 추가" CTA 활성화
└─ [플랜이 없는 유저] → "찜한 스팟 비교하기" CTA + 개별 "비교함에 담기" CTA 우선 노출
                       (+ "날짜 없이 일단 플랜 만들기" 서브 CTA도 함께 제공)
```

**PM**: 이 분기 로직이 중요합니다. 플랜이 없는 유저에게 "일정에 추가"를 강제하면, 플래너 당위성 회의에서 확인한 **3가지 전제 조건(여행 날짜 확정, 최소 선택 완료, 플래너 인지)** 중 1번과 3번이 충족되지 않아 전환이 일어나지 않습니다. 비교하기로 분기하면 "아직 결정 못 한 유저"도 다음 행동에 진입할 수 있어요.

**PO**: 동의합니다. 다만 **"날짜 없이 일단 플랜 만들기"도 서브 옵션으로 제공**하는 건 유지합시다. 2차 회의 §4에서 합의한 **경량 온보딩("날짜 없이 일단 추가")**이 L2→L3 마찰의 전제 조건 1(날짜 확정)을 제거하는 장치이니까요.

**기능 2: 좋아요 클릭 직후 바텀시트**

```
[유저가 스팟에서 좋아요 클릭]
     ↓
[바텀시트 등장]
┌─────────────────────────────────────┐
│ ❤️ 남산타워 야경투어를 찜했어요!       │
│                                     │
│ [플랜 있는 유저]                      │
│ "이 스팟, 여행 일정에 넣어볼까요?"     │
│ [일정에 추가하기]         [나중에]     │
│                                     │
│ [플랜 없는 유저]                      │
│ "이 스팟, 다른 찜한 곳과 비교해볼까요?" │
│ [비교하기]               [나중에]     │
│                                     │
│ 💡 찜한 스팟 3개 중 0개가 일정에 있어요 │
└─────────────────────────────────────┘
```

**PO**: 2차 회의에서 합의한 **"나중에" 3회 후 30일 미노출 가드레일**은 반드시 적용합니다.

#### T1-B: 모바일 환경에서 좋아요 클릭 후 좋아요 페이지로 이동하는 진입점 강화

**Marketer**: 이 부분은 2차 회의에서 직접 다루지 않았던 새로운 요소입니다. 데이터를 보면 **좋아요 행동의 87%가 모바일(앱 55% + 모바일웹 32%)**에서 발생합니다. 그런데 현재 모바일 환경에서 **좋아요를 누른 직후 좋아요 페이지로 가는 경로가 불편**해요.

**PM**: 현재 모바일 유저 플로우를 보면:

```
[현재 — 모바일]
좋아요 클릭 → 하트 애니메이션 → (끝, 같은 페이지에 머무름)
                                  → 좋아요 페이지로 가려면:
                                    앱: 하단 탭 → 마이페이지 → 좋아요 (3탭)
                                    모바일웹: 햄버거 메뉴 → 마이페이지 → 좋아요 (3탭)
```

**좋아요를 눌렀는데 좋아요 페이지까지 3번 탭해야 간다**는 건 마찰이 너무 큽니다. 좋아요를 누른 직후가 **"내가 찜한 것들을 한번에 보고 싶다"는 동기가 가장 높은 순간**인데, 이 순간에 쉬운 경로가 없어요.

**Growth Manager**: 이건 좋아요 리스트 재조회(#11 지표)에 직접적으로 영향을 줍니다. 2차 회의에서 PM이 짚었듯이, **좋아요를 누르는 것보다 "다시 돌아와서 리스트를 보는 행동"이 실제 리텐션**이에요. 진입점이 불편하면 리스트 재조회율이 낮을 수밖에 없습니다.

**PM**: 제안하는 개선입니다.

```
[개선 — 모바일]

방법 1: 좋아요 클릭 직후 바텀시트 (T1-A 기능 2와 통합)
  → 바텀시트 내 "내 좋아요 리스트 보기 →" 링크 추가
  → 1탭으로 좋아요 페이지 도달

방법 2: 좋아요 클릭 직후 토스트 메시지
  → "❤️ 찜 목록에 추가했어요 [보러 가기 →]"
  → 토스트의 "보러 가기" 클릭 시 좋아요 페이지로 이동
  → 1탭으로 좋아요 페이지 도달

방법 3: (앱 전용) 좋아요 카운트 배지
  → 하단 탭의 "찜" 아이콘에 빨간 배지 표시 (예: ❤️³)
  → 좋아요를 누를 때마다 배지 숫자 업데이트
  → 시각적 큐가 "찜 리스트가 여기 있다"는 인지를 지속적으로 제공
```

**Marketer**: 방법 1과 방법 3을 **조합**하는 게 최적이라고 봅니다.

- **방법 1(바텀시트)**: T1-A의 바텀시트에 "내 좋아요 리스트 보기" 링크를 추가하면, 별도 개발 없이 기존 바텀시트를 확장하는 것으로 충분해요.
- **방법 3(배지, 앱 전용)**: 바텀시트를 "나중에"로 닫은 후에도 하단 탭의 배지가 지속적으로 "찜 리스트가 있다"를 상기시켜요. 이건 앱에서만 가능한 것이고, **T3(앱 전용 쿠폰)과 연결**해서 앱 전환 동기를 강화합니다.

**Growth Manager**: 모바일웹에서는 바텀시트(방법 1)만 적용하고, 앱에서는 바텀시트 + 배지(방법 1+3)를 적용하면 앱의 UX 우위가 생기면서 자연스럽게 앱 전환 유인이 됩니다. **T3의 앱 전용 쿠폰과 함께 "앱에서 더 좋은 좋아요 경험"이라는 내러티브가 완성돼요.**

**PO**: 좋은 접근입니다. 정리하면:

| 환경 | 좋아요 클릭 직후 경험 | 좋아요 페이지 진입 |
|------|---------------------|------------------|
| 앱 | 바텀시트(플래너/비교 분기) + "좋아요 리스트 보기" 링크 | 하단 탭 배지(❤️³) → 1탭 도달 |

| 데스크탑 | 바텀시트(플래너/비교 분기) | 기존 GNB 유지 |

### 2-3. How — 기술 구현 범위

**PM**: 개발 범위를 분해합니다.

| 항목 | 범위 | 예상 공수 |

**PO**: 동의합니다. 구현 순서는 **앱 → 모바일웹 → 데스크탑** 순서로, 앱이 1주차에 완성되고 웹은 2주차에 따라갑니다.

### 2-4. Measure — 성공 지표

**Growth Manager**: T1의 성공을 판단하는 지표입니다.

**PO**: 맞습니다. **T1은 좋아요 활용 확장인 동시에, 플래너 인과효과 검증의 자연 실험**이기도 합니다. 이 점을 측정 설계에 반드시 반영합시다.

**전원**: T1 합의.

## 3. T2: 럭키드로우 페이지 이동 & 상품 교차탐색 강화

### 3-1. Why — 데이터 근거 재확인

**Growth Manager**: 2차 회의 §4-B와 §6 캠페인 5의 데이터입니다.

| 지표 | 값 | 의미 |
|------|-----|------|
| 교차탐색(상품 2개+ 탐색) 리텐션 리프트 | **1.63x** | 리프트×모수 = **19.0** (7개 행동 중 **임팩트 1위**) |
| 교차탐색 액션율 | **11.64%** | 모수가 압도적으로 큼 |
| 교차탐색 구매 전환 리프트 | **3.83x** | |
| 럭키드로우 리텐션 리프트 | **2.13x** | 좋아요(2.17x)와 거의 동급 |
| 럭키드로우 구매 전환 리프트 | **5.11x** | 높음 — 뽑기→탐색→구매 패턴 존재 |
| 럭키드로우 액션율 | **0.41%** | 인지도 부족, 진입점 제한 |

**PM**: 핵심 인사이트를 상기합시다. 2차 회의에서 Growth Manager가 정확히 짚었어요:

> **"럭키드로우가 교차탐색을 직접 유도하면, 임팩트 1위(교차탐색)와 리텐션 2위(럭키드로우)가 합쳐지는 것."**

그리고 마케터의 보충:

> **"'앞으로 N개만 더'라는 카운트다운 자체가 Open Loop. 듀오링고의 데일리 퀘스트와 같은 메커니즘."**

### 3-2. What — 기능 설계 확정

**PM**: 2차 회의 §6 캠페인 5의 합의안을 최종 실행 스펙으로 확정합니다.

#### 핵심 메커니즘: "서로 다른 스팟 상세페이지 3회 조회 → 럭키드로우 추가 응모권 1회"

**Growth Manager**: N값에 대해 2차 회의에서 "A/B 테스트로 3회 vs 5회 최적값을 찾자"고 했는데, **초기 런칭은 3회로 시작**하는 게 맞다고 봅니다.

이유:
1. **교차탐색 임계점이 "2개 이상"**입니다. 데이터에서 리텐션 리프트가 확인된 기준이 "상품 2개 이상 탐색"이에요. 3회면 이 임계점을 확실히 넘깁니다.
2. **5회는 진입 장벽이 높아** 달성률이 낮을 수 있어요. 초기에는 "쉽게 달성 → 성공 경험 → 습관화"가 중요합니다.
3. **3회 → 5회 상향은 언제든 가능**하지만, 5회 → 3회 하향은 유저 기대를 꺾을 수 있어요.

**Marketer**: 동의합니다. 그리고 **"서로 다른" 스팟**이라는 조건이 중요합니다. 같은 페이지를 3번 새로고침하는 어뷰징을 막으면서, **자연스럽게 교차탐색(서로 다른 상품 비교)을 유도**합니다.

**PO**: 3회로 확정합시다. A/B 테스트는 3회 vs 4회로 하되, 우선 3회로 런칭합니다.

**Marketer**: STEP 4에서 럭키드로우 페이지 하단에 **"최근 본 스팟"을 표시하고 플래너 연결을 제안**하는 건 T1과의 시너지입니다. 럭키드로우가 교차탐색을 유도하고, 교차탐색 결과가 플래너로 연결되는 **T2 → T1 퍼널**이 자연스럽게 형성돼요.

**Growth Manager**: 중요한 포인트입니다. **T2의 종착점이 럭키드로우로 끝나면 안 됩니다.** 럭키드로우는 **교차탐색의 동기**를 제공하는 장치이지, 그 자체가 목적이 아니에요. 럭키드로우 → 응모 → "자, 이제 이 스팟들 일정에 넣어볼까?" → T1의 플래너 연결. **T2는 T1의 모수를 키우는 상류 장치**입니다.

**PO**: 정확합니다. 전략 포트폴리오의 계층 구조로 보면:

```

     ↓

     ↓

     ↓
```

**T2 → T1은 상류→하류 관계이고, 이 파이프라인이 완성되면 "게이미피케이션 → 교차탐색 → 저장 → 계획"의 풀 퍼널이 작동합니다.**

**PM**: 동의합니다. 실행 타임라인을 정리하면:

## 4. T3: 앱 전용 쿠폰 시스템 도입

### 4-1. Why — 데이터 근거 + 전략적 맥락

**Growth Manager**: 앱 전환의 정당성은 2차 회의 §8에서 확인했습니다.

| 지표 | 앱 | 웹 | 배수 |
|------|-----|-----|------|
| RETURN_D1 구매 전환율 | **6.36%** | 2.77% | **2.3x** |
| 좋아요 행동 비중 | **55%** | 45% | 앱 트래픽 비중(20~25%) 대비 과대 참여 |
| 평균 상품 탐색 수 | 높음 | 낮음 | — |

**Marketer**: 그리고 마켓 스캔에서 확인한 거시적 맥락:
- 여행 앱 시장 CAGR 15.6% (2026~2035)
- 앱은 **웹에서 불가능한 Open Loop 메커니즘의 플랫폼**: 푸시 알림, 오프라인 캐싱, 위치 기반 서비스
- 2차 회의에서 합의한 앱 전환 배너 3종 (좋아요→지도, 탐색→결제, 플래너→리마인더)은 **"기능적 가치"를 말하는 방식**이지만, 즉각적 전환 동기로는 부족할 수 있음

**PM**: 2차 회의 §8에서 **"앱 깔아주세요가 아니라 앱에서 이걸 할 수 있어요"를 말하자**고 합의했습니다. 앱 전용 쿠폰은 이 원칙을 유지하면서 **즉각적인 금전적 인센티브를 추가**하는 겁니다. "앱에서만 쓸 수 있는 쿠폰이 있어요" — 이건 기능적 가치("앱에서 지도로 보세요")보다 훨씬 직접적인 전환 동기예요.

**PO**: 핵심 질문을 먼저 정리합시다. **앱 전용 쿠폰의 목적은 "앱 설치"인가, "앱 사용"인가?**

**Growth Manager**: **둘 다이지만 가중치가 다릅니다.** 설치만 시키고 앱에서 아무것도 안 하면 의미가 없어요. 쿠폰의 설계가 핵심입니다.

- **설치 유인**: "앱 설치 시 첫 구매 X% 할인" — 설치 자체를 트리거
- **사용 유인**: "앱에서만 사용 가능한 쿠폰이 매주 갱신" — 지속적 앱 사용 트리거

전자만 하면 "쿠폰 쓰고 삭제"가 됩니다. **후자를 포함해야 앱 리텐션이 올라가고, 앱의 높은 구매 전환율(6.36%)의 혜택을 지속적으로 받을 수 있어요.**

### 4-2. What — 쿠폰 시스템 설계

**PM**: 앱 전용 쿠폰을 세 가지 레이어로 설계합니다.

#### Layer 1: 앱 설치 즉시 쿠폰 (Welcome Coupon)

| 항목 | 설계 |
|------|------|
| 대상 | 앱 신규 설치 유저 (최초 1회) |
| 혜택 | 첫 구매 **10% 할인** (최대 ₩5,000) |
| 유효기간 | 설치 후 **7일** |
| 목적 | 앱 설치 → 첫 구매 전환 유도 |
| 노출 | 앱 첫 실행 시 풀스크린 모달 + 쿠폰함에 자동 추가 |

**Growth Manager**: 유효기간 7일은 **여행 전 계획 기간**을 고려한 겁니다. 플래너 당위성 회의에서 확인했듯이, **여행 전 2~4주가 구매의 90%가 일어나는 구간**이에요. 7일이면 이 기간 중 상당 부분을 커버합니다.

**Marketer**: 10% 할인의 경제성은 어떨까요?

**Growth Manager**: 앱 유저의 구매 전환율이 웹 대비 2.3배라는 걸 감안하면, **앱으로 전환시킨 유저의 LTV가 웹 유저보다 높을 가능성이 큽니다.** 10% 할인은 첫 구매의 margin을 깎지만, 이후 반복 구매에서 회수됩니다. 정확한 ROAS는 런칭 후 측정해야 하지만, 업계 일반적으로 앱 전환 쿠폰의 ROI는 양호합니다.

#### Layer 2: 앱 전용 주간 쿠폰 (Weekly App-Only Deal)

| 항목 | 설계 |
|------|------|
| 대상 | 앱 설치 유저 전체 |
| 혜택 | 매주 월요일 갱신되는 **앱 전용 할인 쿠폰 1~2장** |
| 유효기간 | 해당 주 일요일까지 (**7일**) |
| 목적 | **매주 앱을 여는 이유** = 주간 Open Loop |
| 노출 | 매주 월요일 푸시 알림 + 쿠폰함 갱신 표시 |

**Marketer**: 이게 **앱 전용 쿠폰의 핵심**입니다. Layer 1은 설치 1회성이지만, Layer 2는 **"매주 앱에서만 받을 수 있는 쿠폰이 있다"는 반복 동기**를 만들어요. 듀오링고의 스트릭이 "매일 하러 와"라면, 앱 전용 주간 쿠폰은 **"매주 확인하러 와"**입니다. 여행 플랫폼의 사용 주기에 맞는 더 현실적인 빈도예요.

**PM**: 주간 쿠폰의 상품 선정이 중요합니다. 매주 다른 카테고리의 상품을 쿠폰 대상으로 하면 **교차탐색도 유도**할 수 있어요.

| 주차 | 쿠폰 대상 예시 | 교차탐색 효과 |
|------|--------------|-------------|
| Week 1 | 투어/액티비티 5% 할인 | 투어 상품 탐색 유도 |
| Week 2 | 뷰티/스파 10% 할인 | K-Beauty 탐색 유도 (마켓 스캔: 뷰티 4.4%) |
| Week 3 | 교통 패스 ₩1,000 할인 | 교통 상품 탐색 유도 |
| Week 4 | 전체 상품 7% 할인 | 자유 탐색 유도 |

**Growth Manager**: 이건 T2(교차탐색 강화)와 시너지가 있습니다. **주간 쿠폰이 특정 카테고리를 하이라이트하면 유저가 그 카테고리를 탐색하게 되고, 탐색 과정에서 T2의 "3회 조회 → 추가 응모권" 메커니즘이 작동**해요.

```
앱 전용 주간 쿠폰 "이번 주 뷰티/스파 10% 할인"
  → 유저가 뷰티 상품 탐색 시작
  → T2: "3개 더 보면 추가 응모권!" 툴팁 등장
  → 유저가 3개 뷰티 상품 조회
  → 럭키드로우 추가 응모권 획득
  → 럭키드로우 응모 + "이 스팟들 일정에 추가?" (T1 연결)
```

**T1 × T2 × T3의 삼각 시너지가 여기서 발생합니다.**

**PO**: 이 시너지 설계가 오늘 회의의 가장 중요한 발견입니다. 세 태스크가 독립적이 아니라 **파이프라인으로 연결**되어야 임팩트가 극대화됩니다.

#### Layer 3: 조건부 쿠폰 (Behavior-Triggered Coupon)

| 항목 | 설계 |
|------|------|
| 대상 | 앱 내 특정 행동 완료 유저 |
| 혜택 | 행동별 맞춤 쿠폰 |
| 목적 | **핵심 행동을 쿠폰으로 인센티브** |

**PM**: 구체적인 행동-쿠폰 연결:

| 행동 | 쿠폰 | 근거 |
|------|------|------|
| 앱에서 좋아요 3개+ 누르기 | 찜한 상품 중 1개 5% 할인 | 좋아요 액션율 상향 + T1 모수 확대 |
| 앱에서 플래너에 스팟 1개+ 추가 | 플래너 내 상품 7% 할인 | 플래너 진입 유도 + 플래너 당위성 회의 Go/No-Go 지원 |
| 앱에서 럭키드로우 3일 연속 참여 | 특별 할인 쿠폰 (10%) | 럭키드로우 리텐션 + T2 강화 |
| 앱에서 첫 리뷰 작성 | 다음 구매 5% 할인 | UGC 생성 유도 |

**Marketer**: Layer 3은 초기에는 **첫 두 가지(좋아요 3개+, 플래너 스팟 추가)**만 런칭하고, 나머지는 데이터를 보고 추가하는 게 맞아요. 한꺼번에 많은 조건부 쿠폰을 풀면 유저가 복잡하게 느낄 수 있습니다.

**PO**: 동의합니다. Layer 3 초기 런칭은 **2개만**으로 제한합니다.

### 4-3. 웹→앱 전환 유도 설계

**Marketer**: 앱 전용 쿠폰이 있어도 **웹 유저가 그걸 모르면 소용없습니다.** 2차 회의 §8의 앱 전환 배너와 결합한 유도 설계가 필요합니다.

| 상황 | 웹 유저에게 보여주는 메시지 | 위치 |
|------|--------------------------|------|
| 상품 상세에서 가격 확인 중 | "앱에서 구매하면 10% 할인! 앱 전용 쿠폰 →" | 가격 옆 인라인 배지 |
| 좋아요 클릭 직후 (T1 바텀시트) | "앱에서 찜하면 할인 쿠폰도 받아요 → 앱 다운로드" | 바텀시트 하단 |
| 결제 페이지 | "앱에서 결제하면 즉시 할인! 앱 전용 쿠폰 확인 →" | 결제 버튼 위 배너 |
| 좋아요 리스트 (모바일웹) | "앱에서 좋아요 3개 누르면 할인 쿠폰 받기 →" | 리스트 상단 배너 |

**PM**: 결제 페이지 배너가 가장 전환율이 높을 거예요. **"지금 결제하려는 금액에서 할인받을 수 있다"는 메시지는 즉각적인 금전적 동기**니까요.

**Growth Manager**: 다만 결제 페이지에서 앱 이동을 유도하면 **결제 퍼널이 끊길 수 있어요.** 웹에서 결제하려던 유저가 앱 설치하러 갔다가 돌아오지 않을 수 있습니다.

**PO**: 좋은 챌린지입니다. **결제 페이지 배너는 "앱 미설치 유저"에게만 노출**하고, 메시지를 **"다음 구매부터 앱 전용 할인을 받으세요"**로 변경합시다. 지금 결제를 방해하지 않으면서 다음 구매에서 앱 전환을 유도하는 구조예요.

**Marketer**: 동의합니다. 수정하겠습니다.

| 상황 | 수정된 메시지 |
|------|-------------|
| 결제 페이지 (앱 미설치 유저) | "다음 구매부터 앱 전용 할인! 지금 앱 설치하면 10% 쿠폰 →" |

### 4-4. How — 기술 구현 범위

**PM**: 개발 범위 분해입니다.

| 항목 | 범위 | 예상 공수 |
|------|------|----------|
| 쿠폰 시스템 백엔드 (쿠폰 생성, 유효기간, 사용 처리, 앱 전용 제한) | 백엔드 — 기존 쿠폰 시스템 확장 or 신규 | 3~5일 |
| Welcome Coupon 발급 로직 (앱 첫 설치 감지 + 자동 발급) | 백엔드 + 앱 | 1일 |
| 앱 내 쿠폰함 UI (보유 쿠폰 리스트, 유효기간, 사용 조건 표시) | 앱 프론트 | 2~3일 |
| 주간 쿠폰 자동 갱신 로직 (매주 월요일 새 쿠폰 발급 + 만료 처리) | 백엔드 — 스케줄러 | 1~2일 |
| 주간 쿠폰 푸시 알림 ("이번 주 앱 전용 쿠폰이 도착했어요!") | 앱 + 백엔드 | 1일 |
| 조건부 쿠폰 트리거 (좋아요 3개+ / 플래너 스팟 추가) | 백엔드 — 이벤트 기반 쿠폰 발급 | 2일 |
| 웹→앱 전환 배너 (상품 상세, 바텀시트, 결제 페이지) | 웹 프론트 | 1~2일 |
| 결제 시 쿠폰 적용 로직 (앱 여부 체크 + 쿠폰 차감) | 백엔드 — 결제 플로우 연동 | 2일 |
| **총 예상 공수** | | **13~17일** |

**Growth Manager**: T3는 T1, T2보다 공수가 큽니다. 쿠폰 시스템 자체가 백엔드 작업이 많아요.

## 9. 최종 합의 사항

**PO**: 오늘 논의를 정리합니다.

### 합의 1: 세 태스크의 실행 순서와 근거

> **T2(럭키드로우×교차탐색) → T1(좋아요→플래너) → T3(앱 전용 쿠폰)**
> T2가 교차탐색 모수를 키우고, T1이 그 모수를 플래너로 연결하며, T3가 앱이라는 최적 환경으로 전체를 감싸는 구조.

### 합의 2: 삼각 파이프라인

> **"앱으로 와(T3) → 더 봐(T2) → 일정에 넣어(T1) → 내일 또 와(RETURN_D1)"**
> 세 태스크는 독립적 가치도 있지만, 파이프라인으로 연결될 때 임팩트가 극대화된다.

### 합의 3: 마켓 스캔 반영

> SupTravel AI의 "AI Itinerary Checker" 선점 위협에 대응하여, T1은 플래너 모수 확보의 전제 조건으로서 시급성이 높다.
> Klook/KKday에 없는 경험(게이미피케이션×교차탐색)으로 switching cost를 구축하는 것이 T2의 차별화 가치다.

### 합의 4: 타임라인

> Week 1 T2 착수 → Week 2 T2 배포 + T1 착수 → Week 3 T1 배포 + T3 착수 → Week 4 T3 배포 + 4/7 Go/No-Go
> **4월 첫째 주에 3개 태스크 모두 라이브, 플래너 당위성 회의의 Go/No-Go 판단에 T1 초기 데이터 제공 가능.**

### 합의 5: 원칙

> **"각 태스크는 독립적으로도 가치가 있어야 한다. 파이프라인의 한 단계가 실패해도 나머지는 단독으로 RETURN_D1에 기여할 수 있어야 한다."**
> - T2 단독: 교차탐색율 18%+ → 임팩트 1위 행동 강화
> - T1 단독: 좋아요 활용도 향상 + 플래너 인과관계 자연실험
> - T3 단독: 앱 유저 비중 28%+ → 높은 구매 전환율 유저 풀 확대
