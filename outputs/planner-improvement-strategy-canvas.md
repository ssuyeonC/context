# Product Strategy: Creatrip 플래너 기능 개선

**Date**: 2026-03-06
**Stage**: Growth (기존 기능의 전략적 리포지셔닝 + 단계적 확장)
**Context**: RETURN_D1 전략의 장기 Lock-in 핵심 축으로서의 플래너 재정의

---

## 1. Vision

**"여행자의 불안을 확신으로 바꾸는, 한국 여행의 유일한 일정 검증 도구."**

현재 전 세계 한국 여행자의 34.3%가 "내 일정이 괜찮은지" 검증받기 위해 Reddit에 글을 올리고 6~24시간 댓글을 기다린다. 크리에이트립 플래너는 이 검증을 즉시 해결함으로써, "일정을 짜는 도구"가 아닌 **"일정을 검증하는 도구"**로 포지셔닝한다. 유저가 일정을 수정하고 재검증하는 루프 안에서 크리에이트립은 여행 준비의 중심이 된다.

---

## 2. Target Segments

| Segment | Size | Pain Level | Current Alternative | Priority |
|---------|------|-----------|-------------------|----------|
| **"일정 짜는 중" 유저** — 여행 확정, 날짜 있음, 스팟 결정 단계 | 일 ~50명 (3월 기준) | **최고** — "빡빡한지", "동선이 맞는지" 불안 | Reddit itinerary check, 구글 스프레드시트, 노션 | **1순위** |
| **"찜만 하는" 유저** — 탐색 중, 좋아요 누르지만 일정 미확정 | 일 ~108명 | 높음 — 후보는 많은데 정리가 안 됨 | 좋아요 리스트에서 머무름, 다른 플랫폼으로 이탈 | **2순위** |
| **서울+부산 멀티시티 여행자** — 3~4일에 2개 도시를 시도 | Reddit 100건/8,349건 (서울만 여행자와 동수) | **최고** — "too packed", "too ambitious" 불안 | Reddit 질문, 여행사 패키지 | **3순위** |
| **재방문 여행자** — 이전 여행 맥락이 크리에이트립에 존재 | 전체 방문의 ~40% (RETURN_D1+D7+D30) | 중간 — 새로운 코스 탐색 | Klook, KKday, 직접 검색 | 4순위 |

**Primary segment**: 여행 날짜가 확정되고 스팟을 선택하기 시작한 "계획 단계 유저". 이 유저가 플래너에 들어오면 리텐션 리프트 1.72x, 구매 전환 리프트 3.49x가 발생한다.

**Explicitly not serving (지금은)**:
- "그냥 구경하는" 1회성 블로그 유입 유저 — 이 유저는 교차탐색 루프, 콘텐츠 시리즈로 별도 대응 (플래너까지의 인지적 거리가 너무 멂)
- 풀 자동화 AI 여행 플래너를 원하는 유저 — Phase 2 이후 검토

## 3. Pain Points & Value Created

### Pain 1: "내 일정이 현실적인지 모르겠다" (Reddit 34.3%)
- **현재 비용**: Reddit에 게시 → 6~24시간 대기 → 피드백 반영 → 재게시 → 반복
- **플래너 제공 가치**: 즉시 피드백 ("3일차 5개 스팟 과밀", "경복궁→남산 이동 40분") → 수정-재검증 루프가 Open Loop로 작동

### Pain 2: "플래너에 뭘 넣어야 할지 모르겠다" (Cold Start)
- **현재 비용**: 빈 플래너 화면 → 이탈 (진입률 0.27%의 핵심 원인)
- **플래너 제공 가치**: 멀티시티 템플릿("서울+부산 3~4일"), 좋아요→플래너 브릿지로 Cold Start 해소

### Pain 3: "좋아요 리스트와 일정이 분리되어 있다" (L2→L3 마찰)
- **현재 비용**: 좋아요(발산)에서 플래너(수렴)로의 심리적 전환 비용이 큼 → 액션율 0.27%
- **플래너 제공 가치**: "찜한 3개로 플랜 만들기" 원클릭 브릿지 → 발산→수렴 전환의 마찰 제거

### Pain 4: "구매 후 크리에이트립에 돌아올 이유가 없다"
- **현재 비용**: 바우처 확인 후 이탈 → 다음 여행 시 Klook/KKday로 유출
- **플래너 제공 가치**: 여행 중 "내 일정 확인", "다음 스팟 확인", "빈 시간 뭐 하지?" → 여행 전~중 지속적 재방문

## 4. Value Propositions

**For "일정 짜는 중" 유저**:
When 한국 여행 일정을 세우는데 빡빡한지 확신이 없을 때, they want 누군가가 "이 일정 괜찮아"라고 확인해주길, so they can Reddit 대기 없이 즉시 검증받고 안심하며 여행을 준비할 수 있다.

**For "찜만 하는" 유저**:
When 좋아요한 스팟이 5개 이상 쌓였는데 일정으로 정리가 안 될 때, they want 찜한 것들을 날짜별로 자동 배치해주길, so they can "이것도 좋아, 저것도 좋아"의 발산 상태에서 "언제 뭘 하지"의 수렴 상태로 자연스럽게 전환할 수 있다.

**For 서울+부산 멀티시티 여행자**:
When 3~4일에 서울과 부산을 모두 돌고 싶은데 가능한지 불안할 때, they want 이동 시간과 동선이 사전 검증된 템플릿을, so they can "too packed" 불안 없이 멀티시티 여행을 확신 있게 계획할 수 있다.

## 5. Strategic Trade-offs

| We Choose | Over | Because |
|-----------|------|---------|
| **"일정 검증 도구"로 리프레이밍** | "일정 생성 도구"로 유지 | 구글 스프레드시트/노션도 일정 "생성"은 가능하지만, "검증"은 크리에이트립만 가능 → 차별화의 원천 |
| **진입점 확대(카테고리 A) 우선** | 플래너 내부 UX 고도화 우선 | 0.27% 진입률에서 내부를 아무리 개선해도 모수가 부족. 먼저 유저를 데려와야 내부 개선의 가치가 생김 |
| **단계적 투자 (Go/No-Go 게이트)** | 대규모 일괄 투자 | 플래너의 인과 효과(사용 → 리텐션)가 아직 완벽히 검증되지 않음. 4주 후 데이터로 투자 확대 여부 결정 |
| **다채널 Open Loop 포트폴리오** | 플래너 단일 의존 | 플래너는 "왕관"이지 "유일한 무기"가 아님. 교차탐색·럭키드로우·비교하기·콘텐츠 시리즈가 플래너에 도달하지 않는 유저도 잡음 |
| **라이트 버전 일정 검증 먼저** | 풀 스펙(휴무일·영업시간·맥락 피드백) | 라이트(빡빡함 경고 + 이동 시간 추정)만으로도 Reddit 34.3% 수요의 핵심을 커버. 풀 스펙은 효과 검증 후 |
| **날짜 없이도 추가 허용** | 날짜 필수 전제 유지 | L2→L3 전환의 전제 조건 3가지(날짜 확정, 최소 선택 완료, 기능 인지) 중 1번 제거 → 마찰 감소 |

## 6. Key Metrics

### North Star Metric
**플래너 활성 사용자의 D+1 재방문률** — 플래너가 실제로 유저를 "다시 돌아오게" 만드는지의 직접적 지표

### Input Metrics (5 levers)

| Lever | 현재 | 4주 후 Go 기준 | 최종 목표 |
|-------|------|---------------|----------|
| **플래너 진입률** (여행 방문자 대비) | 0.27% | **0.5%+** | 1.0%+ |
| **일 플랜 생성 수** | ~34건 (2월 평균) | **50건+** (시즌 보정) | 100건+ |
| **"일정 검증" 사용률** (플래너 유저 중) | 0% (미출시) | 플래너 유저의 30%+ | 50%+ |
| **신규 진입 유저 vs 기존 유저 재방문률 차이** | 미측정 | **±5% 이내 → 인과 효과 확인** | — |
| **멀티시티 템플릿 복사율** | 0% (미출시) | 측정 시작 | — |

### Health Metrics (guardrails)

| Guardrail | 임계치 | 의미 |
|-----------|--------|------|
| 상품 상세 PV 감소율 | **-10% 이상이면 경고** | 플래너 진입이 교차탐색을 잠식하고 있진 않은지 |
| "내 일정에 추가" 버튼 CTR | **0.5% 미만이면 재설계** | 진입점 UX가 적절한지 |
| 플래너 이탈률 (진입 후 스팟 0개 추가 이탈) | **50% 이상이면 경고** | Cold Start 해소가 작동하는지 |

## 7. Growth Engine

### 유저 획득 → 플래너 진입 파이프라인

```
[TOP — 유입]
  메타 카탈로그 광고 (RETURN_D1 결제 +54% 기여)
  + 벚꽃 시즌 컬렉션 광고 → 플래너 템플릿 랜딩
  + Reddit/SEO 오가닉 인지도 구축 (현재 0.1% → 목표 확대)

[MIDDLE — 진입점 확대] ← 핵심 성장 레버
  검색 결과 "내 일정에 추가" 인라인 버튼 (모수 950명/일)
  좋아요 리스트 → "찜한 X개로 플랜 만들기" (모수 108명/일)
  블로그/가이드 본문 내 "일정에 추가"
  스팟 리스트 페이지 인라인 CTA
  "날짜 없이 일단 추가" 경량 온보딩

[BOTTOM — Open Loop 활성화]
  빈 날짜 시각화 → "아직 X일이 비어있어요"
  일정 검증 피드백 → 수정-재검증 루프
  스팟 추가 후 "근처 추천" → 교차탐색 트리거
  2개+ 스팟 시 지도 핀 자동 표시 → 동선 시각화

[재방문 유도 — CRM]
  D+1: "어제 추가한 [남산타워], 일정에 넣어볼까요?"
  D+3: "플랜의 X일에 맞는 추천: [N서울타워 + 명동]"
  D+7: "아직 빈 날짜가 있어요. 이번 추천은 [X]"
```

### 시즌 활용 전략
- **봄(3~5월)**: 벚꽃 플래너 템플릿 + "서울+부산 3~4일" 멀티시티
- **원칙**: 시즌 캠페인은 실제 시즌 3개월 전 시작 (Reddit 데이터: 벚꽃 질문은 1~2월에 피크)

## 9. Defensibility

### 경쟁사가 복제할 수 없는 것

**1. "일정 계획 + 검증 + 구매" 통합 (Klook/KKday 불가)**

| 기능 | Creatrip | Klook | KKday | 구글 스프레드시트 |
|------|----------|-------|-------|-----------------|
| 상품 탐색/구매 | O | O | O | X |
| 일정 계획 | **O** | X | X | O |
| **일정 검증 (이동시간, 과밀 경고)** | **O (유일)** | X | X | X |
| **동선 지도 시각화** | **O (유일)** | X | X | X |
| 좋아요→일정 자동 연결 | **O** | X | X | X |

Klook/KKday는 개별 상품 판매 플랫폼 — "뭘 살까"의 답만 줌.
구글 스프레드시트는 일정을 "적는" 도구 — "괜찮은지"는 알려주지 않음.
**크리에이트립만이 "계획 → 검증 → 구매"를 하나의 플로우에서 완결**할 수 있음.

**2. 맥락 축적에 의한 Switching Cost**

유저가 플래너에 스팟을 추가할수록, 그 여행의 맥락(동선, 날짜 배치, 구매 이력)이 크리에이트립에만 축적된다. 이 맥락을 Klook으로 옮기려면 처음부터 다시 구성해야 함. **사용 깊이가 곧 이탈 장벽.**

**3. Reddit 검증 니즈 흡수 → 오가닉 인지도 루프**

Reddit에서 "itinerary check please" 게시글에 "크리에이트립 플래너에서 자동 검증 가능"이라는 답변이 달리기 시작하면, 크리에이트립 인지도(현재 0.1%) 확대의 오가닉 채널이 됨. **기능 자체가 마케팅 채널로 작동.**

## Strategic Risks

### Risk 1: 인과관계 미확인 — "플래너를 써서 리텐션이 높은 건지, 원래 의도 높은 유저가 플래너까지 도달한 건지"
- **완화**: 카테고리 A(진입점 확대)로 "기존에 플래너를 몰랐던 유저"를 유입시킨 후, 이 신규 유저의 D+1 재방문률을 기존 유저와 비교 → Natural Experiment로 인과 추론
- **Go/No-Go 기준**: 신규 vs 기존 유저 재방문률 차이가 ±5% 이내면 인과 효과 확인

### Risk 2: 진입률 0.5% 미달 — 진입점을 넓혀도 유저가 플래너를 원하지 않을 가능성
- **완화**: 4주 후 Go/No-Go 게이트. No-Go 시 카테고리 B 보류, 교차탐색·비교하기·럭키드로우에 리소스 재배분
- **플래너는 포트폴리오의 일부이지 전부가 아님** — 실패해도 전체 RETURN_D1 전략이 무너지지 않음

### Risk 3: "일정 검증"의 정확도 문제 — 라이트 버전의 피드백이 부정확하면 신뢰 상실
- **완화**: 라이트 버전은 "1일 4스팟 이상 = 과밀"과 직선거리 기반 이동 시간 추정에 한정. 확정적 판단("이 일정은 틀렸다")이 아닌 제안적 톤("빡빡할 수 있어요")으로 UX 설계
