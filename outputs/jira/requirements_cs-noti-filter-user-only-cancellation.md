# 세부 기획안 요구사항 문서

## 배경

### 문제 정의

- CS 채널에 적재되는 **예약 취소 알림 메시지가 과다**하여, CS 담당자가 모든 알림을 일일이 확인해야 하는 운영 부담이 크다.
- 알림 중 상당수는 **수정자가 사용자(user)뿐인 케이스**로, 어드민이나 파트너의 추가 처리가 필요 없는 건이다. 이런 건은 취소 정책에 따라 자동 처리되므로 **CS의 인공 확인 자체가 불필요**하다.
- 현재는 사용자 단독 처리 건과 어드민/파트너 개입 건이 구분 없이 동일하게 알림 채널에 노출되어, **정작 처리가 필요한 알림이 노이즈에 묻히는** 상황이다.

### 가설적 임팩트

- CS 채널 알림 수신량이 감소하여 **확인 부담이 경감**되고, 진짜 처리가 필요한 케이스(파트너/관리자 개입 건)에 **자원을 집중**할 수 있다.
- 정량 효과: {미정} — 최근 30일 기준 "수정자 = user only" 취소 건의 비율을 측정 후 알림 절감률을 산출 필요.

## 구현

### 유저 플로우

- 예약이 취소되면 시스템이 해당 예약의 **수정 내역(history)** 을 조회한다.
- 수정 내역의 **수정자(modifier) 유형**을 확인한다.
  - **케이스 A — 수정자가 모두 사용자(user)인 경우** → CS 알림 채널로 메시지를 **발송하지 않는다**.
  - **케이스 B — 수정자에 파트너(partner) 또는 관리자(admin)가 한 명이라도 포함된 경우** → 기존과 동일하게 CS 알림 채널로 메시지를 **발송한다**.
- 알림 메시지 자체의 포맷/내용은 기존과 동일하며, **발송 여부만** 분기된다.

#### 케이스 예시 (Notion 제안 기준)

| 예약 코드 | 수정 내역 | 알림 발송 |
| --- | --- | --- |
| `260503wte9b3` (id=586531) | user only | ❌ 제외 (신규 동작) |
| `260504cnjxzr` (id=586823) | partner 포함 | ✅ 발송 (기존 동작 유지) |
| `260505kxysau` (id=587186) | user + admin | ✅ 발송 (기존 동작 유지) |

### 프론트엔드 요구사항

- 본 건은 **백엔드 알림 발송 로직**에서 분기되는 변경으로, 프론트엔드 변경 사항은 **없음**.
- (참고) 어드민의 예약 상세 화면 "수정 내역" UI는 현재 그대로 유지된다. 향후 CS가 알림 제외 사유를 즉시 확인할 수 있도록 어드민에 별도 표시가 필요하면 후속 과제로 분리한다. ({미정})

### 백엔드 요구사항

- **대상 알림 식별**
  - 현재 CS 채널(예: 슬랙)로 발송되는 **예약 취소 알림 핸들러**를 식별한다. ({미정 — 코드 위치 확인 필요})
- **수정 내역 조회**
  - 알림 발송 직전, 해당 예약(`reserve.code`)의 수정 내역(history) 레코드를 조회한다.
  - 각 수정 레코드의 **수정자 유형(member role)** 을 판별한다.
    - `USER(1)` → 사용자
    - `TRAVEL_PARTNER(56)`, `TRAVEL_PARTNER_SUPERIOR(57)`, 그 외 파트너 계열 → 파트너
    - `ADMIN(90)`, `SUPER_ADMIN(99)` → 관리자
- **알림 발송 분기 조건**
  - 수정 내역에 **사용자(user) 외의 주체(파트너/관리자)가 한 명이라도 포함**되어 있으면 → **발송**
  - 수정 내역의 수정자가 **모두 사용자**인 경우 → **발송 스킵**
  - 수정 내역이 비어 있거나 조회 실패 시 → **기존대로 발송** (안전한 fallback)
- **로깅 / 관측**
  - 스킵된 케이스를 로그로 남겨, 추후 누락된 알림이 없는지 검증 가능하도록 한다.
  - 적용 후 일정 기간 동안 스킵 비율 및 누락 클레임 발생 여부를 모니터링한다.
- **테스트 케이스**
  - 수정자가 user 1명 → 스킵
  - 수정자가 user 다수 → 스킵
  - 수정자에 partner 포함 → 발송
  - 수정자에 admin 포함 → 발송
  - 수정자가 user + partner / user + admin 혼합 → 발송
  - 수정 내역 조회 실패 / 빈 배열 → 기존대로 발송 (fallback)

## 미해결 사항 (TBD)

- 정확한 알림 핸들러 위치 및 "수정 내역" 데이터 모델 식별 — 개발 진입 시 확인 필요.
- 알림 스킵 적용 범위:
  - 모든 취소 케이스 일괄 적용인지, 특정 취소 유형(자발적 취소 / 시스템 자동 취소 등)에 한정할지 — 정책 확인 필요.
- 사용자(user) 정의 범위:
  - 예약자 본인 외에 **게스트(비회원) 취소**도 "user only"로 간주할지 확인 필요.
- 알림 채널 식별:
  - 본 요청이 어떤 채널(슬랙 워크스페이스/특정 채널/이메일 등)에 한정되는지 명시 필요.
