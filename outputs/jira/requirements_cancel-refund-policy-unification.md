# 세부 기획안: 취소/환불 정책 통합

> **선행 작업:** COM-2343 (무료 상품 환불 정책 적용 토글) — 본 작업으로 옵트인 토글을 전사 룰로 승격
> **확인 완료:** 경영지원팀(하얀님), 사업개발팀, CX팀(가진님)
> **사전 논의록:** `.context/inputs/cancel-refund-policy-merger-discussion.md`

## 배경

### 문제 정의

현재 크리에이트립은 "환불 정책"만 존재하고 별도의 "취소 정책"이 없는 구조다. 이로 인해 다음과 같은 운영상 허점이 구조적으로 남아 있다.

- **무료 예약 상품의 무제한 취소 허점.** 무료 예약 상품은 환불할 금액이 0원이므로, 어드민이 환불 정책상 "환불 불가 기간"을 설정해 두어도 사실상 어느 시점에서든 자유롭게 취소가 가능하다. 피부과·뷰티 등 무료 예약이 표준인 카테고리에서 노쇼/직전 취소가 발생할 경우 파트너의 시간·자원 손실이 발생한다.
- **부분 옵트인의 한계.** COM-2343에서 어드민이 스팟별로 옵트인할 수 있는 토글("무료 상품에도 환불 정책 적용")을 도입했으나, 토글이 OFF인 스팟에는 여전히 동일한 허점이 유지된다. 두 달이 지난 시점에도 운영팀·CX 측에서 "토글 ON을 일괄 강제할 수 있는가"라는 요청이 반복되어 왔다.
- **환불 정책 미설정 상품의 통제 부재.** 환불 정책 자체가 입력되지 않은 스팟은 취소 가능 여부가 불명확한 상태로 운영되고 있다. 어드민이 정책을 빠뜨려도 사용자 측면에서 차단되는 가드레일이 없다.
- **타 OTA와의 정책 모델 괴리.** Klook, KKday, 트립닷컴, 마이리얼트립 모두 "취소 정책"과 "환불 정책"을 통합 운영하며 "미환불 취소" 개념을 별도로 두지 않는다. 사용자가 다른 플랫폼에서 학습한 모델과 다르다.

### 가설적 임팩트

- **무료 예약 노쇼/직전 취소 감소.** 피부과·뷰티 카테고리는 무료 예약 비중이 매우 높다. "환불 불가 구간 = 취소 불가" 룰이 전사 강제되면 무성의한 직전 취소로 인한 파트너 손실이 감소한다.
- **CS 응대 부담 감소.** "이 스팟은 토글 ON이라 안 되고 저 스팟은 OFF라 된다"는 케이스별 분기가 사라져, 정책 일관성 기반 응대가 가능해진다.
- **어드민 등록 효율 개선.** "당일 취소 가능" 토글 도입으로 무료 예약 스팟의 표준 케이스(피부과 등)에서 정책 입력 시간이 단축된다.
- **사용자 신뢰도 향상.** 타 OTA에서 학습한 정책 모델과 정렬되어, 환불 시점 분쟁이 감소한다.

## 구현

### 유저 플로우

**핵심 원칙 (이번 작업의 단일 룰).** "실제 환불 금액"이 아니라 "환불 정책상 어느 구간에 속하는지"가 취소 가능 여부의 기준이 된다. 무료 예약/예약금 있는 상품 모두 동일 룰을 따른다.

| 환불 정책 구간 | 환불 금액 | 취소 가능 여부 |
| --- | --- | --- |
| 100% 환불 가능 구간 | 100% 환불 | 취소 가능 (전액 환불) |
| 부분 환불 구간 (예: 50%) | 일부 환불 | 취소 가능 (부분 환불) |
| 환불 불가 구간 (0%) | 0원 | **취소 불가** |
| 환불 정책 미설정 상품 | — | **취소 불가** |

**사용자 — 예약 취소 시도**

- 사용자가 예약 상세에서 '취소' 버튼을 누르면, 시스템은 해당 예약이 속한 환불 정책 구간을 기준으로 취소 가능 여부를 판단한다.
- 100% 또는 부분 환불 구간 → 취소 가능. 실제 환불액이 0원인 무료 예약도 정책 구간 안이면 취소 가능.
- 환불 불가 구간(0%) 또는 환불 정책 미설정 → '취소' 버튼 비활성화. 클릭 시도 시 비활성 사유 툴팁/모달 노출.
- 사용자에게 노출되는 안내는 두 가지로 분리된다.
  - 정책 구간 밖 (정책은 있으나 0% 구간) → "취소 가능 기간이 지났습니다."
  - 정책 미설정 → "이 예약은 취소가 불가합니다. 문의가 필요하시면 고객센터로 연락주세요."

**사용자 — 무료 예약의 케이스 예시**

- Case 1. 무료 예약 + "3일 전 100% 환불" 정책. 예약일 4일 전 취소 요청 → 100% 환불 가능 구간 → 취소 가능 (실제 환불액 0원이어도). 예약일 2일 전 취소 요청 → 환불 불가 구간 → 취소 불가.
- Case 2. "0일 전 100% 환불" 정책. 예약 당일 취소 요청 → 100% 환불 가능 구간 → 취소 가능. (피부과의 표준 케이스)

**어드민 — 스팟 등록/수정**

- 어드민이 스팟 결제 정보 페이지에 진입하면 섹션 라벨이 "환불 정책"이 아닌 **"취소 및 환불 정책"** 으로 노출된다.
- 어드민은 환불 정책 항목을 직접 입력(예: "예약일 N일 전까지 N% 환불")하거나, **"당일 취소 가능" 토글**을 한 번 눌러 0일 전 100% 환불 정책을 즉시 등록할 수 있다.
- 환불 정책이 한 건도 등록되지 않은 상태에서는 정책 설정 영역 상단에 안내 카피가 노출된다 — "취소 및 환불 정책이 설정되지 않은 경우, 사용자가 직접 취소할 수 없습니다. (CS 콘솔을 통한 수동 환불은 가능)".
- 기존의 '무료 상품에도 환불 정책 적용' 토글은 어드민 UI에서 제거된다. 어드민이 스팟별로 선택하는 단계가 사라지고, 모든 스팟이 동일하게 환불 정책 구간 기준으로 동작한다.

**CS 운영자 — 변경 없음**

- CS 콘솔의 수동 환불 경로는 환불 정책 체크를 우회하므로 기존과 동일하게 동작한다.
- 분쟁/보상/외부 결제수단 환불 등의 케이스는 영향을 받지 않는다.
- 어드민의 강제 취소 옵션도 그대로 유지된다.

**적용 시점**

- **신규 예약부터 적용** (예약 생성일 기준). 배포 이전에 생성된 예약은 기존 정책 기준으로 처리된다.

### 프론트엔드 요구사항

**FR-1. 어드민 — 라벨 통합**

- 스팟 결제 정보 페이지 및 관련 모달의 섹션 라벨, 안내 카피, 다이얼로그 타이틀에서 "환불 정책"으로 표현된 모든 텍스트를 **"취소 및 환불 정책"** 으로 변경한다.
- 환불률 입력 안내, 표 컬럼 헤더 등 세부 문구도 함께 변경 대상이다.

**FR-2. 어드민 — '무료 상품 환불 정책 적용' 토글 제거**

- 결제 정보 페이지에서 해당 토글을 제거한다. 무료 상품이 포함된 스팟이라 하더라도 별도의 옵트인 단계 없이 환불 정책 구간 기준이 그대로 적용된다.
- 토글을 통해 노출되던 보조 설명 문구도 함께 제거한다.

**FR-3. 어드민 — '당일 취소 가능' 토글 신규**

- 환불 정책 등록 영역에 "당일 취소 가능" 토글을 추가한다.
- 보조 설명: "활성화하면 예약 당일까지 100% 환불 정책이 적용됩니다."
- 토글 ON 동작
  - 기존에 등록된 정책이 한 건도 없으면 → "예약일 0일 전 100% 환불" 정책이 즉시 등록되고 토글 ON 상태로 표시.
  - 동일한 정책("0일 전 100% 환불")이 이미 존재하면 → 토글이 ON 상태로 표시된다.
  - 다른 환불 정책이 이미 존재하는 상태에서 토글을 ON 하면 → "기존 정책과 함께 등록됩니다." 모달 확인 후 추가.
- 토글 OFF 동작
  - "0일 전 100% 환불" 정책만 단독 제거. 다른 정책은 영향 없음.

**FR-4. 어드민 — 정책 미설정 안내**

- 환불 정책이 0건인 스팟의 결제 정보 페이지 상단에 인포 배너 노출.
- 워딩: "취소 및 환불 정책이 설정되지 않았습니다. 정책을 추가하지 않으면 사용자는 이 스팟의 예약을 직접 취소할 수 없습니다."
- 색상/아이콘은 경고(Warning) 톤. 정책이 1건 이상 등록되면 배너 자동 비노출.

**FR-5. 사용자 웹 — 취소 버튼 활성화 조건 통합**

- 예약 상세의 취소 버튼 활성화 여부를 백엔드가 응답하는 단일 "취소 가능 여부" 값에 따라 통일한다.
- 비활성 상태 UI
  - 정책 구간 밖: 버튼 disable + 툴팁 "취소 가능 기간이 지났습니다."
  - 정책 미설정: 버튼 disable + 툴팁 "이 예약은 취소가 불가합니다. 문의가 필요하시면 고객센터로 연락주세요."
- 회원/비회원/공통 취소 버튼이 모두 동일한 기준으로 동작한다.

**FR-6. 사용자 웹 — 무료 예약 안내 카피**

- 환불 정책 구간 안에 있는 무료 예약(환불 금액 0원) 케이스에서 취소 확인 모달에 추가 안내 라인을 노출한다.
- 워딩: "이 예약은 결제 금액이 없어 환불 처리는 없지만, 정상적으로 취소됩니다."

**FR-7. 사용자 웹 — 상품 상세의 환불 정책 표시**

- 상품 상세에서 환불 정책 안내 모듈의 타이틀과 본문에서 "환불"을 "취소 및 환불"로 통합한다.
- 표시되는 정책 표/문구 자체는 변경 없음.

**FR-8. i18n**

- 취소/환불 관련 메시지 문구를 "취소 및 환불" 톤으로 통합 업데이트한다.
- 운영 중인 다국어 전체에 동일한 의미로 통합 카피 적용. (한/영/일/중-간/중-번/태/베/러/프/스/독/이/인/몽)

### 백엔드 요구사항

**적용 범위.** 코드 변경은 트립 도메인(예약/환불 정책)에 집중된다. 어학당, 숙박은 무료 예약이 존재하지 않거나 외부 공급사 정책을 그대로 사용하므로 코드 변경 범위에서는 제외되나, 정책 룰 자체는 전사 동일하게 적용된다는 점을 문서화한다.

**BR-1. 사용자 셀프 취소 판정 로직 일원화**

- 사용자가 예약을 직접 취소할 수 있는지 판단하는 로직을 다음 단일 룰로 통합한다.
  1. 환불 정책 미설정 스팟 → 취소 불가
  2. 환불 정책 구간 안 (현재 시점에서 가장 가까운 정책의 환불률이 1% 이상) → 취소 가능
  3. 환불 정책 구간 밖 (환불률 0% 또는 가장 가까운 정책에 도달하지 못한 시점) → 취소 불가
- 무료/유료 여부는 더 이상 분기 조건으로 사용하지 않는다. 정책 구간 소속 여부만 기준이 된다.
- 기존에 "환불 가능 윈도우" 여부를 판단하기 위해 사용하던 유틸은 신규 룰에서도 그대로 재사용한다.

**BR-2. 환불 정책 미설정 처리 명시화**

- 환불 정책이 0건인 스팟의 사용자 셀프 취소 시, 환불액 계산 과정에서 발생하는 예외를 "취소 불가"로 명확히 변환하여 응답한다.
- CS 수동 환불 경로는 환불액 계산 로직을 호출하지 않으므로 영향 없음.

**BR-3. API 응답 — 취소 가능 여부 단일 필드 도입**

- 예약 응답에 "취소 가능 여부"를 단일 boolean으로 노출한다. 사용자 클라이언트(웹/모바일)는 이 단일 필드만 보고 취소 버튼 활성 여부를 결정한다.
- 기존에 사용되던 "0원 환불 취소 가능 여부" 필드는 deprecate 처리. 사용처 마이그레이션 완료 후 후속 작업으로 제거한다.
- 기존의 '무료 상품에 환불 정책 적용' 토글 관련 필드와 그 설정 mutation도 deprecate 처리. 후속 작업으로 DB 컬럼과 함께 제거한다.

**BR-4. 적용 시점 분기 (예약 생성일 기준)**

- 신규 판정 로직은 예약 생성 일시를 기준으로 분기한다.
- 배포 시점 이전에 생성된 예약 → 기존 룰로 처리.
- 배포 시점 이후에 생성된 예약 → 신규 단일 룰로 처리.
- 분기 기준 시점은 운영팀과 합의된 datetime으로 관리하며, 마이그레이션 시점에 확정한다.

**BR-5. 데이터 마이그레이션 — 운영팀 리스트 기반**

- 운영팀이 다음 컬럼의 CSV를 전달한다.
  - 스팟 식별자
  - 정책 변경 종류: (a) 기존 옵트인 토글 전환 / (b) 환불 정책 신규 입력
  - 신규 환불 정책 값 (기준일·환불률 배열) — (b) 케이스용
  - 적용 제외 여부 — "당분간 취소 불가가 맞다"는 스팟 표시
- 마이그레이션은 3단계 배포로 진행한다.
  1. **1단계 — 코드 배포.** 신규 룰 활성화 + 기존 필드 deprecate. 기존 토글 데이터는 유지된 상태에서 신규 로직은 더 이상 참조하지 않는다.
  2. **2단계 — 데이터 마이그레이션.** 운영팀 리스트 기반으로 정책 일괄 등록/갱신. 입력 CSV는 `.context/inputs/`에 보관.
  3. **3단계 — 컬럼 정리.** 기존 토글 컬럼 및 GraphQL 필드 제거. 충분한 모니터링 기간 후 진행.
- 리스트에 없는 스팟은 "변경 없음"(기본 동작 유지)으로 처리한다. 즉 화이트리스트 방식.
- 롤백 대비: 입력 CSV 보관 + 마이그레이션 스크립트의 dry-run 옵션 제공.

**BR-6. CS 콘솔 수동 환불 — 변경 없음 (검증 결과)**

- CS 수동 환불 경로는 취소 가능 여부 판정 로직이 환불 정책에 의존하지 않는 별도 검증 단계를 거치며, 어드민 강제 취소 옵션이 유지되므로 신규 정책 도입 후에도 정책 외 환불을 정상 처리할 수 있다.
- 코드 변경 없음.

**BR-7. 부분환불 처리 — 변경 없음**

- 환불 정책의 환불률 값(1~99%)이 자연스럽게 부분환불을 발생시킨다.
- 부분환불 상태값 처리 로직은 기존 그대로 유지.

**BR-8. GA/로그 — 변경 없음**

- 신규 GA 이벤트는 추가하지 않는다. 분석 요청이 발생하면 후속 작업으로 처리.

## 비범위 (Out of Scope)

- **어학당·숙박 도메인의 코드 변경.** 정책 룰 자체는 전사 동일하나 무료 예약이 없어 실제 코드 변경 범위에서는 제외.
- **기존 옵트인 토글 컬럼의 실제 drop.** 본 작업에서는 deprecate만 진행. 충분한 모니터링 후 후속 작업으로 처리.
- **기존 예약에 대한 소급 적용.** 배포 이전 생성된 예약은 기존 정책으로 처리 유지.
- **부분환불 상태값/로직 변경.** 기존 부분환불 처리는 그대로 유지.
- **취소/환불 관련 GA 이벤트 신규 정의.**
- **환불 정책 입력 UI 자체의 디자인 개편.** 라벨 변경과 토글 추가만 진행, 입력 폼 자체는 기존 디자인 유지.

## 검증 포인트

- **사용자 — 환불 정책 구간별 취소 동작.** 100% / 부분 / 0% / 정책 미설정 네 케이스 각각에서 취소 버튼 활성 여부와 안내 카피가 기획안과 일치하는지.
- **사용자 — 무료 예약의 취소 가능 시나리오.** Case 1 (3일 전 100%, 4일 전 시도 vs 2일 전 시도) / Case 2 (0일 전 100%, 당일 시도) 각각에서 의도된 결과가 나오는지.
- **사용자 — 무료 예약 취소 확인 모달.** "결제 금액이 없어 환불 처리는 없지만 정상적으로 취소됩니다." 카피가 무료 예약 케이스에서만 노출되는지.
- **사용자 — 회원·비회원·일반 취소 버튼 동일 기준 동작.** 세 진입점 모두에서 동일한 활성/비활성 결과를 보이는지.
- **어드민 — 라벨 변경.** 스팟 결제 정보 페이지 전체에서 "환불 정책" 표기가 모두 "취소 및 환불 정책"으로 바뀌었는지.
- **어드민 — 당일 취소 토글.** ON 시 "0일 전 100%" 정책 생성 / 다시 ON 시 토글 상태 유지(중복 생성 X) / OFF 시 해당 정책만 제거되고 다른 정책은 영향 없음.
- **어드민 — 정책 미설정 배너.** 정책 0건일 때만 배너 노출, 1건 이상 등록 시 자동 비노출.
- **어드민 — 기존 무료 상품 토글 미노출.** 모든 스팟의 결제 정보 페이지에서 해당 토글이 더 이상 보이지 않는지.
- **백엔드 — 적용 시점 분기.** 배포 시점 이전 생성 예약은 기존 룰, 이후 생성 예약은 신규 룰로 판정되는지.
- **백엔드 — 정책 미설정 스팟의 사용자 취소 차단.** 응답상 취소 가능 여부가 false로 반환되는지.
- **백엔드 — CS 수동 환불 정상 동작.** 환불 정책 미설정/0% 구간 예약에 대해서도 CS 콘솔에서 수동 환불이 정상 처리되는지.
- **백엔드 — 어드민 강제 취소 정상 동작.** 어드민이 정책과 무관하게 취소를 강제할 수 있는 경로가 유지되는지.
- **마이그레이션 — dry-run.** 운영팀 CSV 입력 후 dry-run으로 변경 대상 목록이 정확히 출력되는지.
- **마이그레이션 — 화이트리스트 동작.** 리스트에 없는 스팟은 변경되지 않는지.
- **i18n — 다국어 카피.** 한/영/일 최소 3개 언어에서 통합 카피가 정상 노출되는지.
