# 세부 기획안: 취소/환불 정책 통합

> **선행 작업:** COM-2343 (무료 상품 환불 정책 적용 토글) — 본 작업으로 옵트인 토글을 전사 룰로 승격
> **확인 완료:** 경영지원팀(하얀님), 사업개발팀, CX팀(가진님)
> **사전 논의록:** `.context/inputs/cancel-refund-policy-merger-discussion.md`

## 배경

### 문제 정의

현재 크리에이트립은 "환불 정책"만 존재하고 별도의 "취소 정책"이 없는 구조다. 이로 인해 다음 두 가지 운영상 허점이 구조적으로 남아 있다.

- **무료 예약 상품의 무제한 취소 허점.** 무료 예약 상품은 환불할 금액이 0원이므로, 어드민이 환불 정책상 "환불 불가 기간"을 설정해 두어도 사실상 어느 시점에서든 자유롭게 취소가 가능하다. 피부과·뷰티 등 무료 예약이 표준인 카테고리에서 노쇼/직전 취소가 발생할 경우 파트너의 시간·자원 손실이 발생한다.
- **부분 옵트인의 한계.** COM-2343에서 어드민이 스팟별로 옵트인할 수 있는 토글(`isRefundPolicyOnFreeItems`)을 도입했으나, 토글이 OFF인 스팟에는 여전히 동일한 허점이 유지된다. 두 달이 지난 시점에도 운영팀·CX 측에서 "토글 ON을 일괄 강제할 수 있는가"라는 요청이 반복되어 왔다.
- **환불 정책 미설정 상품의 통제 부재.** 환불 정책 자체가 입력되지 않은 스팟은 취소 가능 여부가 불명확한 상태로 운영되고 있다. 어드민이 정책을 빠뜨려도 사용자 측면에서 차단되는 가드레일이 없다.
- **타 OTA와의 정책 모델 괴리.** Klook, KKday, 트립닷컴, 마이리얼트립 모두 "취소 정책"과 "환불 정책"을 통합 운영하며 "미환불 취소" 개념을 별도로 두지 않는다. 사용자가 다른 플랫폼에서 학습한 모델과 다르다.

### 가설적 임팩트

- **무료 예약 노쇼/직전 취소 감소.** 피부과·뷰티 카테고리는 무료 예약 비중이 매우 높다. "환불 불가 구간 = 취소 불가" 룰이 전사 강제되면 무성의한 직전 취소로 인한 파트너 손실이 감소한다.
- **CS 응대 부담 감소.** "이 스팟은 토글 ON이라 안 되고 저 스팟은 OFF라 된다"는 케이스별 분기가 사라져, 정책 일관성 기반 응대가 가능해진다.
- **어드민 등록 효율 개선.** "당일 취소 가능" 토글 도입으로 무료 예약 스팟의 표준 케이스(피부과 등)에서 정책 입력 시간이 단축된다.
- **사용자 신뢰도 향상.** 타 OTA에서 학습한 정책 모델과 정렬되어, 환불 시점 분쟁이 감소한다.

## 구현

### 유저 플로우

**핵심 원칙 (이번 작업의 단일 룰).** "실제 환불 금액"이 아니라 "환불 정책상 어느 구간에 속하는지"가 취소 가능 여부의 기준이 된다. 무료 예약/예약금 있는 상품 모두 동일 룰을 따른다.

| 환불 정책 구간 | 환불 금액 | 취소 가능 여부 |
| --- | --- | --- |
| 100% 환불 가능 구간 | 100% 환불 | 취소 가능 (전액 환불) |
| 부분 환불 구간 (예: 50%) | 일부 환불 | 취소 가능 (부분 환불) |
| 환불 불가 구간 (0%) | 0원 | **취소 불가** |
| 환불 정책 미설정 상품 | — | **취소 불가** |

**사용자 — 예약 취소 시도**

- 사용자가 예약 상세에서 '취소' 버튼을 누르면, 시스템은 해당 예약이 속한 환불 정책 구간을 기준으로 취소 가능 여부를 판단한다.
- 100% 또는 부분 환불 구간 → 취소 가능. 실제 환불액이 0원인 무료 예약도 정책 구간 안이면 취소 가능.
- 환불 불가 구간(0%) 또는 환불 정책 미설정 → '취소' 버튼 비활성화. 클릭 시도 시 비활성 사유 툴팁/모달 노출.
- 사용자에게 노출되는 안내는 두 가지로 분리된다.
  - 정책 구간 밖 (정책은 있으나 0% 구간) → "취소 가능 기간이 지났습니다."
  - 정책 미설정 → "이 예약은 취소가 불가합니다. 문의가 필요하시면 고객센터로 연락주세요."

**사용자 — 무료 예약의 케이스 예시**

- Case 1. 무료 예약 + "3일 전 100% 환불" 정책. 예약일 4일 전 취소 요청 → 100% 환불 가능 구간 → 취소 가능 (실제 환불액 0원이어도). 예약일 2일 전 취소 요청 → 환불 불가 구간 → 취소 불가.
- Case 2. "0일 전 100% 환불" 정책. 예약 당일 취소 요청 → 100% 환불 가능 구간 → 취소 가능. (피부과의 표준 케이스)

**어드민 — 스팟 등록/수정**

- 어드민이 스팟 결제 정보 페이지에 진입하면 섹션 라벨이 "환불 정책"이 아닌 **"취소 및 환불 정책"** 으로 노출된다.
- 어드민은 환불 정책 항목을 직접 입력(예: "예약일 N일 전까지 N% 환불")하거나, **"당일 취소 가능" 토글**을 한 번 눌러 0일 전 100% 환불 정책을 즉시 등록할 수 있다.
- 환불 정책이 한 건도 등록되지 않은 상태에서는 정책 설정 영역 상단에 안내 카피가 노출된다 — "취소 및 환불 정책이 설정되지 않은 경우, 사용자가 직접 취소할 수 없습니다. (CS 콘솔을 통한 수동 환불은 가능)".
- 기존의 '무료 상품에도 환불 정책 적용' 토글(`isRefundPolicyOnFreeItems`)은 어드민 UI에서 제거된다. 어드민이 스팟별로 선택하는 단계가 사라지고, 모든 스팟이 동일하게 환불 정책 구간 기준으로 동작한다.

**CS 운영자 — 변경 없음**

- CS 콘솔의 수동 환불 경로(`setRefundedAfterManualRefund`, `setRefundedWithoutRefund`)는 환불 정책 체크를 우회하므로 기존과 동일하게 동작한다.
- 분쟁/보상/외부 결제수단 환불 등의 케이스는 영향을 받지 않는다.
- 어드민의 강제 취소 옵션(`isCancellationOverrideAllowed: true`)도 유지된다.

**적용 시점**

- **신규 예약부터 적용** (예약 생성일 기준). 배포 이전에 생성된 예약은 기존 정책 기준으로 처리된다.

### 백엔드 요구사항

**적용 범위.** 코드 변경은 **Trip 서비스(`backend/apps/trip/`)** 에 집중된다. Language School(어학당), Stay(숙박)는 무료 예약이 존재하지 않거나 외부 공급사 정책을 그대로 사용하므로 코드 변경은 없으나, 정책 룰 자체는 전사 동일하게 적용된다는 점을 문서화한다.

**BR-1. 취소 가능 여부 판정 로직 일원화**

- 대상 파일
  - `backend/apps/trip/src/modules/reservation/reservation/repos/loaders/reservation-cancelable-by-user.loader.ts` — `canCancelWithZeroRefund`
  - `backend/apps/trip/src/modules/reservation/reservation/reservation-cancel.service.ts` — `isZeroRefundCancelableByUser`
- 기존 4분기 로직(`isRefundPolicyOnFreeItems` 토글 분기 포함)을 제거하고 단일 룰로 통합한다.
  1. 환불 정책 미설정 스팟 → 취소 불가
  2. 환불 정책 구간 소속 (해당 시점 기준 `closestPolicy.percent > 0`) → 취소 가능
  3. 환불 정책 구간 밖 (0% 또는 가장 가까운 정책에 도달하지 못함) → 취소 불가
- 무료/유료 여부(`isFreeOfCharge`)는 분기 조건에서 제거된다. 정책 구간 소속 여부만 기준.
- 기존의 `isWithinRefundableWindow()` 유틸은 재사용 (dummy amount로 환불액 > 0 여부 확인). 단, 명명을 의미에 맞게 정리 검토.

**BR-2. 환불 정책 미설정 처리 명시화**

- `backend/apps/trip/src/modules/reservation-policy/reservation-policy/reservation-policy.entity.ts` — `calculateRefundAmountByReservation`
- 정책이 0건일 때 현재는 `ZeroRefundableAmountError`를 던지고 있다. 사용자 취소 경로(`canCancelWithZeroRefund`)에서는 이 예외를 catch 하지 않고 "취소 불가"로 명확히 처리되도록 호출부 분기를 보강한다.
- CS 수동 환불 경로는 `calculateRefundAmountByReservation`을 호출하지 않으므로 영향 없음.

**BR-3. 신규 GraphQL 필드 도입 및 기존 필드 deprecate**

- `Reservation` 타입에 신규 boolean 필드 `isCancelableByPolicy`(가칭) 추가. 사용자 측에서 취소 가능 여부를 단일 필드로 조회.
- 기존 `Reservation.isZeroRefundCancelableByUser`는 `@deprecated("isCancelableByPolicy 사용")`으로 마킹. 사용처 마이그레이션 완료 후 후속 PR에서 제거.
- `ReservationPolicy.isRefundPolicyOnFreeItems`를 `@deprecated`로 마킹. 후속 PR에서 컬럼 drop과 함께 제거.
- `setIsRefundPolicyOnFreeItems` mutation도 `@deprecated` 처리.
- 컨슈머 호환성 원칙(`backend/CLAUDE.md`)에 따라 nullable→non-null 전환은 동반 프론트엔드 PR로 처리한다.

**BR-4. 적용 시점 분기 (예약 생성일 기준)**

- 신규 판정 로직에 예약의 `createdAt`을 기준으로 분기 추가.
- 배포 시점 이전에 생성된 예약 → 기존 4분기 로직 유지.
- 배포 시점 이후 생성된 예약 → 신규 단일 룰 적용.
- 분기 기준 시점은 환경변수 또는 빌트인 상수로 관리. 마이그레이션 시점에 운영팀과 합의된 datetime을 설정.

**BR-5. 데이터 마이그레이션 — 운영팀 리스트 기반**

- 운영팀이 다음 컬럼의 CSV를 전달한다.
  - `spotId`
  - 정책 변경 종류: `(a) isRefundPolicyOnFreeItems 전환` 또는 `(b) 환불 정책 신규 입력`
  - 신규 환불 정책 값 (`baseDate`, `percent` 배열) — (b) 케이스용
  - 적용 제외 여부 — "당분간 취소 불가가 맞다"는 스팟 표시
- 마이그레이션은 3단계 배포로 진행한다.
  1. **1단계 — 코드 배포.** 신규 룰 활성화 + 기존 필드 deprecate. `isRefundPolicyOnFreeItems` 컬럼은 유지된 상태로 신규 로직은 컬럼 값을 참조하지 않는다.
  2. **2단계 — 데이터 마이그레이션.** 운영팀 리스트 기반으로 정책 일괄 등록/갱신. 입력 CSV를 `.context/inputs/`에 보관.
  3. **3단계 — 컬럼 drop.** `is_refund_policy_on_free_items` 컬럼 제거 + GraphQL 필드 제거. 충분한 모니터링 기간 후 진행.
- 리스트에 없는 스팟은 "변경 없음"(기본 동작 유지)으로 처리한다. 즉 화이트리스트 방식.
- 롤백 대비: CSV 보관 + 마이그레이션 스크립트의 dry-run 옵션 제공.

**BR-6. CS 콘솔 수동 환불 — 변경 없음 (검증 결과)**

- `setRefundedAfterManualRefund`, `setRefundedWithoutRefund` 경로는 `verifyCancelable()`만 호출하며 환불 정책에 의존하지 않는다는 점을 `reservation.entity.ts:1451-1471`에서 검증 완료.
- 어드민 강제 취소 경로의 `isCancellationOverrideAllowed: true` 옵션이 유지되므로 신규 정책 도입 후에도 CS는 정책 외 환불을 처리할 수 있다.
- 코드 변경 없음.

**BR-7. 부분환불 처리 — 변경 없음**

- 환불 정책의 `percent` 값(1~99)이 자연스럽게 부분환불을 발생시킨다.
- `Reservation.status = PARTIAL_REFUND` 상태값 처리 로직은 기존 그대로 유지.

**BR-8. GA/로그 — 변경 없음**

- 신규 GA 이벤트는 추가하지 않는다. 분석 요청이 발생하면 후속 작업으로 처리.

## 비범위 (Out of Scope)

- **Stay(숙박)·Language School(어학당) 코드 변경.** 정책 룰 자체는 전사 동일하나 무료 예약이 없어 실제 코드 변경 범위에서는 제외.
- **`is_refund_policy_on_free_items` 컬럼의 실제 drop.** 본 PR에서는 deprecate만 진행. 충분한 모니터링 후 후속 PR로 처리.
- **기존 예약에 대한 소급 적용.** 배포 이전 생성된 예약은 기존 정책으로 처리 유지.
- **부분환불 상태값/로직 변경.** 기존 `PARTIAL_REFUND` 상태 처리는 그대로 유지.
- **취소/환불 관련 GA 이벤트 신규 정의.**
- **환불 정책 입력 UI 자체의 디자인 개편.** 라벨 변경과 토글 추가만 진행, 입력 폼 자체는 기존 디자인 유지.

## 검증 포인트

- **사용자 — 환불 정책 구간별 취소 동작.** 100% / 부분 / 0% / 정책 미설정 네 케이스 각각에서 취소 버튼 활성 여부와 안내 카피가 기획안과 일치하는지.
- **사용자 — 무료 예약의 취소 가능 시나리오.** Case 1 (3일 전 100%, 4일 전 시도 vs 2일 전 시도) / Case 2 (0일 전 100%, 당일 시도) 각각에서 의도된 결과가 나오는지.
- **사용자 — 무료 예약 취소 확인 모달.** "결제 금액이 없어 환불 처리는 없지만 정상적으로 취소됩니다." 카피가 무료 예약에서만 노출되는지.
- **어드민 — 라벨 변경.** SpotPaymentInfo 페이지 전체에서 "환불 정책" 표기가 모두 "취소 및 환불 정책"으로 바뀌었는지.
- **어드민 — 당일 취소 토글.** ON 시 `baseDate=0, percent=100` 정책 생성 / 다시 ON 시 토글 상태 유지(중복 생성 X) / OFF 시 해당 정책만 제거되고 다른 정책은 영향 없음.
- **어드민 — 정책 미설정 배너.** 정책 0건일 때만 배너 노출, 1건 이상 등록 시 자동 비노출.
- **어드민 — 무료 상품 토글 제거.** `ToggleIsRefundPolicyOnFreeItemsSwitch` 컴포넌트가 더 이상 렌더되지 않는지.
- **백엔드 — 적용 시점 분기.** 배포 시점 이전 생성 예약은 기존 룰, 이후 생성 예약은 신규 룰로 판정되는지. (테스트용 createdAt 조작 시나리오)
- **백엔드 — 정책 미설정 스팟의 사용자 취소 차단.** GraphQL `isCancelableByPolicy`가 false로 반환되는지.
- **백엔드 — CS 수동 환불 정상 동작.** `setRefundedAfterManualRefund` 호출이 환불 정책 미설정/0% 구간 예약에 대해서도 정상 처리되는지.
- **백엔드 — `isCancellationOverrideAllowed: true` 어드민 강제 취소 정상 동작.**
- **마이그레이션 — dry-run.** 운영팀 CSV 입력 후 dry-run으로 변경 대상 목록이 정확히 출력되는지.
- **마이그레이션 — 화이트리스트 동작.** 리스트에 없는 스팟은 변경되지 않는지.
- **i18n — 다국어 카피.** 한/영/일 최소 3개 언어에서 통합 카피가 정상 노출되는지.
