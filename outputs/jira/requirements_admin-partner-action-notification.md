# 세부 기획안 요구사항 문서 양식

## 배경
### 문제 정의

파트너가 어드민에서 수행하는 액션 중 일부는 어드민(운영팀)의 승인/반려 처리를 거쳐야 한다. 그러나 현재는 운영팀이 파트너의 요청 발생을 능동적으로 인지할 수 있는 어드민 내부 알림 채널이 없으며, 일부 액션은 즉시 자동 확정되어 운영팀이 사후 모니터링하기 어렵다.

또한 파트너가 직접 처리하지 않고 어드민에 요청을 위임하는 액션(예: 예약 거절 요청, 스팟 정보 수정 요청)에 대해 슬랙 등 외부 알림 채널도 없어, 운영팀이 요청을 놓치거나 처리가 지연되는 위험이 있다.

스팟 수정 요청과 같이 어드민 액션이 필요한 건은 파트너 입장에서 즉시 처리되지 않으면 매장 운영에 직접적인 영향을 주므로, 운영팀이 확실하게 인지할 수 있는 알림 시스템이 필요하다.

### 가설적 임팩트

- 운영팀의 파트너 요청 처리 리드타임 단축
- 파트너 요청 누락으로 인한 운영 사고 감소 (특히 가격/예약정책처럼 매출에 직접 영향을 주는 변경)
- 자동 확정되는 파트너 액션에 대한 운영 가시성 확보로 이상 패턴 조기 감지 가능
- 슬랙 알림 신설로 어드민에 상시 접속하지 않아도 즉각 인지 가능

## 구현
### 유저 플로우

**알림 발생 (파트너 → 어드민)**

1. 파트너가 어드민에서 다음 액션 중 하나를 수행한다
   - 어드민 승인 필요 액션: 스팟 등록 요청 / 스팟 기본정보·상품 옵션·예약정책 수정 요청 / 예약 거절 요청
   - 자동 확정 액션: 예약 확정 / 예약 변경 요청 / 타임슬롯 직접 수정 / 스팟 리뷰 답글 작성
2. 시스템이 해당 스팟의 'Creatrip 담당자'(`spot.manager_code`)에 매핑된 운영팀 멤버에게 어드민 인앱 알림을 발송한다
3. 동시에 슬랙 채널로도 알림이 발송된다 (수정 요청·등록 요청 카테고리는 신규로 슬랙 알림 신설)

**알림 수신 (운영팀)**

1. 운영팀이 어드민 상단 알림 영역에서 미확인 알림 개수를 확인한다
2. 알림 패널을 열어 카테고리·상태별로 알림 리스트를 본다
3. 알림 항목을 클릭하면 다음 두 가지가 동시에 일어난다
   - 해당 알림이 '읽음' 상태로 변경된다 (배경색 변화로 구분)
   - 알림이 가리키는 상세 화면이 자동으로 열린다 (수정 요청·등록 요청은 해당 어드민 페이지로 이동 + 쿼리스트링으로 상세 모달 자동 오픈, 자동 확정 액션은 관련 도메인 상세로 이동)
4. 다른 어드민이 먼저 처리한 경우 알림 패널을 다시 열거나 새로고침 버튼을 눌렀을 때 '승인됨'/'반려됨' 상태로 갱신된다

### 프론트엔드 요구사항

**어드민 - 알림 영역 (신규)**

- 어드민 헤더 또는 우측 영역에 알림 아이콘과 배지 노출
- 알림 아이콘 배지 노출 정책
  - `처리 필요` 상태의 알림이 1건 이상 있는 경우: `처리 필요` 알림 수를 숫자 배지로 노출
  - `처리 필요` 상태 알림이 없고 `처리됨`(승인됨/반려됨) 또는 `자동확정` 상태 알림만 있는 경우: 숫자 없이 빨간색 원(dot)만 아이콘 우측 상단에 노출
  - 알림이 전혀 없는 경우: 배지 미노출
  - 모든 상태가 섞여 있는 경우는 첫 번째 규칙이 적용되어 `처리 필요` 알림 수만 숫자로 노출
- 알림 패널(드롭다운 또는 모달) UI
  - 카테고리 위계 표시
    - 상위: `스팟 수정 요청` / `스팟 수정 요청 - 자동확정` / `스팟 등록 요청` / `예약 상태 변경`
    - 하위: 해당 액션 식별자(예: `스팟 기본정보 수정 요청`, `예약 거절 요청`, `파트너의 예약 확정`)
  - 알림 항목별 표시 정보
    - 카테고리(상위·하위)
    - 대상 스팟명 / 스팟코드
    - 액션 수행 파트너명
    - 발생 일시
    - 처리 상태 라벨 (`처리 필요` / `승인됨` / `반려됨` / `자동확정`)
  - 읽음 여부는 배경색으로만 구분 (별도 텍스트 라벨 없음)
  - 알림 항목 자체가 클릭 영역이며 별도 '확인하기' 버튼은 두지 않음
  - 알림 삭제 아이콘
    - 상태가 `승인됨` / `반려됨` / `자동확정`인 알림에만 노출
    - `처리 필요` 상태에서는 노출하지 않음 (처리 전 누락 방지)
    - 클릭 시 해당 알림 단건 삭제, 알림 본문 클릭 동작과는 분리 (이벤트 버블링 차단)
- 필터: 처리 상태 기준 — `처리 필요` / `처리됨`(승인됨+반려됨 통합) / `자동확정`
- 새로고침 버튼: 알림 리스트 재조회 (refetch)
- 알림 패널을 다시 열 때마다 자동 refetch
- 알림 항목 클릭 시 (모든 상태 공통)
  - 읽음 처리 mutation 호출
  - 알림이 가리키는 상세 화면으로 이동
    - 수정 요청 / 등록 요청: 해당 어드민 페이지로 이동 + 쿼리스트링(`?requestId={id}`)으로 상세 모달 자동 오픈
    - 자동 확정 알림: 관련 도메인 상세로 이동 (예약은 예약 상세, 리뷰 답글은 리뷰 상세 등)

**어드민 - 스팟 수정 요청 페이지 수정**

- 파일: `frontend/apps/admin/src/page/TravelPartnerSpotUpdateRequestList.tsx` 또는 인근 경로
- 변경 내용
  - 페이지 마운트 시 쿼리스트링의 `requestId`가 있으면 해당 수정 요청 상세 모달을 자동 오픈
  - 기존 리스트에서 행을 클릭해 모달을 여는 동작과 동일한 모달 컴포넌트를 재사용

**어드민 - 라우팅**

- 알림 클릭 시 이동 경로 (모든 상태 동일하게 적용)
  - 스팟 수정 요청: `/travel-partner-spot-update-request/{spotCode}?requestId={id}`
  - 스팟 등록 요청: 등록 요청 상세 페이지에 동일한 쿼리스트링 패턴 적용
  - 예약 상태 변경(어드민 액션 필요/자동확정): 관련 예약 상세 페이지로 이동
  - 스팟 리뷰 답글 / 타임슬롯 직접 수정: 관련 도메인 상세 페이지로 이동

### 백엔드 요구사항

**1. 테이블**

- 어드민 알림 테이블 (신규)
  - 알림의 발생 카테고리(상위/하위), 대상 스팟, 수행 파트너, 처리 상태, 읽음 여부, 생성 일시 등을 저장
  - 처리 상태와 읽음 여부는 별개의 필드로 관리 (다른 어드민이 요청을 처리해도 본인이 안 봤으면 미읽음 유지)
- 알림 ↔ 수신자 매핑 (구조에 따라 옵션)
  - 현재는 `spot.manager_code` 기반으로 단일 수신자 → 알림 row에 `recipient_member_code` 단일 필드로 충분
  - 향후 1:N 확장이 필요할 경우를 위해 별도 매핑 테이블로 분리하는 것도 검토 가능
- 30일 경과 알림 자동 삭제를 위한 배치/스케줄러 정의

**2. 쿼리**

- 본인 수신 알림 리스트 조회: 카테고리·처리 상태 필터 적용 가능, 페이지네이션
- 알림 배지용 카운트 조회: `처리 필요` 알림 수 + `처리됨`/`자동확정` 알림 존재 여부 (배지 노출 정책 분기에 사용)

**3. 뮤테이션**

- 알림 읽음 처리: 단일 알림 또는 일괄 (전체 읽음)
- 알림 단건 삭제: `승인됨` / `반려됨` / `자동확정` 상태에서만 허용, `처리 필요` 상태는 거부
- (필요 시) 알림 수신자 매핑 갱신: `spot.manager_code` 변경 시 기존 알림의 수신자 갱신 정책은 별도 결정 필요

**4. 알림 발송 트리거**

각 파트너 액션에서 알림을 생성한다.

| 카테고리 | 하위 액션 | 발송 시점 |
|---------|---------|----------|
| 스팟 수정 요청 | 스팟 기본정보 수정 요청 | `createPartnerSpotInfoUpdateRequest` 성공 시 |
| 스팟 수정 요청 | 상품 옵션 수정 요청 | `createPartnerSpotItemInfoUpdateRequest` 성공 시 |
| 스팟 수정 요청 | 예약정책 수정 요청 | `createPartnerSpotReservationPolicyInfoUpdateRequest` 성공 시 |
| 예약 상태 변경 | 예약 거절 요청 | `requestReservationRejectionByPartner` 성공 시 |
| 스팟 수정 요청 - 자동확정 | 타임슬롯 직접 수정 | `updateTimeSlot` 성공 시 (수행자가 파트너인 경우만) |
| 스팟 수정 요청 - 자동확정 | 스팟 리뷰 답글 작성 | 리뷰 답글 작성 성공 시 (수행자가 파트너인 경우만) |
| 예약 상태 변경 - 자동확정 | 파트너의 예약 확정 | `confirmReservation` 성공 시 (수행자가 파트너인 경우만) |
| 예약 상태 변경 - 자동확정 | 파트너의 예약 변경 요청 | 예약일 변경 처리 성공 시 (수행자가 파트너인 경우만) |
| 스팟 등록 요청 | 스팟 등록 요청 | 스팟 생성 요청 성공 시 |

수신자: 대상 스팟의 `spot.manager_code` 에 매핑된 운영팀 멤버.

파트너의 '수정 요청' 1회 클릭으로 여러 필드가 한 번에 묶여 들어와도, 알림은 카테고리 단위 1건으로 그룹화한다.

**5. 처리 상태 동기화**

- 어드민 액션이 필요한 알림은 연결된 수정 요청(`partner_spot_update_request` 또는 `partner_spot_create_request`)의 상태와 연동
- 모달이 열릴 때 또는 알림 리스트 refetch 시점에 최신 처리 상태를 함께 조회하여 표시

**6. 슬랙 알림 발송**

- 어드민 액션이 필요한 카테고리(스팟 수정 요청·스팟 등록 요청)에 대해 슬랙 채널로도 동시 발송
- 기존 `slack_outbox` 기반 슬랙 인프라 재사용
- 발송 채널: {TBD}
- 메시지에는 카테고리, 대상 스팟, 수행 파트너, 어드민 페이지 링크 포함

**7. 보존 정책**

- 알림 생성 후 30일 경과 시 자동 삭제 (cron/배치)

### 향후 확장 고려사항 (이번 범위 제외)

- 파트너 알림 시스템: 어드민 처리 결과(승인/반려)나 유저 액션을 파트너가 어드민 알림으로 받을 수 있도록 확장. 알림 테이블의 수신자 모델을 운영팀/파트너 모두 수용 가능하게 설계
- 1 스팟 ↔ N 담당자 매핑: 현재 `spot.manager_code` 단일 필드 기반이지만, 추후 다수 담당자에게 동시 알림이 필요한 경우 매핑 테이블로 확장
