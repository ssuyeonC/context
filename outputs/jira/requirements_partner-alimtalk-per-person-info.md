# 세부 기획안 요구사항 문서 양식

## 배경
### 문제 정의

현재 파트너에게 발송되는 Bizppurio 알림톡(예약 접수/확정)에는 예약자 대표 정보와 "추가정보(최대 3개)"만 포함되어, **예약자 인원별로 입력한 요구정보 응답이 누락**되어 있다. 파트너는 시술/서비스 준비를 위해 예약자별 세부 응답(예: 시술 옵션, 알러지, 요청사항 등)이 필요하지만, 현재는 어드민 또는 예약전달방에서 별도로 재확인해야 한다.

- 파트너 어드민 인지도/활용도가 낮아 (제휴팀 인터뷰 결과) 알림톡 본문만으로 즉시 준비 가능한 정보가 많을수록 파트너 응대 품질이 올라감
- 어드민에서 재확인하는 과정은 CS/제휴팀의 파트너 응대 리소스로도 이어짐

### 가설적 임팩트

- 파트너가 알림톡 1회 확인으로 예약자별 준비 사항을 즉시 파악 가능 → 어드민/카톡방 문의 감소
- 인원별 응답 누락으로 인한 현장 재확인/컴플레인 케이스 감소
- 제휴팀·CS팀의 "예약자별 요구정보 전달" 수동 업무 제거

## 구현
### 유저 플로우

**파트너**
1. 예약이 생성(접수)되면 Bizppurio 알림톡을 수신한다
2. 알림톡 본문 하단의 `▼ 인원별 예약정보` 섹션에서 예약자별 요구정보 응답을 확인한다
3. 각 예약자 라인에 스팟에 연결된 하위 요구정보 응답이 순서대로 노출된다

**스팟 세팅 (기존)**
- 스팟에 '인원별 요구정보' 및 그 하위 요구정보가 이미 설정되어 있어야 함 (본 요구사항 범위 아님)

### 프론트엔드 요구사항

- 본 요구사항은 백엔드 알림톡 템플릿 본문 변경 중심이며, 별도 프론트엔드 변경은 없음
- 어드민에서 파트너에게 발송된 알림톡 본문을 확인하는 화면이 있을 경우, 본문 변경분이 동일하게 반영되어야 함 (조회 로직이 템플릿 재사용이라면 자동 반영)

### 백엔드 요구사항

**1. 템플릿 본문 변경**

- 대상 파일
  - `backend/apps/trip/src/infrastructure/bizppurio/strategies/talk-template.reservation-confirm-required.service.ts` (예약 접수)
  - `backend/apps/trip/src/infrastructure/bizppurio/strategies/talk-template.reservation-completed.service.ts` (예약 확정) {적용 여부 미정 - 정책 결정 필요}
- 기존 본문 가장 하단(결제금액 뒤)에 아래 섹션을 추가

```
▼ 인원별 예약정보
예약자 1: {응답1} / {응답2} / {응답3}
예약자 2: {응답1} / {응답2} / {응답3}
...
```

- 포맷 규칙
  - 예약의 선택 인원 수만큼 `예약자 N` 라인 반복
  - 한 라인 내에서는 스팟에 연결된 '인원별 요구정보'의 **하위 요구정보 순서대로** 응답값을 구분자로 나열
  - 구분자 초안: ` / ` (Bizppurio 심사 호환성 및 응답값 내 `/` 포함 가능성 고려해 최종 확정 필요)
  - 라벨 포함 여부(`{라벨}: {응답}`) 여부는 정책 결정 필요 (가독성 vs 본문 길이)

**2. 데이터 조회**

- 예약(`Reservation`) 기준으로 다음을 조회
  - 스팟에 설정된 '인원별 요구정보'와 하위 요구정보 스키마 (순서 포함)
  - 해당 예약의 예약자별 인원별 요구정보 응답값
- 예약자 수가 여러 명인 경우 응답을 예약자 인덱스 기준으로 그룹핑하여 라인 구성
- 구체적 테이블명/엔티티명은 구현 시점에 스키마 실측 후 확정 {TBD}

**3. 예외 처리 정책 (정책 결정 필요)**

| 케이스 | 처리 방안 (초안) |
|---|---|
| 스팟에 인원별 요구정보가 설정되어 있지 않음 | `▼ 인원별 예약정보` 섹션 전체 미노출 |
| 일부 예약자가 일부 하위 요구정보 미응답 | 해당 필드 ` - ` 또는 공백 처리 (정책 결정 필요) |
| 응답값에 구분자(`/`)가 포함 | 이스케이프 처리 또는 다른 구분자 사용 |
| 본문 전체 길이가 Bizppurio 최대 길이(1,000자) 초과 | 말줄임 + 어드민 상세 링크 버튼으로 대체 |
| 인원 수가 매우 많은 케이스 | 최대 노출 인원 수 상한 정책 필요 (예: 10명까지, 초과는 "이하 생략") |

**4. Bizppurio 외부 템플릿 재등록**

- 본문 구조 변경이므로 **신규 `templateCode` 발급 필요** (기존 `templateCode` 재사용 불가)
- Bizppurio 관리자 페이지에서 신규 템플릿 등록 및 심사 (심사 기간 1~3일 예상)
- 심사 완료 후 발급된 새 `templateCode`를 Strategy 파일 `TEMPLATE_CONFIGS`에 반영
- 일정 산정 시 심사 리드타임을 필수 버퍼로 포함

**5. 참고 사항 / 범위 제외**

- 파트너 알림톡 외 **고객향 알림톡**(`notification-message-template.ts`)은 범위 제외
- 예약 취소/변경 템플릿은 이번 범위 제외 (필요 시 후속 과제)
- 스팟에 '인원별 요구정보' 자체를 세팅하는 기능은 기존 기능 사용 (범위 제외)
- 어드민 UI에서 본 알림톡 본문을 프리뷰할 수 있는 기능은 본 요구사항에서 고려하지 않음

## 오픈 이슈 (기획 결정 필요)

1. 적용 범위: 예약 접수 알림톡만 vs 예약 접수 + 확정 알림톡 모두
2. 라벨 포함 여부: `{응답}`만 vs `{라벨}: {응답}`
3. 구분자: `/` vs 개행 vs 다른 기호
4. 응답 누락 표기: 공백 vs `-` vs 생략
5. 본문 길이 초과 시 처리: 말줄임 vs 어드민 링크 vs 둘 다
6. 최대 노출 인원 수 상한
