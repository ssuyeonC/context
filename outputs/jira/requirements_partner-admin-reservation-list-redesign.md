# 세부 기획안 요구사항 문서 양식

## 배경
### 문제 정의

파트너 어드민의 '예약내역' 페이지는 운영팀(`ADMIN`/`SUPER_ADMIN`)과 파트너(`TRAVEL_PARTNER`/`TRAVEL_PARTNER_SUPERIOR`)가 동일 화면을 공유하는 구조다. 이로 인해 다음의 문제가 있다.

- 파트너 입장에서 불필요한 운영 전용 컬럼·필터(거래처, 매출일, 예약 ID, 안심케어 청구, 위험 거래건, 자동확정 예약건 토글 등)가 다수 노출되어 인지 부하가 크다
- 필터가 3개 카드로 분리되어 있고 각 행이 독립적으로 동작하여, 현재 어떤 필터가 적용되어 있는지 한눈에 파악이 어렵다
- 일자별 예약 분포를 시각적으로 파악할 수단이 없어, 매장 운영자가 "이번 주 어떤 날에 몇 건이 잡혀있는지" 같은 시간 기반 인지가 어렵다
- 예약 변경 요청과 거절 요청처럼 즉각 처리해야 할 건의 양을 한눈에 파악할 수 있는 시각 신호가 부족하다

순시키 등 어드민 처리율이 높은 파트너들은 이미 어드민에 정기적으로 접속해 일괄 처리하는 패턴이 있어, **파트너 입장에서 정제된 UI**가 제공되면 처리 효율이 개선될 가능성이 높다. 또한 매장 운영 특성상 **달력 형태의 시간 분포 뷰**는 다른 예약 플랫폼에서 보편적으로 제공되는 형태이므로, 미사용 파트너의 진입 장벽도 함께 낮출 수 있다.

### 가설적 임팩트

- 파트너의 예약 처리 인지 부담 감소로 처리 속도 향상
- 일자별 예약 분포 가시성 확보로 매장 운영 계획 수립이 쉬워지며, 어드민 진입 동기 강화
- 변경 요청·거절 요청 등 어드민 액션이 필요한 건의 인지율 향상으로 응답 시간 단축
- 운영팀 화면과 파트너 화면이 분리되며 향후 각 사용자 그룹별 최적화가 용이해짐

## 구현
### 유저 플로우

**파트너의 예약 처리 일과**
1. 파트너가 어드민 - 예약내역 페이지에 진입한다
2. 기본은 리스트 뷰로 진입하며, 상단 필터 카드에서 검색·기간·주문상태·변경요청 등을 적용해 예약을 좁힌다
3. 적용된 필터는 즉시 화면 상단 칩 영역에 요약되어 표시되고, 각 칩의 ✕ 또는 '모두 초기화'로 빠르게 해제할 수 있다
4. '캘린더 뷰' 토글로 전환하면 현재 월의 일자별 예약 카드가 그리드 형태로 보여 매장 일정 기반 확인이 가능하다
5. 캘린더 셀의 예약 카드를 클릭하면 해당 예약 상세를 확인할 수 있다
6. 예약 변경 요청·거절 요청처럼 즉시 처리해야 할 건은 변경요청 필터 영역의 카운트 배지(빨간 숫자)로 인지 후 클릭하여 빠르게 좁힌다

**운영팀(어드민/슈퍼어드민)**
- 본 변경은 파트너용 예약내역 화면에 한정. 운영팀 사용 동선과 권한을 위한 컬럼·필터(거래처, 매출일, 안심케어 청구, 위험 거래건 등)는 운영팀 화면(또는 동일 페이지의 권한 분기)에서 유지된다

### 프론트엔드 요구사항

**대상 페이지**

- 파일: `frontend/apps/admin/src/page/Reservations.tsx` 및 인근 컴포넌트
  - `frontend/apps/admin/src/components/reservation/Filters.tsx`
  - `frontend/apps/admin/src/components/reservation/ReservationTable.tsx`
- 파트너 권한(`TRAVEL_PARTNER` / `TRAVEL_PARTNER_SUPERIOR`)일 때 노출되는 형태를 본 요구사항에 맞춰 변경한다. 운영팀 권한은 기존 동작 유지

**1. 필터 영역 — 단일 카드로 응축**

기존 3개 분리 카드(언어·검색·기간 / 주문상태·결제상태 / 변경요청·기타)를 **하나의 카드 + 4개 행** 구조로 통합한다.

- Row 1 (검색·셀렉트·초기화)
  - 좌측: 검색 인풋 (placeholder: `예약 ID, 예약코드, 이름, 이메일, 회원코드`)
  - 우측: '언어' 드롭다운, '스팟' 드롭다운, '↻ 초기화' 버튼
  - 스팟 옵션은 파트너의 보유 스팟 기준. 2개 이상이면 '전체' 옵션 추가, 1개면 그 스팟만 노출
- Row 2 (기간)
  - SegmentedControl: 생성일 / 예약일
  - 날짜 RangePicker
  - 빠른 기간 버튼: 오늘 / 7일 / 30일 / 이번 달 (리스트 뷰에서만 노출)
  - 캘린더 뷰일 때 RangePicker는 현재 월로 자동 고정되며, 시각적으로 점선 테두리 + 옅은 파란 배경으로 "자동 지정 중"임을 표시
- Row 3 (주문상태)
  - Pill 버튼 그룹: `전체 / 결제대기 / 승인대기 / 확정 / 취소 / 부분 환불 / 사용 완료`
  - 우측에 '특이사항' 드롭다운(`모두 / 특이사항 O / 특이사항 X`)
  - 주문상태가 `취소` 또는 `부분 환불`일 때 하위에 '↳ 취소 사유' Pill 그룹이 계층적으로 노출(`모두 / 수동환불 / 결제만료 / 미환불 취소`)
- Row 4 (변경요청)
  - Pill 버튼 그룹 + 카운트 배지: `전체 / 파트너 예약 변경 요청 N / 유저 예약 변경 요청 N / 관리자 예약 변경 요청 N / 예약 변경 완료 / 파트너 거절 요청`
  - 카운트 배지는 처리 필요 건수를 빨간 텍스트로 강조

**2. 적용된 필터 칩 영역 — 신규**

- 필터 카드 아래, 테이블 위에 위치
- 좌측: `적용된 필터 N개` 라벨
- 활성 필터별 칩(예: `언어: 한국어`, `상태: 확정`, `취소 사유: 수동환불`)
- 각 칩 우측 ✕ 클릭 시 해당 필터만 해제
- 우측 끝에 '모두 초기화' 텍스트 버튼

**3. 운영팀 전용 필터·토글 제거 (파트너 화면 한정)**

다음 항목은 파트너 화면에서 노출하지 않는다. 운영팀 화면 동작에는 영향 없음.

- 카테고리(대분류·중분류) 필터
- 거래처(파트너십) 검색 / 패키지명 검색
- 토글: 수동 환불 필요 / 패키지 판매 / 영수증 리뷰 등록 / 위험 거래건 / 안심케어 / 안심케어 청구 / 자동확정 예약건 / 긴급 리스트만

**4. 툴바 — 뷰 모드 토글 신설**

- 좌측: 전체 건수 (`전체 N건`) + **'리스트 뷰' / '캘린더 뷰' SegmentedControl 토글**
- 우측 (리스트 뷰일 때): 정렬 드롭다운 2종 (오름/내림차순, 기준: 생성일순/예약일순/결제일순) + '패키지 엑셀' / '예약내역 엑셀' 다운로드 버튼
- 우측 (캘린더 뷰일 때): 정렬 드롭다운 숨김, 엑셀 다운로드 버튼은 유지

**5. 리스트 뷰 — 컬럼 정제**

파트너 화면에서 운영팀 전용 컬럼을 제거하고 다음 컬럼만 유지한다.

- 예약 코드 / 예약 상점 / 예약 상품 / 예약자 이름 / 예약 생성일 / 예약 날짜 / 예약 시간 / 예약 상태 / 예약 언어 / 패키지명

제거 대상: 거래처(`partnershipName`), 안심케어(`hasCareOption`), 위험 거래(`reservationDanger`), 결제일(`paymentDate`), 매출일(`salesDate`), 예약 ID(`reservationId`)

**6. 캘린더 뷰 — 신규**

- 월간 그리드(7열 × 5~6행)로 렌더
- 헤더
  - 좌측: 이전/다음 월 이동 버튼 + 현재 연/월 + '오늘' 버튼
  - 우측: 안내 텍스트 `ⓘ 캘린더 뷰는 예약일 기준 · 기간 필터는 현재 월로 자동 지정됨`
- 셀
  - 일자(빨강), 토요일(파랑), 평일(기본) 컬러 차별화
  - 셀 내부에 예약 카드 노출 (시간 + `|` + 라벨)
    - 라벨은 스팟 필터가 '전체'일 때 스팟명, 특정 스팟 단일 선택 시 상품명
  - 셀당 최대 3건 표시, 초과 시 `+N건 더보기` 텍스트로 축약. 클릭 시 해당 일의 전체 예약 리스트 확장 (모달 또는 셀 인플레이트)
  - 예약 카드 클릭 시 해당 예약 상세로 이동
- 캘린더 뷰 진입 시 기간 필터는 현재 월(1일~말일)로 자동 설정되며, 사용자가 이전/다음 월 이동 시 자동 갱신
- 변경요청·주문상태·언어·스팟·검색어 등 다른 필터는 캘린더 뷰에도 동일하게 적용

**7. 권한 분기**

- `MemberRoleType.TravelPartner` / `MemberRoleType.TravelPartnerSuperior`: 본 요구사항의 새 UI 적용
- 운영팀(`Admin` / `SuperAdmin`): 기존 UI 유지 (별도 분기 또는 같은 페이지에서 권한별 분기 처리)
- 권한 분기 구현 위치는 페이지 진입 시 권한 체크 후 렌더 분기 또는 같은 컴포넌트 내 조건부 렌더링 — 구현 디테일은 작업 시점에 결정

### 백엔드 요구사항

**1. 변경요청 카운트 조회**

변경요청 Pill의 카운트 배지를 채우기 위한 데이터가 필요하다.

- 파트너의 권한·소속 스팟 기준으로 다음 카운트를 한 번에 반환하는 쿼리
  - 파트너 예약 변경 요청 수 (`date_change_request_status`가 파트너 요청 상태인 건)
  - 유저 예약 변경 요청 수
  - 관리자 예약 변경 요청 수
  - 파트너 거절 요청 수 (`reservation_has_partner_rejection`에 row가 있는 미처리 건)
- 조회 시점에 카운트만 반환하면 충분하므로 별도 집계 테이블 신설 없이 실시간 카운트로 처리 가능

**2. 캘린더 뷰용 데이터 조회**

- 기존 예약 리스트 쿼리에 다음 입력을 받아 처리하도록 확장
  - 기간(시작일/종료일, 예약일 기준)
  - 스팟·언어·주문상태·변경요청·검색어 등 다른 필터
- 응답은 기존 페이지네이션 결과와 동일한 형태로 충분 (프론트가 일자별로 그룹화)
- 한 달 분의 데이터를 한 번에 받아야 하므로 페이지네이션 한도와 정렬 정책을 확인하여 충분한 양이 반환되도록 조정

**3. 파트너 권한·소속 스팟 필터 적용**

- 기존 예약 리스트 쿼리는 이미 파트너 권한 시 본인 소속 스팟의 예약만 반환하도록 동작 중. 본 요구사항에서는 동작 변경 없음
- 캘린더 뷰 데이터도 동일한 권한 기반 필터를 따른다

**4. 참고 사항**

- 본 요구사항은 주로 파트너 어드민 화면의 UI 변경이며, 신규 도메인 모델·테이블·뮤테이션은 필요하지 않다
- 변경요청 카운트는 기존 데이터를 집계하는 쿼리 추가 수준이며, 캘린더 뷰는 기존 예약 리스트 쿼리를 재사용한다
- 운영팀 화면 동작은 변경하지 않으므로, 백엔드 API 또한 추가 필드를 더하는 형태로 호환성을 유지할 수 있다
