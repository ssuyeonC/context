# 하위 요구정보 Date 타입 추가

## 배경

### 문제 정의

- 예약 시 인원별 하위 요구정보에서 생년월일을 텍스트로 입력받고 있으며, 정해진 형식(YYYY/MM/DD)과 조금만 다르게 입력해도 에러가 발생하여 예약 이탈 유발
  - 예: 띄어쓰기 실수(`YYYY /MM/DD`), 다른 구분자 사용(`YYYY.MM.DD`) 등
- 현재 인원별 하위 요구정보에는 텍스트와 리스트 타입만 존재하고, Date 타입이 구현되지 않음
- 고객은 "자신의 잘못"이 아닌 "사이트 예약 불편"으로 인식하여 이탈 가능성이 높음
- 번역 워딩도 불명확하여 추가적인 혼란 발생

### 가설적 임팩트

- 생년월일 입력 오류로 인한 예약 이탈률 감소
- 예약 완료율 개선 → 전환율 상승
- 고객 경험 개선으로 크리에이트립 서비스 신뢰도 향상
- "처음 사용해도 원활하게 예약할 수 있는" 무장벽 UX 구현

---

## 구현

### 유저 플로우

1. 예약 과정에서 인원별 하위 요구정보 중 생년월일 항목에 진입한다.
2. 텍스트 입력이 아닌 캘린더 UI가 노출된다.
3. 유저는 캘린더에서 년/월/일을 선택한다.
4. 선택된 날짜가 정해진 형식으로 자동 저장된다.

### 프론트엔드 요구사항

- **Date 타입 입력 컴포넌트 추가**
  - 인원별 하위 요구정보에서 타입이 `DATE`인 경우 캘린더 UI 노출
  - 기존 상위 요구정보의 Date 타입 UI를 참고하여 구현

- **캘린더 UI 사양**
  - 년/월/일 선택 가능한 Date Picker
  - 생년월일 특성상 과거 날짜 선택이 주 용도이므로, 년도 선택이 용이한 UI 구성 고려
  - 선택 완료 시 지정된 형식(예: `YYYY/MM/DD`)으로 표시

- **유효성 검증**
  - 캘린더 선택 방식이므로 형식 오류 원천 차단
  - 미래 날짜 선택 불가 등 비즈니스 로직에 따른 검증 추가 가능

### 백엔드 요구사항

- **데이터베이스 스키마 대응**
  - 인원별 하위 요구정보 타입에 `DATE` 추가
  - 기존 텍스트/리스트 타입과 동일한 구조로 확장

- **API 수정**
  - 하위 요구정보 조회 시 `DATE` 타입 반환 지원
  - 예약 생성/수정 시 `DATE` 타입 값 저장 지원
  - 날짜 형식 통일(예: ISO 8601 형식 `YYYY-MM-DD`)하여 후속 처리(자동화 등)에 용이하게 구성

- **어드민 대응**
  - 제휴팀이 하위 요구정보 설정 시 `DATE` 타입 선택 가능하도록 어드민 UI 추가
