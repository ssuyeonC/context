# 세부 기획안 요구사항 문서

## 배경

### 문제 정의

스팟 이미지는 본래 스팟 단위(언어 무관)의 **레거시 데이터 소스**에서 관리되었으나, 약 2~3개월 전 일반 어드민의 이미지 편집 영역이 **언어별로 분리 저장하는 새 데이터 소스**로 마이그레이션되었다.

이 마이그레이션은 부분 적용에 그쳐, 레거시 데이터 소스를 여전히 참조하는 화면·서비스가 코드베이스 곳곳에 남아 있을 가능성이 있다. 그 결과 다음과 같은 문제가 누적된다.

- 데이터 정합성 문제 재발 가능성 (대표 사례: 파트너 어드민 수정 요청 화면에서 이미지 불일치 — 별도 티켓 `requirements_travel-partner-spot-image-source-migration.md`로 해결 진행 중)
- 두 데이터 소스를 동시에 유지하는 동기화 비용 누적
- 신규 화면·기능이 어느 소스를 참조해야 할지 모호한 상태가 지속

### 가설적 임팩트

- 스팟 이미지의 데이터 소스가 일원화되어 향후 신규 화면·기능이 자동으로 정합성을 유지한다.
- "데이터 소스 불일치" 유형의 파트너·운영팀 피드백 재발 가능성이 제거된다.
- 데이터 모델 단순화로 신규 개발자의 인지 부담이 감소하고 관련 운영 문의가 줄어든다.
- 정량 임팩트는 본 작업의 사전 조사 단계에서 측정 가능 (레거시 소스 행 수, 참조처 수, 정합성 일치율 등).

## 구현

### 유저 플로우

본 작업은 데이터 모델 정리 중심이며 **유저가 체감하는 UX 변경은 없다.**

- 마이그레이션 전후로 모든 화면(어드민·파트너 어드민·유저 웹·검색 결과 등)에서 보이는 이미지는 동일하게 유지되어야 한다.
- 마이그레이션 누락이 발생한 화면이 있다면 정합성 검증 단계에서 발견·보완된다.

### 프론트엔드 요구사항

**(1) 영향 화면 전수 조사 (선행)**

레거시 데이터 소스를 참조하는 화면을 모두 식별한다. 어드민·파트너 어드민·유저 웹·블로그 등 모든 앱이 대상.

식별 결과는 다음 항목으로 정리한다.
- 화면 이름 / 노출 위치 / 현재 참조 소스 / 새 소스로 대체 가능 여부

**(2) 화면별 마이그레이션**

식별된 모든 화면을 새 데이터 소스 기준으로 교체한다. 레거시 소스(언어 무관)에서 새 소스(언어별)로 옮길 때 화면별로 **노출 언어 정책**을 정한다.

- 유저 웹: 유저가 보고 있는 현재 언어 기준
- 어드민: 현재 어드민의 언어 탭 기준 (또는 한국어 디폴트)
- 검색 카드·리스트: 검색 요청 언어 기준

메인 이미지 판정 로직도 새 소스 기준으로 일원화한다.

캐시·CDN 키 변경 가능성을 점검하고, 변경 시 무효화 계획을 함께 수립한다.

### 백엔드 요구사항

**(1) 참조처 전수 조사 (선행)**

레거시 데이터 소스를 참조하는 백엔드 코드를 모두 식별한다. 대상은 리졸버·서비스·유즈케이스·리포지토리·검색 인덱서·일회성 스크립트·외부 연동까지 포함한다. 스팟 엔티티의 "메인 이미지 캐시 컬럼"의 사용처도 함께 파악한다.

**(2) 단계적 마이그레이션**

데이터 손실과 정합성 사고를 최소화하기 위해 단계적으로 진행한다.

- **Phase 1 — 양방향 동기화 안전망 강화 (필요 시)**
  레거시 소스에 쓰는 코드 경로가 남아 있다면, 새 소스에도 함께 쓰도록 보강한다. 양방향 동기화가 보장된 상태에서 다음 단계로 진행한다.
- **Phase 2 — 읽기 경로 마이그레이션**
  모든 화면·코드의 이미지 읽기 경로를 새 소스 기준으로 교체한다. 이 단계에서 레거시 소스는 여전히 존재하지만 읽기 경로에서는 사라진다.
- **Phase 3 — 쓰기 경로 마이그레이션**
  레거시 소스에 쓰는 코드 경로를 모두 제거한다. 일정 기간 모니터링하여 더 이상 신규 데이터가 들어가지 않음을 확인한다.
- **Phase 4 — 데이터 모델 폐기**
  레거시 데이터를 백업하고, GraphQL 스키마에서 관련 타입·필드를 deprecated 처리한 뒤 충분 기간 경과 후 삭제한다. DB의 레거시 테이블은 운영 정책에 따라 archive 또는 drop. 스팟 엔티티의 "메인 이미지 캐시 컬럼"도 사용처가 사라지면 함께 제거한다.

**(3) 검증 절차**

- 마이그레이션 전후 모든 스팟의 이미지 노출이 동일함을 확인하는 자동 검증 스크립트를 운영한다.
- 샘플 스팟에 대해 모든 화면(어드민·파트너 어드민·유저 웹·검색)을 수동 QA한다.
- 검색 인덱싱 결과가 마이그레이션 전후 동일함을 확인한다 (이미지 메타가 검색 결과에 영향을 주는 경우).

### 함께 결정해야 할 운영 정책

본 작업 진행 중 운영팀·기획자와 합의가 필요한 항목.

- 스팟 엔티티의 "메인 이미지 캐시 컬럼"을 새 구조에서도 캐시로 유지할지, 함께 폐기할지
- 레거시 테이블의 최종 처리 방식 (즉시 drop / archive 후 유지 / soft-delete)
- 마이그레이션 단계의 배포 주기 (한 번에 진행 / Phase별 단계 배포)
- 캐시·CDN 무효화 정책 (즉시 / 점진적 / 유지)
- 유저 웹의 이미지 노출 언어 기준 (유저 언어 / 한국어 fallback 등)
- 레거시 GraphQL 타입의 deprecated 유지 기간

### 영향 범위

- 백엔드: 레거시 이미지 엔티티·리포지토리·서비스·유즈케이스 전반
- 검색 인덱서 (이미지 메타를 참조하는 경우)
- 스팟 엔티티의 메인 이미지 캐시 컬럼
- 모든 클라이언트 fragment에서 사용 중인 레거시 GraphQL 타입·필드
- 유저 웹의 스팟 상세, 스팟 카드, 검색 결과 카드 등 이미지를 노출하는 모든 화면
- 어드민 내 레거시 소스를 참조하는 화면 (파트너 어드민 수정 요청 화면은 별도 티켓에서 선행 처리)

### 의존 관계

- **선행 의존**: 없음 (단독 진행 가능)
- **병행 관계**: 파트너 어드민 수정 요청 화면 마이그레이션 티켓(`requirements_travel-partner-spot-image-source-migration.md`)은 본 작업의 일부를 선행 적용한 부분 마이그레이션. 본 작업이 완료되면 해당 티켓의 화면 단위 차단 구조는 자연스럽게 흡수된다.
- **후속 영향**: 본 작업 완료 후 신규 이미지 관련 기능은 단일 소스 기준으로 자동 일관성을 갖춘다.
