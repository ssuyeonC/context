# 세부 기획안 요구사항 문서 양식

## 배경
### 문제 정의

한국 여행을 준비하는 외국인의 가장 큰 불안은 "내 일정이 현실적인지 모르겠다"는 것

- r/koreatravel 10,000건 분석 결과, 검증 요청(19.6%) + 가능성 확인(14.7%) = 34.3%로 추천 요청(24.6%)보다 10pp 높음
- "Is my 3-day Seoul+Busan itinerary too packed?" 등의 질문이 반복적으로 등장
- 현재 이 니즈를 해소하는 유일한 방법은 Reddit에 글을 올리고 6~24시간 댓글을 기다리는 것 (피드백 품질 편차 큼)
- Klook, KKday에는 일정 계획/검증 기능이 없고, 구글 스프레드시트/노션은 일정을 "적는" 도구일 뿐 "괜찮은지" 알려주지 않음
- SupTravel AI가 "AI Itinerary Checker"를 이미 유료로 운영 중 — 시장 선점 압박 존재

목표: 유저가 자유로운 방식으로 여행 계획을 업로드하면, 사람이 직접 검토하여 일정의 현실성을 피드백하는 파일럿 서비스를 구축한다.

핵심 전략:
- 1단계(파일럿): 사람이 직접 검증 (+ AI 보조) → 즉시 시작
- 2단계: 반복 패턴 규칙 기반 자동화 (빡빡함 경고, 이동 시간 자동 계산) → 검증 패턴 데이터 축적 후
- 3단계: AI/알고리즘 기반 자동 검증 + 사람 보완 → 2단계 효과 검증 후

"한국 여행 전문가가 직접 검증" 포지셔닝으로 AI 기반 경쟁사와 차별화

### 가설적 임팩트

검증 요청 규모 추정:
- 현재 일 플랜 생성 34~50건
- 플랜 생성자 중 검증 요청 비율 30~50% 가정 → 일 10~25건 수준
- 1~2명이 수동으로 처리 가능한 범위

리텐션 효과 가설:
- 플래너 사용 유저의 리텐션 리프트 1.72x, 구매 전환 리프트 3.49x (기확인)
- 검증 요청 → 결과 대기(24시간) → 결과 확인이라는 플로우 자체가 D+1 재방문의 명시적 Open Loop
- "수정 → 재검증" 루프가 반복되면 여행 전 2~4주간 지속적 재방문 동기 발생

구매 전환 가설:
- 검증 과정에서 "이 스팟은 예약이 필요해요" 등의 가이드 → 자연스러운 구매 연결
- Reddit에서는 불가능한 "검증 → 즉시 예약" 원스톱 플로우

성공 기준 (런칭 후 4주 평가):

| 판단 | 조건 | 대응 |
|------|------|------|
| 성공 (Go) | 일 10건+ 요청 + 검증 유저 D+1 재방문 +15% | 운영 확대 + 2단계(규칙 자동화) 착수 |
| 보류 | 일 5~10건 요청 + 재방문 효과 불명확 | 진입점 확대(P1) 추가 + 2주 연장 관찰 |
| 중단 | 일 5건 미만 요청 or 재방문 효과 없음 | 파일럿 종료, 리소스 재배분 |

## 구현
### 유저 플로우

**검증 요청 플로우**

1. 로그인한 유저는 여행 플래너 인덱스 페이지(/itinerary)에서 "여행 계획 검증하기" CTA를 통해 검증 요청 화면(페이지 또는 모달)에 진입할 수 있다.

2. 유저는 한 번에 하나의 검증 요청만 진행할 수 있다.
   - 진행 중(접수됨) 상태의 검증 요청이 있는 경우: "검증 요청" 버튼을 노출하지 않고 "검증 진행 중..." 안내를 표시하여 추가 요청을 차단한다.
   - 이전 검증이 완료된 상태에서 새 검증을 요청하는 경우: "새로운 검증을 요청하면 기존 검증 결과는 제거됩니다. 계속하시겠습니까?" 확인 모달을 노출한다.
     - "계속하기" 선택 시: 기존 검증 요청 및 답변을 숨김 처리하고, 새 검증 요청 화면으로 진입한다.
     - "취소" 선택 시: 모달을 닫고 현재 상태를 유지한다.

3. 유저는 검증 요청 화면에서 아래 세 가지 방식으로 여행 계획을 제출할 수 있다.
   - 텍스트 입력: 자유 형식으로 여행 계획 작성 (최소 50자 이상)
   - 이미지 첨부: 스크린샷, 구글 스프레드시트 캡처 등
   - 플랜 데이터 첨부: 크리에이트립에서 생성한 기존 플랜 선택

4. 유저가 Submit하면 "검증 요청이 접수되었습니다. 24시간 내 결과를 확인할 수 있어요." 확인 화면이 노출된다.
   - 대기 건이 30건을 초과한 경우 예상 응답 시간을 "48시간 이내"로 변경 안내

**검증 결과 확인 플로우**

5. 어드민에서 답변이 완료되면 유저에게 푸시/이메일 알림이 발송된다.

6. 유저는 /itinerary 페이지에서 현재 활성 검증 요청의 상태(접수됨/완료)를 확인할 수 있다. 숨김 처리된 이전 검증 요청은 노출되지 않는다.

7. 검증 완료 상태의 요청을 클릭하면 검증 결과 상세 화면에서 답변 텍스트를 확인할 수 있다.

**유저 피드백 플로우 (추가 논의 반영)**

8. 유저는 검증 결과 상세 화면에서 답변에 대한 피드백을 남길 수 있다.
   - 👍/👎 리액션 (필수, 하나 선택)
   - 👎 선택 시 불만족 사유 객관식 (필수, 복수 선택 가능): "이미 알고 있는 정보였어요" / "더 구체적인 피드백이 필요해요" / "제 일정에 맞지 않는 내용이에요" / "응답이 너무 늦었어요" / "기타"
   - 자유 텍스트 코멘트 (선택)

**운영 플로우**

9. 유저가 Submit하면 슬랙 채널(#itinerary-check)에 알림이 발송된다.
   - 알림에 포함할 정보: 요청자 ID, 디바이스(앱/웹), 언어, 입력 방식, 여행 기간, 스팟 수, 텍스트 요약, 어드민 링크

10. 어드민에서 검증 요청을 확인하고 답변을 작성하여 저장한다.
    - 저장 시 유저에게 자동으로 푸시/이메일 알림이 발송된다.

11. 유저가 👎 피드백을 남기면 슬랙 채널에 즉시 알림이 발송된다.

### 프론트엔드 요구사항

**/itinerary 페이지 — 검증 CTA 및 상태 표시**

- /itinerary 페이지에 "여행 계획 검증" 섹션 추가. 검증 상태에 따라 UI를 분기한다:
  - **검증 요청 없음**: "검증 요청" 버튼 노출
  - **검증 진행 중 (접수됨)**: "검증 요청" 버튼 비노출 + "검증 진행 중..." 텍스트 노출
  - **검증 완료**: "검증 결과 보러 가기" 버튼 노출 + "검증 요청" 버튼 노출. "검증 요청" 버튼 클릭 시 확인 모달 노출: "새로운 검증을 요청하면 기존 검증 결과는 제거됩니다. 계속하시겠습니까?" + "계속하기" / "취소" 버튼
  - 숨김 처리된 이전 검증 요청은 유저 화면에 노출하지 않음

**검증 요청 화면/모달**

- CTA 클릭 시 검증 요청 화면(페이지 또는 모달) 오픈
- 입력 필드:
  - 텍스트 에어리어: 자유 형식 여행 계획 입력 (최소 50자 유효성 검증)
  - 이미지 업로드: 이미지 파일 첨부 기능
  - 플랜 선택: 유저가 생성한 기존 플랜 목록에서 선택하는 UI
  - 텍스트 입력 또는 플랜 데이터 중 최소 하나는 필수
- Submit 버튼 + 접수 완료 확인 화면 (예상 응답 시간 안내)

**검증 결과 상세 화면**

- 어드민에서 작성한 답변 텍스트 렌더링
- 답변 하단에 유저 피드백 영역:
  - 👍/👎 리액션 버튼
  - 👎 선택 시 불만족 사유 객관식 노출 (복수 선택)
  - 자유 텍스트 코멘트 입력 영역 (선택)
  - 제출 버튼

**진입점 (P1 — 파일럿 데이터 확인 후 추가 여부 결정)**

- 스팟 홈: "여행 계획, 괜찮은지 확인해볼까요?" 배너 → 클릭 시 /itinerary로 이동
- 스팟 리스트: 동일 배너 → 클릭 시 /itinerary로 이동

### 백엔드 요구사항

**데이터 모델 (신규 엔티티)**

| 엔티티 | 주요 필드 | 비고 |
|--------|----------|------|
| `ItineraryCheckRequest` | id, userId, textContent, imageUrls[], planId?, status(SUBMITTED/COMPLETED/HIDDEN), device, language, createdAt | 검증 요청 핵심 엔티티 |
| `ItineraryCheckAnswer` | id, requestId, answerText(리치텍스트), createdAt, updatedAt | 어드민 답변 |
| `ItineraryCheckFeedback` | id, requestId, reaction(LIKE/DISLIKE), dissatisfactionReasons[], comment?, createdAt | 유저 피드백 |

**검증 요청 API**

- 검증 요청 생성 API (유저용)
  - 입력: 텍스트 내용, 이미지 URL 목록, 플랜 ID (선택), 유저 디바이스/언어 정보
  - 유효성 검증: 텍스트 50자 이상 or 플랜 ID 필수
  - 동시 검증 차단: 진행 중(SUBMITTED) 상태의 검증 요청이 있으면 생성 차단 (에러 반환)
  - 이전 검증 숨김: 생성 시 이전 완료(COMPLETED) 상태의 검증 요청이 있으면 해당 요청을 HIDDEN으로 변경
  - 생성 완료 시 슬랙 웹훅 발송 (#itinerary-check 채널)
- 검증 요청 조회 API (유저용)
  - 유저의 현재 활성 검증 요청 + 상태(SUBMITTED/COMPLETED) 반환
  - HIDDEN 처리된 요청은 제외
  - 프론트에서 이 API의 응답으로 UI 분기:
    - 활성 요청 없음 → "검증 요청" 버튼 노출
    - SUBMITTED 상태 존재 → "검증 진행 중..." 노출
    - COMPLETED 상태 존재 → "검증 결과 보러 가기" + "검증 요청" 버튼 노출
- 검증 요청 조회 API (어드민용)
  - 전체 요청 리스트 (접수 순 정렬, 상태 필터: 미답변/완료/숨김). HIDDEN 요청도 데이터 분석용으로 조회 가능
  - 요청 상세: 유저 입력 텍스트 + 이미지 + 플랜 데이터 + 유저 피드백

**이미지 업로드**

- S3 등 스토리지 연동하여 이미지 업로드 처리 (기존 이미지 업로드 인프라 재활용 검토)
- 업로드된 이미지 URL을 검증 요청에 연결

**검증 답변 API**

- 답변 생성/수정 API (어드민용)
  - 입력: 답변 텍스트 (리치 텍스트)
  - 저장 시 검증 요청 상태를 COMPLETED로 변경
  - 저장 시 유저에게 푸시/이메일 알림 자동 발송
- 답변 조회 API (유저용)
  - 검증 요청 ID로 답변 텍스트 반환

**유저 피드백 API**

- 피드백 생성 API
  - 입력: 검증 요청 ID, 리액션(LIKE/DISLIKE), 불만족 사유 목록 (DISLIKE 시 필수, 복수 선택), 텍스트 코멘트 (선택)
  - DISLIKE 피드백 생성 시 슬랙 웹훅 발송
- 피드백 조회 API (어드민용)
  - 검증 요청별 피드백 데이터 반환

**슬랙 알림 연동**

- 새 검증 요청 시 슬랙 웹훅 발송 (#itinerary-check)
  - 포함 정보: 요청자 ID, 디바이스, 언어, 입력 방식, 여행 기간, 스팟 수, 텍스트 요약, 어드민 링크
- DISLIKE 유저 피드백 시 슬랙 웹훅 발송 (#itinerary-check)
  - 포함 정보: 요청 ID, 불만족 사유, 코멘트

**유저 알림**

- 답변 완료 시 푸시/이메일 발송: "여행 계획 검증이 완료되었어요!"

**핵심 비즈니스 로직 요약**

| 규칙 | 설명 |
|------|------|
| 동시 검증 차단 | SUBMITTED 상태 요청이 있으면 새 요청 생성 불가 (에러 반환) |
| 이전 검증 숨김 | 새 요청 생성 시 기존 COMPLETED 요청을 HIDDEN으로 변경 |
| 활성 요청만 반환 | 유저 조회 시 HIDDEN 제외, 어드민은 HIDDEN 포함 가능 |
| 상태 전이 | SUBMITTED → COMPLETED (답변 저장 시) / COMPLETED → HIDDEN (새 요청 생성 시) |

**어드민 기능**

- 검증 요청 리스트 페이지
  - 접수 순 정렬, 상태 필터 (미답변/완료/숨김)
  - 유저 피드백 상태 컬럼 (LIKE / DISLIKE / 미응답)
- 요청 상세 + 답변 입력 페이지
  - 유저 입력 표시 (텍스트 + 이미지 + 플랜 데이터)
  - 리치 텍스트 에디터로 답변 작성
  - 유저 피드백(리액션 + 사유 + 코멘트) 표시
