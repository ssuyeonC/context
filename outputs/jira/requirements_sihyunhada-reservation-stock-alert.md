# 세부 기획안 요구사항 문서 양식

## 배경
### 문제 정의

시현하다 예약이 크리에이트립에 접수되면 CX팀이 시현하다 자체 예약 시스템에서 수동으로 재고(가용 슬롯)를 확인한 뒤 예약을 확정해야 한다. 이 과정에서 예약일에 가용 슬롯이 없는 경우 유저가 입력한 예비 후보 날짜/시간으로 조율이 필요하며, 이 또한 수동으로 확인하고 있다.

예약 접수 시 시현하다 외부 API를 호출하여 가용 여부를 자동으로 판단하고, 결과를 슬랙으로 알림 발송하면 CX팀의 수동 확인 부담을 줄이고 예약 확정 속도를 높일 수 있다.

### 가설적 임팩트

- CX팀의 시현하다 예약 건별 재고 확인 작업 시간 절감
- 예약 확정까지의 리드타임 단축
- 예비 예약일 가용 시 빠른 대안 안내로 고객 경험 개선

## 구현
### 유저 플로우

1. 유저가 크리에이트립에서 시현하다 스팟 예약을 접수한다
2. 시스템이 해당 예약의 스팟코드로 시현하다 예약임을 식별한다
3. 예약 정보(지점, 상품 카테고리, 예약일)를 기반으로 시현하다 외부 API를 호출하여 가용 슬롯을 확인한다
4. 유저가 입력한 예비 후보 날짜/시간이 있으면 해당 날짜/시간도 추가로 확인한다
5. 확인 결과를 지정된 슬랙 채널에 알림으로 발송한다
   - 예약일 가용 여부
   - 예비 후보 날짜/시간별 가용 여부 (있는 경우)

**2. 예약 이벤트 감지 및 시현하다 판별**

- 기존 `ReservationEventHandler`의 예약 접수 이벤트(`CONFIRM_REQUIRED` / `COMPLETED`)에서 시현하다 스팟코드 여부를 판별
- 기존 `handle-special-cases` 패턴을 따라 `sihyunhada/` 디렉토리로 분리 구현

**3. 예약 데이터 추출**

- **예약일**: `reserve.reserve_date` (예: `2026-05-30T13:00:00`)
- **예비 후보 날짜/시간**: `reserve.infos` JSON 내 `type: "change_reservation_date"` (required_info code: 796)
  - 필드명(korean_name): `"원하는 후보 날짜/시간대 있나요?"`
  - 값 형식: `{"values": [{"date": "2026-05-30T04:15:00.000Z", "isAllDay": false}, ...]}`
  - 복수 후보 가능 (1~4개)
- **스팟코드**: `reserve.spot_code` → 지점(branch) 매핑에 사용
- **상품 카테고리**: 예약한 옵션명으로 판별
  - `제출사진` → 베이직
  - `기록사진` → 오리지널
  - `프리미엄` → 프리미엄

**4. 시현하다 외부 API 호출**

- 엔드포인트: `GET https://booking-api.sihyunhada.com/api/v1/branches/{branch}/available-slots?date={YYYY-MM-DD}`
- 예약일 기준으로 호출하여 해당 날짜의 가용 슬롯 확인
- 예비 후보 날짜가 있으면 각 날짜별로 추가 호출
- 기존 `@nestjs/axios` HttpService 활용 (`handle-special-cases/skin-clinic` 패턴 참고)
- 에러/타임아웃 시 슬랙 알림에 "확인 실패" 표기 (예약 프로세스 자체를 블로킹하지 않음)

**5. 슬랙 알림 발송**

- 기존 예약 슬랙 알림의 **댓글(thread reply)** 로 가용 여부를 추가 발송

- 댓글 내용:
  - 예약일 가용 여부 (가능/불가능)
