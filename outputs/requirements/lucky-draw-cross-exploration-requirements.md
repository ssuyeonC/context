# 럭키드로우 × 교차탐색 연동 — 요구사항 문서

## 배경

### 문제 정의

- 럭키드로우는 리텐션 리프트 2.13x, 구매 전환 리프트 5.11x로 효과가 검증되었으나, 액션율이 0.41%로 극도로 낮음 (진입점 부족)
- 추가 응모권 획득 경로가 "공유"뿐이어서, RETURN_D1 임팩트 1위인 "교차탐색(상품 2개 이상 탐색)" 행동과 연결되지 않음
- 상세페이지의 "최근 본 상품" 플로팅 버튼 클릭률이 낮아, 교차탐색 유도 수단으로 기능하지 못하고 있음

### 가설적 임팩트

- 기본 시나리오(1개 탐색 이탈 유저의 10%가 추가 탐색으로 전환) 기준:
  - 상품 2개+ 탐색 유저: ~1,025명 → ~1,453명/일 (+42%)
  - RETURN_D1 순증: ~22명/일 (+3.1%)
  - 추가 예약: ~39건/일 (+13.4%), 추가 매출 ~4.0M원/일
  - 럭키드로우 참여자: ~22명 → ~53~106명/일 (+2.4~4.8x)

## 구현

### 유저 플로우

1. 유저가 상품 상세페이지를 조회하면, 프론트에서 고유 상세페이지 조회 수를 카운팅한다 (같은 상품 중복 조회 제외)

2. 유저가 상세페이지에서 Precaution(주의사항) 섹션까지 스크롤하면, 하단에 스낵바가 슬라이드업으로 등장한다
   - 단, 백엔드에서 당일 `PRODUCT_BROWSE` 응모권을 이미 수령한 경우 스낵바를 노출하지 않는다

3. 스낵바는 현재 조회 진행 상태에 따라 문구가 달라진다
   - 진행 중: "상품 페이지 2개 더 보면 추가 응모권 증정! →"
   - 임박: "상품 페이지 1개 더 보면 추가 응모권 증정! →"
   - 달성: "응모권 증정 완료! 상품 응모하러 가볼까요? →"

4. 스낵바는 5초 후 자동으로 사라진다

5. 스낵바를 클릭하면 최근 본 상품 모달(모바일은 바텀시트)이 열린다
   - 모달 상단에 럭키드로우 배너가 노출된다
   - 미달성 시: "상세페이지 N개 더 보면 추가 응모권!" + 프로그레스 바
   - 달성 시: "추가 응모권 사용 가능! → 럭키드로우 바로가기"

6. 유저가 N개 고유 상세페이지 조회를 달성하면, 프론트에서 백엔드에 응모권 발급을 요청한다
   - 백엔드가 당일 `PRODUCT_BROWSE` 이력을 확인하여, 미수령 시 발급 / 이미 수령 시 거부

7. 유저가 최근 본 내역을 삭제하거나 기기를 전환해도, 백엔드에서 이미 수령한 경우 스낵바가 노출되지 않고 카운팅도 시작되지 않는다

### 프론트엔드 요구사항

**상세페이지 조회 카운팅**
- 상세페이지 진입 시 고유 상품 조회 수를 로컬 스토리지에 카운팅
- 같은 상품 중복 조회는 카운트에서 제외
- 페이지 진입 시 백엔드에 당일 `PRODUCT_BROWSE` 수령 이력 조회 → 이미 수령한 경우 카운팅 중단 + 스낵바 미노출

**스낵바 UI**
- 위치: 상세페이지 하단 (모바일 기존 스티키바 위)
- 트리거: 기존 `useCurrentSection` 훅에서 `SpotSectionKey.Precaution` 감지 시 노출
- 동작: 슬라이드업 등장 → 5초 후 자동 소멸
- 상태별 문구: 진행 중 / 임박 / 달성 (위 유저 플로우 참조)
- 클릭 시: 최근 본 상품 모달(모바일은 바텀시트) 오픈

**최근 본 상품 모달 내 럭키드로우 배너**
- 위치: 모달(바텀시트) 상단
- 미달성 시: "상세페이지 N개 더 보면 추가 응모권!" + 프로그레스 바
- 달성 시: "추가 응모권 사용 가능! → 럭키드로우 바로가기" (클릭 시 럭키드로우 페이지 이동)

### 백엔드 요구사항

**응모권 수령 타입 추가**
- 기존 응모권 수령 타입(receipt_type)에 `PRODUCT_BROWSE` 타입 신설
  ```
  응모권 수령 타입 (receipt_type)
  ├── DAILY_FREE      — 매일 1회 기본 응모 (기존)
  ├── SHARE           — 공유 통한 추가 응모 (기존)
  └── PRODUCT_BROWSE  — 상세페이지 3개 조회 (신규)
  ```

**수령 가능 여부 조회 API (`canReceiveProductBrowse`)**
- 프론트에서 상세페이지 진입 시 호출
- 해당 유저의 당일 `canReceiveProductBrowse` (boolean) 반환
  - `true` = 수령 가능 (아직 미수령) → 프론트에서 카운팅 + 스낵바 노출
  - `false` = 이미 수령 → 프론트에서 카운팅 중단 + 스낵바 미노출
- 프론트는 이 값을 기반으로 모달 내 배너도 분기:
  - `canReceiveProductBrowse` + 로컬 미달성: 프로그레스 바
  - `canReceiveProductBrowse` + 로컬 달성: 럭키드로우 바로가기
  - `!canReceiveProductBrowse`: "오늘의 추가 응모권을 이미 받았어요. 내일 다시 와주세요!"

**응모권 발급 API**
- 프론트에서 3개 달성 시 호출
- 서버가 해당 유저의 당일 `canReceiveProductBrowse` 확인
- `true`(수령 가능) 시: 응모권 발급 + `false`로 변경 + 성공 응답
- `false`(이미 수령) 시: 발급 거부 + 이미 수령 상태 응답
