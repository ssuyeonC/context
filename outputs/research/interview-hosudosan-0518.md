### 스팟 구조 특이점

- 스팟 13165는 `is_publish=0` (스팟 단위 미공개)이지만 `spot_translation.is_publish=1` (어권별 공개) → **번역 단에서만 ON**. 호수도산이 이 구조를 인지하는지 미상.
- 스팟 자체의 `member_code`는 **541411 (Jiyoon Kim @creatrip — 콘텐츠팀 계정)**. 파트너(838900)는 자기 명의 스팟이 없고 콘텐츠팀 계정 아래의 스팟을 어드민에서 처리만 함 → 권한·소유의 분리. 이 구조를 파트너가 의식하는지/불편한지 미상.
- 단일 스팟이지만 시즌·이벤트 단위로 신규 spot이 추가 생성됨(예: 15032 "[크트 단독 오픈 이벤트] 호수도산 건대성수점") → 이벤트성 신규 스팟 운영 방식이 별도로 존재할 가능성.

## 진행 원칙

- **추상 질문 금지** — "어드민 편하세요?" 대신 "오늘 아침 첫 예약 들어왔을 때 어떻게 처리하셨어요?"

- **현장 시연** — 어드민에서 가격이나 이벤트 배너를 한 번 바꿔보라고 했을 때 어디로 들어가는지 보기
- **알림 도달 끝까지 따라가기** — 메일·SMS가 누구 폰/메일함에 떨어져서 누가 어드민을 여는지까지

## 인터뷰 후 정리 템플릿

| 차원 | 항목 | 메모 |
| --- | --- | --- |
| 행동 | 누가 / 언제 / 어디서 / 어떻게 (구체 동작) | |
| 워크어라운드 | 어드민에 없어서 카톡·전화로 처리하는 것 | |
| 시간/사람 | 작업당 분, 사람 수, 처리 주기 | |

특별히 추출해야 할 것:
- **단일 스팟 / 고볼륨 / 수동 승인의 안정적 운영 비결** (Type A 변형 패턴)
- **예약 영역 vs 콘텐츠 영역의 어드민 활용 양극화의 진짜 이유** (어드민 가시성 vs 카톡 편의성 가설)

- **고볼륨 알림 폭주 시 파트너가 정리하는 방식** (알림 채널 다중화·필터링 욕구)

- **시술 자원(시간·시술자) ↔ 크리에이트립 예약 일정 동기화 구조** (이중 운영 패턴)
